NewsAtlas — Security Review

Reviewed against the intended deployment: Docker on a Hetzner Linux VPS, directly internet-facing

Scope: backend (FastAPI), frontend (React), Docker/Compose deployment config
Reviewed at: commit 8dc14cd (end of Phase 5 implementation)
Remediated at: all 17 findings fixed same-session, starting commit 79720f4
Method: manual source review, no dynamic/pen-testing performed

This review assumes the deployment target described by the project owner: the app is built into Docker images via docker-compose.yml and run on a bare Hetzner VPS reachable directly from the internet (no managed load balancer, WAF, or platform-level network isolation in front of it by default). Findings and priorities below are calibrated for that threat model — i.e. the app's own containers and configuration are the only thing standing between the public internet and the database/API.

Update — 2026-07-03: all 17 findings below have been remediated. Every item, including all three criticals, was fixed in the same session this review was written (commit range starting at 79720f4). Each finding section below now includes a green "Fixed" note describing exactly what changed and where. See the remediation checklist at the end for the full list with file references, and re-run a fresh review after any further changes to this area.

C1. No TLS — all traffic is plaintext Critical

docker-compose.yml — backend (port 8000) and frontend (port 4173) services

Fixed: Added a Caddy reverse proxy (Caddyfile, docker-compose.yml) that automatically obtains and terminates HTTPS via Let's Encrypt (self-signed for local DOMAIN=localhost). The backend and frontend services no longer publish any host port at all — only Caddy does (80/443).

Issue

Nothing in the stack terminates TLS. docker-compose.yml publishes the FastAPI backend on 8000 and the static frontend on 4173 directly, with no reverse proxy (Caddy/Traefik/nginx) in front of either. As currently written, deploying this to a Hetzner VPS and pointing DNS at it means the app is served over plain HTTP on the public internet.

Impact

Login/signup passwords and JWT bearer tokens travel in cleartext. Anyone on-path — a router, an ISP, another tenant on shared infrastructure, a coffee-shop network the user is on — can passively capture credentials and session tokens, or actively tamper with responses. Given JWTs are valid for 7 days by default (backend/app/core/config.py:13), a single captured token gives a week of full account access.

Recommendation

Add a reverse proxy service to docker-compose.yml (Caddy is the simplest — automatic Let's Encrypt certs with a ~10-line Caddyfile) that terminates HTTPS on 443 and proxies to the backend/frontend over the internal Docker network. Stop publishing 8000 and 4173 to the host at all — only the proxy should have a host-published port. Enforce HTTPS redirects and set Secure on any cookies if cookies are introduced later.

C2. Postgres port published to the internet with hardcoded credentials Critical

docker-compose.yml:6-12

Fixed: Removed the ports: ["5432:5432"] mapping entirely from docker-compose.yml — Postgres is now only reachable over the internal Compose network. POSTGRES_PASSWORD is now a required interpolated variable (${POSTGRES_PASSWORD:?...}) sourced from .env, with no hardcoded fallback; docker compose config now refuses to run at all if it isn't set.
db:
  image: postgres:16-alpine
  environment:
    POSTGRES_USER: newsatlas
    POSTGRES_PASSWORD: newsatlas
    POSTGRES_DB: newsatlas
  ports:
    - "5432:5432"

Issue

The Postgres container's port is mapped to the host's 0.0.0.0:5432 by default in Docker (Compose port publishing binds all interfaces unless a host IP is specified), and the username/password are hardcoded as newsatlas/newsatlas directly in a file that is committed to source control. The same credentials are duplicated in the backend service's DATABASE_URL override (line 25) and in .env.example.

Impact

On a Hetzner VPS with no additional firewalling, this exposes the full database — every user's password hash, every target company, every AI-generated signal and outreach snippet — to anyone on the internet who can reach the VPS's IP on port 5432 and knows (or guesses, since it's in this project's own git history) the default credentials. This is the single highest-impact item in this review for the stated deployment target.

Recommendation

  • Remove the ports: ["5432:5432"] mapping entirely. The backend reaches Postgres over the internal Compose network by service name (db) and needs no host-published port. Only re-add a host mapping temporarily, bound to 127.0.0.1 specifically (e.g. "127.0.0.1:5432:5432"), if you need local psql access for debugging — and remove it again afterward.
  • Move POSTGRES_PASSWORD out of the committed compose file and into .env (gitignored), generated as a long random value per deployment — not the literal string newsatlas.
  • As defense in depth, also configure the Hetzner Cloud Firewall (or ufw/nftables on the VPS) to only allow inbound 80/443 from the internet and to drop everything else at the network edge, independent of what Docker publishes.

C3. Public, unauthenticated signup grants full workspace access Critical

backend/app/api/auth.py:14-27 (POST /auth/signup)

Fixed: Signup now requires a matching invite_code (constant-time compared via hmac.compare_digest in backend/app/api/auth.py), sourced from the SIGNUP_INVITE_CODE env var. If that var is unset, signup is disabled entirely (fails closed) rather than defaulting to open.

Issue

NewsAtlas is designed as a single shared workspace for one company's sales team (per docs/planning.html), but /auth/signup has no invite code, admin approval, email-domain allow-list, or any other gate — it is reachable by anyone and immediately grants a fully-privileged account. There are no user roles; every account can read every signal, rewrite the company's offering description (which drives every future AI prompt), add/delete target companies, and trigger ingestion or digest sends.

Impact

Once this is reachable on the public internet, it is not really "your sales team's private tool" — it is an openly-joinable application. Anyone who finds the URL (trivial via routine internet-wide scanning, not even targeted) can create an account and get full read/write access to your company's sales intelligence and target list, tamper with the AI context data, or run up your Mistral/NewsAPI bill (see M5).

Recommendation

Before this is exposed to the internet, add one of: (a) an invite-token requirement on signup (a per-workspace secret the admin shares with teammates), (b) an admin-approval step (new accounts start disabled until an existing user/admin approves them), or (c) restrict signup to a configured email domain (e.g. only @proair.com addresses). Given the single-tenant design, an invite-token gate is the least invasive change relative to the current schema.

H1. No guard against the default JWT_SECRET placeholder High

backend/app/core/config.py:11

Fixed: Added assert_secure_for_production() in backend/app/core/config.py, called at import time in main.py — the app now refuses to boot when APP_ENV=production and either JWT_SECRET is still the default/too short, or SIGNUP_INVITE_CODE is unset. Verified live: boots fine in dev, refuses to start in production with insecure config, boots with proper secrets.
jwt_secret: str = "change-me-in-production"

Issue

This placeholder value is committed to source control (readable by anyone with repo access) and the app starts up and signs tokens with it silently if the operator forgets to set JWT_SECRET in .env. There is no startup check, warning, or refusal to boot.

Impact

If this ships to production unset, anyone can forge a valid JWT for any user ID (including IDs they can guess/enumerate) and gain full account access without ever going through login. This is a common real-world misconfiguration source — exactly the kind of thing that's easy to forget when moving from local dev to a VPS deploy.

Recommendation

In app/core/config.py or at FastAPI startup, add a check: if jwt_secret == "change-me-in-production" (or shorter than e.g. 32 chars) and the app is not explicitly running in a recognized "dev" mode, raise and refuse to start. Document generating a real secret with openssl rand -hex 32 in the README/.env.example.

H2. Containers run as root High

backend/Dockerfile, frontend/Dockerfile — no USER instruction in either

Fixed: Both Dockerfiles now create a non-root appuser (uid 1000) and switch to it via USER before running the app.

Issue

Neither Dockerfile creates or switches to a non-root user, so both the FastAPI process and the serve static file server run as root inside their containers.

Impact

This doesn't grant root on the VPS host by itself, but it removes a standard layer of defense-in-depth: if either app process is ever compromised (e.g. via a future dependency CVE or injection bug), the attacker starts with root inside the container, which meaningfully widens the range of possible container-escape and lateral-movement techniques compared to a low-privilege user.

Recommendation

Add a non-root user in each Dockerfile and switch to it before CMD, e.g.: 

RUN useradd --create-home --uid 1000 appuser
USER appuser

Make sure any writable paths (none currently needed at runtime for this app) are owned by that user.

H3. Missing .dockerignore High

no backend/.dockerignore or frontend/.dockerignore; both Dockerfiles use COPY . .

Fixed: Added backend/.dockerignore and frontend/.dockerignore excluding .env, .venv/node_modules, .git, tests, and caches.

Issue

Docker build context is the entire backend/ (or frontend/) directory with no exclusions. If a developer's local .env, .venv/, node_modules/, or .git/ happens to be sitting in that directory at build time (all plausible during local dev, per this repo's own .gitignore patterns), COPY . . bakes it into an image layer.

Impact

Secrets baked into an image layer persist in that layer's history even if a later layer "removes" the file — anyone who can pull or inspect the image (e.g. from a registry, or via docker save) can extract it. This is a common way real API keys/DB passwords leak from otherwise-careful projects.

Recommendation

Add a .dockerignore to both backend/ and frontend/ excluding at minimum:

.env
.venv/
__pycache__/
*.pyc
.git/
node_modules/
dist/
tests/
.pytest_cache/

H4. No rate limiting on auth or ingestion endpoints High

backend/app/api/auth.py (/auth/login, /auth/signup), backend/app/api/ingestion.py (/ingestion/run-now)

Fixed: Added per-IP rate limiting via slowapi (backend/app/core/limiter.py): 10/minute on /auth/login, 5/hour on /auth/signup, 10/hour on the manual ingestion/digest triggers — plus a separate global cooldown (see M5). Verified live: the 11th login attempt in a minute gets 429. Disabled under pytest via ENABLE_RATE_LIMITING=false.

Issue

There is no rate limiting, throttling, or lockout anywhere in the API (no slowapi or equivalent dependency is present). /auth/login can be hit an unlimited number of times per second per IP/account; /auth/signup likewise.

Impact

Combined with C3 (open signup) and M3 (no backend password policy), this allows unrestricted credential-stuffing/brute-force against /auth/login, and unrestricted automated account creation against /auth/signup. It also means /ingestion/run-now (which calls paid external APIs — NewsAPI and Mistral — for every active target company) can be hit in a tight loop by any authenticated user, single-handedly running up API costs or exhausting quota (see M5).

Recommendation

Add rate limiting at the reverse-proxy layer (Caddy/nginx can do simple per-IP rate limits cheaply) and/or in the app via a library like slowapi. At minimum: throttle /auth/login and /auth/signup per-IP, and add a cooldown/lock on /ingestion/run-now (e.g. reject if the last run was under N minutes ago) independent of who calls it.

M1. Unvalidated article URL scheme → potential stored XSS Medium

backend/app/services/news_client.py:60-81 (NewsClient._parse_article) → rendered at frontend/src/pages/SignalDetail.tsx (<a href={signal.article_url}>)

Fixed: NewsClient._parse_article now rejects any URL that doesn't start with http:// or https:// before it's ever stored, so a javascript:/data: URL from a malicious feed can no longer reach the dashboard or digest emails as a live link.

Issue

url = item.get("url")
if not url:
    return None
...
return NewsArticle(..., url=url, ...)

The URL from the NewsAPI response is stored and later rendered as-is as a clickable link, with no check that it actually uses http:/https:. React does not sanitize href values — an anchor with href="javascript:...” will execute script in the page's origin if a user clicks it.

Impact

If the upstream news source (or a future different/compromised provider, or a MITM against NewsAPI itself) ever returns a javascript: or data: URL in the url field, NewsAtlas will happily store it and present it as "View original article" in the dashboard. A user who clicks it would execute attacker-controlled JavaScript in the context of the NewsAtlas frontend (session token theft, given M4's localStorage token storage). Likelihood is low (requires upstream compromise) but impact is high enough, and the fix is cheap, to flag as medium.

Recommendation

In NewsClient._parse_article, validate url.startswith(("http://", "https://")) and skip/reject the article otherwise. As defense in depth, the frontend could also validate the scheme before rendering a live link.

M2. Email enumeration via signup Medium

backend/app/api/auth.py:16-19

Fixed: The duplicate-email response on /auth/signup now returns a generic "Unable to create an account with the provided details" message instead of "Email already registered". Note: the distinct 409 status code itself is still a residual, lower-bandwidth signal (full elimination would require an email-verification redesign, out of scope here) — now mitigated further by C3's invite-code gate limiting who can even probe this.
existing = db.query(User).filter(User.email == payload.email).first()
if existing is not None:
    raise HTTPException(status_code=status.HTTP_409_CONFLICT, detail="Email already registered")

Issue

Returning a distinct 409 for "already registered" lets anyone enumerate which email addresses have accounts by POSTing candidate addresses to /auth/signup and observing the status code. (Login already avoids this correctly — both bad-email and bad-password return the same generic 401.)

Impact

Low on its own, but it's a useful reconnaissance primitive for an attacker combined with H4 (unlimited requests) — e.g. confirming a target's work email is in the system before a targeted phishing/credential-stuffing attempt.

Recommendation

Lower priority than the criticals/highs above, and arguably moot once C3 (open signup) is fixed with an invite/approval gate. If signup remains open, consider a generic response either way, or accept this as a minor known tradeoff.

M3. No backend-side password strength enforcement Medium

backend/app/schemas/auth.py:6-9 (SignupRequest)

Fixed: SignupRequest.password now enforces min_length=10, max_length=72 (the max matches bcrypt's input limit) in backend/app/schemas/auth.py, so this can no longer be bypassed by calling the API directly.
class SignupRequest(BaseModel):
    email: EmailStr
    password: str
    name: str

Issue

password has no length or complexity constraint in the Pydantic schema. The frontend sets minLength={8} on the <input> (frontend/src/pages/Login.tsx), but that's a client-side HTML attribute only — anyone calling the API directly (curl, Postman, a script) can set a 1-character password.

Impact

Combined with H4 (no rate limiting on login), weak/short passwords are practically brute-forceable.

Recommendation

Add password: str = Field(min_length=10) (or your preferred policy) to SignupRequest, and consider checking against a common-password blocklist (e.g. via zxcvbn) if this ever opens up beyond a small internal team.

M4. Long-lived JWT in localStorage, no revocation Medium

backend/app/core/config.py:13 (jwt_expires_minutes: int = 60 * 24 * 7), frontend/src/api/client.ts (localStorage token storage)

Fixed: Default JWT_EXPIRES_MINUTES cut from 7 days to 24h. Added real server-side revocation: User.token_version (new column, embedded in every token as a ver claim) plus a genuine POST /auth/logout endpoint that increments it, so get_current_user rejects any token minted against an older version. (First implementation used a timestamp cutoff compared against the JWT iat claim — live browser testing caught a real bug there: iat's whole-second precision made a login immediately after logout spuriously fail. A version counter sidesteps the precision issue entirely, so switched to that instead; see backend/tests/test_auth.py::test_login_immediately_after_logout_still_works.) Verified live: the same token that worked before logout gets 401 immediately after, and a fresh login right after logout works.

Issue

Access tokens are valid for 7 days by default and stored in localStorage. There is no refresh-token rotation, no server-side session/allow-list, and "logout" (AuthContext.tsx) only clears the client-side copy — the token itself remains valid server-side until it naturally expires.

Impact

Any token leak (XSS per M1, a synced/shared browser profile, a malicious browser extension, a log line that accidentally captures an Authorization header, etc.) grants up to a week of account access with no way for the app operator to revoke it short of rotating JWT_SECRET (which invalidates every user's session at once).

Recommendation

For an MVP this is a reasonable tradeoff, but consider shortening the default expiry (e.g. 24h) and/or adding a minimal server-side revocation list (a table of invalidated token IDs/jti, checked on each request) if this is going to hold real customer/company data on the public internet long-term. This is a "harden later" item, not a launch-blocker, given the smaller blast radius once C1/C3 are fixed.

M5. Unbounded external-API spend via ingestion trigger Medium

backend/app/api/ingestion.py (POST /ingestion/run-now), backend/app/services/ingestion.py

Fixed: Added a DB-persisted, caller-independent cooldown (WorkspaceSettings.last_manual_ingestion_at / last_manual_digest_at, enforced in app/services/workspace_settings.py) — a second call to either manual trigger within MANUAL_TRIGGER_COOLDOWN_SECONDS (default 60s) now gets 429 with a Retry-After header, regardless of who calls it. Verified live alongside the per-IP rate limit from H4.

Issue

Any authenticated user can call this endpoint as many times as they like. Each call makes one NewsAPI request per active target company, plus one Mistral API call per new article found — both metered/billed external services.

Impact

Combined with C3 (anyone can get an account) and H4 (no rate limiting), this is a direct path to running up your NewsAPI/Mistral bill or exhausting NewsAPI's daily quota, effectively a resource-exhaustion / financial-DoS vector against the app operator rather than the app itself.

Recommendation

Add a simple cooldown (e.g. reject if the last successful run finished less than N minutes ago, independent of caller) so the manual trigger can't be hammered in a loop. This is naturally resolved once the recurring scheduled job is the primary path and the manual endpoint is only for occasional testing/demo use.

L1. No security response headers Low

backend (FastAPI/uvicorn), frontend (serve in frontend/Dockerfile)

Fixed: Added Strict-Transport-Security, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy headers (and removed the Server header) at the Caddy layer, applied uniformly to both the API and the frontend.

Neither service sets Content-Security-Policy, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, or (once TLS is added) Strict-Transport-Security. Low severity on its own, but cheap to fix once a reverse proxy exists (C1) — add these headers at the proxy layer so they apply uniformly to both services.

L2. Aging/low-maintenance auth dependencies Low

backend/requirements.txt — python-jose[cryptography]==3.3.0, passlib[bcrypt]==1.7.4

Fixed: Replaced python-jose with PyJWT and passlib with direct bcrypt calls (backend/app/core/security.py) — both actively maintained, and bcrypt is no longer pinned to an old version for passlib compatibility.

Both libraries see infrequent upstream maintenance. The way they're used here is safe today — decode_access_token in app/core/security.py correctly pins algorithms=[settings.jwt_algorithm] (good practice, prevents algorithm-confusion attacks), and bcrypt is pinned to 4.0.1 specifically to stay compatible with passlib's outdated version-detection code. Longer-term, consider migrating to the more actively maintained PyJWT and calling bcrypt directly (or argon2-cffi) instead of through passlib. Not urgent; track via routine dependency updates (e.g. Dependabot / pip-audit).

L3. No request body size limits Low

app-wide; e.g. backend/app/schemas/settings.py (offering_description capped at 8000 chars after parsing)

Fixed: Added request_body { max_size 2MB } in the Caddyfile as the primary control, plus an app-level MaxBodySizeMiddleware (backend/app/core/body_limit.py) checking Content-Length as defense-in-depth for when the app is reached without Caddy in front.

Pydantic max_length constraints only reject a payload after it's been fully read and parsed into memory — they don't cap the raw request size up front. There's no reverse-proxy-level or ASGI-level body size limit configured. Low risk for a small internal tool, but worth adding a cap (e.g. Caddy/nginx client_max_body_size, or a small ASGI middleware) as cheap defense-in-depth against memory-exhaustion DoS once the app has a public reverse proxy anyway.

L4. No auth audit logging Low

backend/app/api/auth.py

Fixed: Added a dedicated newsatlas.audit logger (backend/app/core/audit.py) that records signup/login success and failure, logout, and rate-limit/invite-code rejections with the requesting IP — never passwords or tokens. Verified live in the server log.

Failed logins, signups, and password changes aren't logged anywhere distinguishable from normal request logs. On an internet-facing deployment this makes it hard to notice brute-force attempts or account-creation spam after the fact. Consider structured logging of auth events (without logging passwords/tokens) once basic operational logging is set up.

L5. Frontend build defaults to a plaintext localhost API origin Low

docker-compose.yml:33-36, frontend/Dockerfile:3-4

Fixed: VITE_API_BASE_URL now defaults to the relative /api in both the Dockerfile ARG and the removed docker-compose override, matching the new same-origin Caddy design — confirmed the production build no longer contains any localhost:8000 reference.
build:
  context: ./frontend
  args:
    VITE_API_BASE_URL: http://localhost:8000

VITE_API_BASE_URL is baked into the static bundle at build time (Vite env vars are compile-time, not runtime). If this is deployed to a real domain without overriding this build arg to the real HTTPS API origin, the deployed frontend will keep calling http://localhost:8000 from the visitor's browser — which will simply fail (or, worse, silently hit whatever happens to be listening on the visitor's own localhost:8000). Make sure the real deployment sets VITE_API_BASE_URL to the actual HTTPS API domain before building the production image; consider failing the build if it's left as the default in a "production" build profile.

What's already done right

Parameterized queries throughout. All database access goes through SQLAlchemy's ORM query builder — no raw/string-interpolated SQL was found anywhere, so classic SQL injection is not a concern here.
JWT algorithm is pinned on decode. decode_access_token (app/core/security.py) passes an explicit algorithms=[...] allow-list to jwt.decode, which prevents the classic "alg: none" / algorithm-confusion bypasses.
Bearer-token auth avoids CSRF entirely. Because auth is a header-based bearer token rather than an ambient cookie, the API is not vulnerable to classic CSRF — there's no need for CSRF tokens as currently designed.
Login doesn't leak which field was wrong. /auth/login returns the same generic 401 for both "no such user" and "wrong password", avoiding enumeration there (signup's 409 is the one place this slipped — see M2).
Digest emails HTML-escape all external content. app/services/digest.py's _render_digest_html runs every dynamic field (article title, summary, business relevance, outreach snippet, company name) through html.escape() before interpolating into the email body — correctly treating ingested news content as untrusted when building HTML.
React's JSX escaping protects the dashboard by default. Article titles, summaries, and snippets rendered in the frontend go through normal JSX text interpolation ({signal.summary} etc.), which auto-escapes — no dangerouslySetInnerHTML is used anywhere in the codebase.
External API clients fail closed with clear errors, not crashes. NewsClient/AIClient/EmailClient all raise typed errors on missing config or failed calls, and the ingestion/digest orchestration isolates failures per-company/per-article/per-user rather than letting one bad response take down a whole run — good resilience hygiene, even though it's not itself a security control.

Prioritized remediation checklist

Grouped by when it should happen relative to putting this on a public Hetzner IP for the first time.

Must fix before the first internet-facing deploy

  • (C1) Add a reverse proxy (Caddy recommended) terminating TLS; stop publishing backend/frontend ports directly.
  • (C2) Remove Postgres's host port mapping; move DB credentials to a generated secret in .env, not hardcoded in docker-compose.yml.
  • (C3) Gate /auth/signup behind an invite token, admin approval, or email-domain allow-list.
  • (H1) Add a startup check that refuses to boot on the default JWT_SECRET.
  • (H4) Add basic rate limiting to /auth/login and /auth/signup.

Fix soon after (first hardening pass)

  • (H2) Run both containers as a non-root user.
  • (H3) Add .dockerignore to both backend/ and frontend/.
  • (M1) Validate article URL scheme (http/https only) in NewsClient.
  • (M3) Enforce a minimum password length in SignupRequest.
  • (M5) Add a cooldown to /ingestion/run-now independent of caller.
  • (L5) Confirm VITE_API_BASE_URL is set correctly for the real deployment before building the production frontend image.

Longer-term hardening

  • (M2) Consider a generic signup response if open signup is ever kept.
  • (M4) Shorten default JWT expiry and/or add server-side token revocation.
  • (L1) Add CSP/HSTS/security headers at the reverse-proxy layer.
  • (L2) Migrate off python-jose/passlib to more actively maintained equivalents.
  • (L3) Add a request body size cap at the proxy or ASGI layer.
  • (L4) Add structured audit logging for authentication events.
