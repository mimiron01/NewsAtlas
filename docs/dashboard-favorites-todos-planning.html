Dashboard, Favorites & Signal Todos

Planning document — gap analysis and implementation spec for the non-admin user home workflow

Status: Implemented (2026-07-07) — all 5 phases shipped on claude/signal-dashboard-workflow-1dll3r: data layer, favorites, todos, dashboard, and polish/tests. Verified end-to-end in a live browser session (desktop + mobile) in addition to 197 passing backend tests. One pre-existing, unrelated bug was found and fixed along the way (mobile sidebar not fully hiding — see styles.css .sidebar); a second pre-existing mobile layout issue in .signal-row-actions (action buttons overlapping the title on narrow screens) was found but left as-is — it predates this feature and needs its own redesign decision, not a one-line fix.
Builds on: planning.html (base platform), role-based-company-management-planning.html (per-user company scoping)
Doc purpose: Audit of the requested login → dashboard → signals → favorite → todo workflow against the current app, with a concrete technical plan a future Claude Code session can implement directly from this document

This document evaluates one specific user journey for a non-admin (regular user-role) account: log in → land on a homepage/dashboard surfacing the newest and most important signals → favorite a signal for later → browse a broader "Signals" page that also surfaces favorites → attach a todo/note to any signal → mark it complete to track open vs. finished work. Every step is checked against the current codebase (as of this audit) and, where it doesn't exist yet, a technical solution is specified in enough detail to implement without further discovery.

1. Verdict

The workflow as described does not exist today and is not achievable without new work. The app currently has exactly one authenticated screen for signals (frontend/src/pages/SignalsFeed.tsx, mounted at /) plus a detail page (SignalDetail.tsx). There is no dashboard/homepage distinct from the full signals list, no favoriting concept anywhere in the schema or UI, and no notes/todos concept anywhere in the schema or UI (verified: no matches for favorit, bookmark, todo, or note across the backend or frontend). Everything the user described is buildable on top of the existing architecture (FastAPI + SQLAlchemy + Alembic, React + react-router, per-user company scoping already in place) without redesigning anything — it is three additive features: a dashboard route, a favorites join table + toggle, and a todos table + checklist UI. Sections 5–10 give the concrete plan.

That scoping model — a user only ever sees signals for companies they follow — already does the right thing for a non-admin user and needs no changes; favorites/todos just need their own user-scoped tables following the same pattern.

3. Goals & non-goals

Goals

  • Add a dashboard/homepage at / that surfaces the newest and highest-relevance signals for the logged-in user's followed companies, plus at-a-glance counts (new signals, favorites, open todos).
  • Add per-user signal favoriting with a toggle control on both the feed and detail views.
  • Move the existing full-featured signals list from / to /signals, and add a "favorites only" filter to it.
  • Add per-user, per-signal todo/note items with free text and a done/not-done flag, manageable from the signal detail view, with an aggregate "open todos" view reachable from the dashboard.
  • Keep every new feature scoped to the current user (favorites and todos are private, not shared workspace state) and consistent with existing per-user company-follow scoping — no admin-only gating needed since this is about individual workflow, not workspace configuration.

Confirmed with stakeholder (2026-07-07): all four open questions raised before implementation were resolved in favor of the plan's recommended defaults — pure relevance sort with no recency window, relevance_score alone (no new sentiment scoring) for "highly positive," the / → /signals route change ships without a redirect shim, and dashboard panel sizes stay 8 top signals / 5 recent favorites / 5 open todos. §11 updated accordingly.

Non-goals (this pass)

  • Shared/team todos or @-mentions — todos are private to the user who created them.
  • Due dates, reminders, or notifications for todos — just text + done flag.
  • Changing Signal.status semantics or the existing triage workflow — todos are additive, not a replacement.
  • A "seen/unseen" read-tracking system for the dashboard (e.g. "3 new since your last visit") — noted as a future enhancement (§12), not required to satisfy the requested workflow.
  • Folding article_external_sentiment (NewsData.io-only, inconsistent coverage) into dashboard ranking — see assumption in §11.

4. Key decisions

"Most important" ranking: reuse the existing relevance_score (1–5, already computed against the workspace's offering description) as the sole importance signal for the dashboard's primary section, ordered by relevance_score DESC, created_at DESC, restricted to status IN (new, reviewed) so archived/dismissed signals don't clutter the top view. confirmed No new scoring logic needed — the AI pipeline already produces exactly the "highly positive / directly relevant" signal the workflow calls for; the gap is presentation, not computation.
Dashboard vs. Signals split: new route / → new Dashboard.tsx (curated, small, read-mostly); existing SignalsFeed.tsx moves to /signals unchanged in its filtering/search/bulk-action capability, plus a favorites filter. confirmed Matches the requested UX directly and avoids rewriting the already-solid full-list page.
Favorites storage: a dedicated signal_favorites join table (user_id, signal_id), not a boolean column on signals. confirmed Favoriting is inherently per-user (two users who both follow the same company must be able to favorite independently), so it needs the same join-table shape as company_follows.
Todos are a list, not a single note field: a signal_todos table where a signal can have zero or more todo items per user (each with its own text + is_done), rather than one note+done pair per signal. confirmed "Add a todo/note … mark complete" reads as an ongoing checklist (e.g. "call back Monday", "send follow-up deck") more naturally than a single field, and it costs nothing extra structurally — the UI can still default to feeling like "one note" when a signal only ever gets one todo.
Todo completion vs. signal status: keep Signal.status (new/reviewed/archived/dismissed) and todo is_done as two separate concepts. Status is the shared triage stage of the underlying signal; todo completion is the user's personal task tracking against it. A signal's "open/finished" state for the requested overview is derived purely from its todos (open_todo_count > 0), not from Signal.status. confirmed Conflating them would mean marking a todo done silently changes shared workspace state other users see, which the user didn't ask for and which would surprise teammates on a shared company.
No admin gating: favorites, todos, and the dashboard are available to every authenticated user (admin or not) over whatever signals they can already see via existing company-follow scoping. confirmed This is personal productivity tooling, not workspace configuration — it doesn't belong behind RequireAdmin.

5. Data model changes

Two new tables, both following the existing UUIDPrimaryKeyMixin + TimestampMixin pattern used by CompanyFollow:

# backend/app/models/signal_favorite.py
import uuid

from sqlalchemy import ForeignKey, UniqueConstraint
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import Mapped, mapped_column

from app.db.base import Base
from app.models.mixins import TimestampMixin, UUIDPrimaryKeyMixin


class SignalFavorite(Base, UUIDPrimaryKeyMixin, TimestampMixin):
    __tablename__ = "signal_favorites"
    __table_args__ = (
        UniqueConstraint("user_id", "signal_id", name="uq_signal_favorites_user_signal"),
    )

    user_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), ForeignKey("users.id", ondelete="CASCADE"), nullable=False
    )
    signal_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), ForeignKey("signals.id", ondelete="CASCADE"), nullable=False
    )

# backend/app/models/signal_todo.py
import uuid
from datetime import datetime

from sqlalchemy import Boolean, DateTime, ForeignKey, Text
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import Mapped, mapped_column

from app.db.base import Base
from app.models.mixins import TimestampMixin, UUIDPrimaryKeyMixin


class SignalTodo(Base, UUIDPrimaryKeyMixin, TimestampMixin):
    __tablename__ = "signal_todos"

    user_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), ForeignKey("users.id", ondelete="CASCADE"), nullable=False
    )
    signal_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), ForeignKey("signals.id", ondelete="CASCADE"), nullable=False
    )
    text: Mapped[str] = mapped_column(Text, nullable=False)
    is_done: Mapped[bool] = mapped_column(Boolean, nullable=False, default=False)
    completed_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)

Indexes: add a composite index on signal_todos(user_id, signal_id) and signal_todos(user_id, is_done) (the latter drives the "open todos" dashboard query across all of a user's signals); signal_favorites only needs its unique constraint plus an index on user_id for the "list my favorites" query — both come for free from the FK + unique constraint in Postgres but the extra (user_id, is_done) index on todos is worth adding explicitly in the migration since it's not implied by any constraint.

No changes needed to Signal, Article, TargetCompany, or CompanyFollow — favorites/todos attach alongside them via FK only.

6. API surface changes

Favorites — new endpoints in backend/app/api/signals.py

EndpointBehavior
POST /signals/{signal_id}/favorite Creates a SignalFavorite row for the current user if one doesn't already exist (idempotent — no error on double-favorite). 404 if the signal isn't accessible to the user (reuse _accessible_row scoping). Returns the updated SignalResponse.
DELETE /signals/{signal_id}/favorite Deletes the current user's favorite row if present (idempotent). Returns the updated SignalResponse.

Todos — new router backend/app/api/signal_todos.py

EndpointBehavior
GET /signals/{signal_id}/todos Lists the current user's todos for that signal, oldest first. 404 if signal not accessible.
POST /signals/{signal_id}/todos Body { "text": str } (1–1000 chars, trimmed, rejects empty). Creates a todo owned by the current user.
PATCH /todos/{todo_id} Body { "text"?: str, "is_done"?: bool }. Only the owning user may update (404, not 403, if it belongs to someone else — avoid leaking existence). Setting is_done=true stamps completed_at=now(); setting it back to false clears completed_at.
DELETE /todos/{todo_id} Deletes if owned by current user, else 404.
GET /todos?open=true Cross-signal listing of the current user's todos (default: open only via open=true; open=false or omitted-filter variants can be added later if needed). Each row embeds enough signal/article context to render a clickable list without a second round trip: signal_id, article_title, target_company_name. Powers the dashboard's open-todos widget and could back a future dedicated "My tasks" page.

Dashboard — new router backend/app/api/dashboard.py

EndpointBehavior
GET /dashboard Single aggregate call for the landing page. Scoped to the current user's followed, non-muted companies (same scoping as _scope_to_follows(..., include_muted=False) in signals.py — extract that helper to a shared module so both routers use it, see §7 refactor note). Returns:
  • top_signals: up to 8 signals, status IN (new, reviewed), ordered by relevance_score DESC NULLS LAST, created_at DESC.
  • new_signal_count: count of status = 'new' signals in scope.
  • favorite_count: total favorited signals for the user.
  • recent_favorites: up to 5 favorited signals, most recently favorited first.
  • open_todo_count: total open todos for the user.
  • open_todos: up to 5 open todos (same shape as GET /todos?open=true), most recently created first.

Existing endpoint changes

ChangeDetail
SignalResponse gains fields is_favorited: bool and open_todo_count: int, computed per-user with a left-outer subquery/aggregate in _base_query (mirrors how CompanyFollow is already joined for scoping) so list_signals stays a single query rather than N+1.
GET /signals gains favorited: bool | None query param When true, filters to only the current user's favorited signals — powers the Signals page's "Favorites only" toggle. Composes with existing company_id/status filters.

8. UI/UX details

  • Optimistic updates: favorite toggling and todo checkbox toggling should update local state immediately and roll back with a toast on API failure — consistent with how transitionSignal already works in SignalsFeed.tsx.
  • Accessibility: star button needs aria-label="Favorite" / "Unfavorite" (toggling text, not just icon swap) and must be reachable/operable by keyboard since it sits inside a clickable row; todo checkboxes use native <input type="checkbox"> with an associated <label>, same pattern as the existing "Select all" checkbox.
  • Empty states: Dashboard with zero signals reuses SetupChecklist (same condition as today: no offering description, no target company, or no signals). "Recent favorites" and "Open todos" panels each need their own one-line empty state rather than being hidden entirely, so first-time users discover the features exist ("Star a signal to pin it here.").
  • Todo input UX: single-line text input with Enter-to-submit and a small "Add" button, matching the existing TagInput.tsx interaction pattern already used for keywords in SettingsTargets.tsx; cap visible length and rely on the 1000-char backend limit as a hard stop, not a UI-enforced one, to avoid surprising truncation.
  • Loading states: Dashboard uses the existing <Skeleton /> component while GET /dashboard is in flight, matching SignalsFeed and SignalDetail.
  • Mobile: Dashboard's stat/panel layout should stack under the existing @media (max-width: 720px) breakpoint in styles.css, same as the current sidebar-collapse behavior — no new breakpoints needed, just apply flex-direction: column to the new dashboard grid container under that query.
  • Navigation highlighting: since / is no longer the index route for "Signals," the sidebar's active-link styling (NavLink's built-in .active class) naturally distinguishes "Dashboard" (exact match on /) from "Signals" (prefix match on /signals) with no extra logic.

9. Migration & rollout plan

New Alembic revision, down_revision = "d4e8a2f6b913" (current head, from news_source_expansion.py), adding signal_favorites and signal_todos tables plus the composite index on signal_todos(user_id, is_done) noted in §5. Both tables are purely additive — no backfill needed (existing users simply start with zero favorites/todos), no changes to existing tables/columns, and fully reversible via a straightforward op.drop_table downgrade. Safe to deploy without a maintenance window.

Frontend route change (/ → Dashboard, list moves to /signals) is a breaking change for any bookmarked/shared links to the old root-as-list URL; since this is an internal tool with no public deep links documented elsewhere in the repo, no redirect shim is planned — call this out if the user population relies on bookmarks to / expecting the list specifically.

10. Implementation roadmap

Phase 1 Data layer

Add SignalFavorite and SignalTodo models, Alembic migration, Pydantic schemas. No API or UI yet — establishes the foundation the rest builds on.

Phase 2 Favorites API + UI

Favorite/unfavorite endpoints, is_favorited on SignalResponse, favorited query param on GET /signals, star button on SignalDetail and SignalsFeed rows, "Favorites only" filter on the Signals page. Independently shippable and testable end to end.

Phase 3 Todos API + UI

Todo CRUD endpoints, open_todo_count on SignalResponse, "Notes & todos" section on SignalDetail, open-todo pill on feed rows. Independently shippable.

Phase 4 Dashboard

Shared query-scoping refactor, GET /dashboard endpoint, new Dashboard.tsx page, route/nav changes moving the list to /signals, extract shared SignalRow component. Depends on Phases 2 and 3 being in place since the dashboard surfaces both favorites and todos.

Phase 5 Polish

Empty-state copy for new panels, accessibility pass (aria-labels, keyboard nav through the row-level star button), mobile layout check on the new dashboard grid, backend tests (test_signal_favorites_api.py, test_signal_todos_api.py, test_dashboard_api.py following the existing backend/tests/test_signals_api.py pattern).

11. Open questions & assumptions

"Highly positive signal" interpretation — confirmed: ranks by the existing relevance_score alone, not article sentiment. confirmed No changes to the AI pipeline. The only sentiment field in the schema (article_external_sentiment) is NewsData.io-specific and absent for NewsAPI.org/Google News RSS articles, so it stays out of ranking entirely (§12 revisits this only if sentiment coverage becomes uniform across sources later).
"Newest and most important" combined ranking — confirmed: pure relevance_score DESC, created_at DESC sort, no recency window/cutoff. confirmed A stale high-score signal can persist at the top of the dashboard until its status changes (reviewed/archived/dismissed) — acceptable since that status transition is already the expected way users clear a signal off their plate.
Todo scope — one list per signal, not per (signal, company-follow): assumed each user's todos on a signal are independent of which company-follow relationship gave them visibility (irrelevant in practice since a signal belongs to exactly one company, but noted since company_follows is itself a join table — no ambiguity here, just documenting the reasoning).
Favoriting a signal you can no longer see: if an admin removes a user's follow on a company after the user favorited one of its signals, the favorite row stays (FK cascade only fires on signal/user deletion, not on unfollow) but becomes unreachable through normal scoped queries. This is consistent with how the rest of the app already behaves when access is revoked and needs no special handling — the row is orphaned data, not a bug, and disappears if the signal itself is ever deleted.

12. Future enhancements (out of scope)

  • "Unseen since last visit" tracking on the dashboard (would need a users.last_dashboard_visit_at timestamp or a full per-signal read-receipt table).
  • Todo due dates / reminder notifications (would need a scheduler job similar to services/scheduler.py's existing digest job).
  • A dedicated "My tasks" page listing every open/closed todo across all signals with its own filters (the GET /todos endpoint in §6 is designed to support this later without rework).
  • Sharing a todo with teammates who also follow the same company (currently strictly private per user).
  • Folding article_external_sentiment into ranking once/if sentiment is available uniformly across all news sources (see assumption above).
