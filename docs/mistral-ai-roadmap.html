1. Summary

NewsAtlas has exactly one Mistral integration point today: a single chat-completions call per ingested article that produces a summary, a business-relevance explanation, and an outreach snippet. It works, but it uses Mistral as a one-shot text formatter rather than as a sales-intelligence engine. Every signal is treated as equally important, duplicate coverage of the same event from multiple outlets produces duplicate signals, useful metadata already collected (target company industry) never reaches the prompt, and there is no memory across signals for the same account. The ideas below are organized around one question: what would make a sales rep trust this feed enough to act on it first, before checking anything else? That means better prioritization, less noise, more account context, and outputs shaped for how reps actually work outreach — not just more AI calls.

2. Current implementation

Mistral is called from exactly one place: AIClient.summarize_article() in backend/app/services/ai_client.py, invoked once per newly-ingested article from backend/app/services/ingestion.py.

AspectCurrent state
ClientNo SDK — raw httpx.post to https://api.mistral.ai/v1/chat/completions
Modelmistral-large-latest (env-configurable via MISTRAL_MODEL, one model for every call)
Call shapeOne synchronous request per article; temperature=0.3, Mistral JSON mode (response_format: json_object), no max_tokens/top_p/streaming
Prompt inputsWorkspace company_name + offering_description, target company name, article title + description
Output contract{summary, business_relevance, outreach_snippet} — validated against a Pydantic model
RetryOne retry on JSON/validation/HTTP failure, then the article is skipped (no signal created)
Fallback providerNone — single provider by design
Downstreamsignals table → dashboard (SignalDetail.tsx, SignalsFeed.tsx) → daily HTML/text email digest (digest.py)

3. Gaps & limitations

These are things the codebase either does today in a way that limits insight quality, or was explicitly planned but never implemented:

GapWhere it shows up
No prioritizationEvery signal is equal weight — the digest email and dashboard list signals in creation order, not by importance. A rep has to read everything to find the one deal-relevant item.
Unused metadataTargetCompany.industry is collected in settings but never passed into the prompt (ai_client.py only receives target_company_name) — industry-specific framing is thrown away.
No cross-source dedupeIngestion dedupes only on exact article url (ingestion.py:52); the same story from two outlets produces two full Mistral calls and two near-duplicate signals in the feed.
No memory across signalsEach article is summarized in isolation — the model has no idea a target company already generated 3 signals this month, so it can't say "this is a continuation of X."
No cost trackingThe original plan called for logging token usage per call for cost tracking (planning.html:422) — never implemented; spend is invisible until the Mistral invoice arrives.
Single fixed model for every callSimple filtering ("is this article even worth summarizing?") uses the same large model as full synthesis — no cost/latency tiering.
One-size-fits-all outputA single outreach snippet regardless of channel (email vs. LinkedIn vs. cold call) — noted as a future idea in planning.html:607, never built.
No feedback loopMarking a signal "dismissed" in the dashboard has no effect on future summarization/filtering — also called out in planning.html:606 as deferred.

4. Guiding principle for new ideas

Sales reps are not paid to read news — they're paid to have relevant, timely conversations. Every idea below is filtered through: does this reduce the time between "news happens" and "rep has a specific, confident reason to reach out"? Ideas that just generate more AI text without improving signal, ranking, or specificity were deliberately left out.

5. Signal quality & prioritization

Urgency/relevance scoring High impact

Add a numeric relevance_score (1–5) and a short signal_type (e.g. funding, leadership_change, expansion, hiring_surge, competitor_mention, partnership) to the existing structured-output contract — one extra field, same call, no extra Mistral spend. This directly fixes the "everything is equal weight" gap: the dashboard can sort/filter by score and type, and the email digest can lead with a "Top signals today" section instead of a flat chronological list.

Implementation: extend AISummaryResult and the system prompt with the two new fields and a short rubric (e.g. "5 = a clear, immediate buying trigger directly tied to our offering; 1 = tangentially related background news"); add relevance_score and signal_type columns to signals; sort SignalsFeed.tsx and digest.py's query by score.

Buying-signal entity extraction High impact

Have the same call extract structured facts a rep can scan in two seconds: dollar amounts mentioned, people/titles named, and a boolean-style tag set for common outreach triggers (new funding round, new office/expansion, executive hire, layoffs, product launch, competitor switching). This turns free-text summaries into filterable dashboard facets — "show me every funding-round signal from this month across all target companies" — which is exactly the kind of query a sales manager asks in a pipeline review.

Implementation: Mistral's JSON mode already supports nested objects; add an entities object to the schema and a signal_tags array. Store tags in a join table or a Postgres array column for filtering.

Grounding/confidence guardrail Medium impact

Ask the model to include a supporting_quote field — a short verbatim excerpt from the article description that justifies the business_relevance claim — and a confidence value. Low-confidence signals (or ones where the quote can't be found in the source text) get a "low confidence" badge in the UI instead of being presented with the same authority as a well-grounded one. This matters for trust: a rep who acts on a hallucinated claim once stops trusting the whole feed.

6. Retrieval, memory & embeddings

Semantic dedupe across sources with Mistral embeddings High impact

Today dedupe is exact-URL-match only, so the same event covered by three outlets produces three Mistral calls and three near-identical signals in a rep's feed — actively hurting trust in the product. Embed each new article's title+description with mistral-embed, compare against recent articles for the same target company (cosine similarity via pgvector), and either skip re-summarizing near-duplicates or merge them into one signal citing multiple sources. This is both a quality win and a direct cost reduction (fewer chat-completion calls).

Account memory / continuity-aware summaries Medium impact

Before summarizing a new article, retrieve the target company's most recent 2-3 prior signals (simple recency query, no vector search needed at this scale) and include them as context in the prompt. Ask the model to note continuity where relevant ("this is the second leadership change at Acme Corp in two months — may indicate a broader reorg"). Turns isolated blips into a narrative a rep can use, and is the single biggest lever for the "why does this matter more than yesterday's 40 headlines" problem.

Conversational account Q&A Medium impact

On the Signal Detail / target-company page, let a rep type a free-form question ("has Acme Corp made any executive hires in the last quarter?") and answer it with a Mistral chat call grounded in that target company's stored signals/articles (retrieved by recency or embedding similarity). Turns the accumulated signal history into an on-demand research assistant instead of something a rep has to scroll through manually before a call.

7. Sales-workflow fit & personalization

Multi-channel outreach variants High impact

Generate outreach_snippet_email, outreach_snippet_linkedin, and outreach_snippet_call_opener instead of one generic snippet — explicitly called out as a future idea in the original plan (planning.html:607) and still one of the highest-leverage, lowest-effort additions available: same call, richer structured output, directly usable by reps in whatever channel they're actually working that day.

Use the industry field that's already collected Medium impact

TargetCompany.industry exists in the schema and settings UI today but is silently dropped before reaching the prompt. Passing it through lets the model tailor business_relevance with industry-specific framing (e.g. compliance angle for healthcare, supply-chain angle for manufacturing) instead of generic language. Essentially free — no schema change, just plumb an existing field into summarize_article().

Relevance feedback loop Medium impact

The dashboard already has a "dismissed" status on signals (SignalStatus.DISMISSED) but it currently has zero effect on future output — also called out as deferred in planning.html:606. Feed a rolling sample of dismissed vs. reviewed signals back into the prompt as few-shot examples ("these were marked not relevant — avoid surfacing similar low-value angles"), or use dismissal patterns to tune the relevance-scoring rubric per workspace over time.

Weekly account intelligence briefing Medium impact

Once a week, batch all of a target company's signals from that period into a single synthesis call producing one "account brief": what changed, the strongest outreach angle, and suggested next action. This is the kind of digestible, high-altitude output a sales manager (not just an individual rep) would actually forward to leadership — a different value proposition than the per-article feed.

8. Cost, resilience & efficiency

Two-stage triage before full summarization High impact

Run a cheap, fast mistral-small-latest classification pass first ("is this article relevant enough to a sales outreach use case to warrant full summarization? yes/no + one-line reason"), and only call mistral-large-latest for articles that pass. NewsAPI keyword matching is coarse, so a meaningful fraction of ingested articles are likely borderline-relevant; this directly reduces spend on the expensive model and reduces feed noise at the source, rather than filtering after the fact.

Token usage & cost logging Medium impact

The original architecture doc explicitly wanted this ("log token usage per call for later cost tracking," planning.html:422) and it was never built. Mistral's chat-completions response includes a usage object (prompt/completion/total tokens) that AIClient._call() currently discards entirely. Capture it per call, store on the Signal or a separate log table, and surface a simple spend dashboard — this is a prerequisite for safely turning on any of the higher-volume ideas above (weekly briefings, two-stage triage, embeddings) with visibility into what they cost.

Graceful model fallback Lower impact

docs/security-review.html already flags single-provider dependency as a residual risk. Rather than a full multi-provider abstraction, a smaller step: on repeated mistral-large-latest failure, retry once against mistral-small-latest before giving up on the article — cheap resilience win using a model already in the same account.

Streaming for the manual "run now" trigger Lower impact

POST /ingestion/run-now currently blocks until the whole batch finishes with no progress feedback. Streaming Mistral's response token-by-token (via SSE to the frontend) for a manually-triggered run would make the "run now" demo/testing path feel responsive — a UX polish item, not a core insight-quality lever.

10. Suggested phasing

Phase 1 — Same-call wins Weeks 1-2

Extend the existing structured-output schema in one pass, since it's the same Mistral call either way: relevance_score, signal_type, multi-channel outreach variants, the industry field, and token-usage logging. This alone turns the flat, unranked feed into a prioritized one and should be the highest ratio of sales value to engineering effort available.

Phase 2 — Noise reduction & cost control Weeks 3-5

Two-stage triage (small-model pre-filter) and semantic dedupe via mistral-embed + pgvector. Both reduce noise in the feed and reduce/offset the added spend from Phase 1 and 3's extra calls.

Phase 3 — Context & continuity Weeks 6-8

Account memory in the per-article prompt, the weekly account briefing job, and the relevance feedback loop. These need the scoring/tagging groundwork from Phase 1 and the token-cost visibility from Phase 1 to be rolled out with confidence.

Phase 4 — Interactive intelligence Later

Conversational account Q&A and streaming UX. Highest engineering cost, best done once the underlying signal data is richer (Phases 1-3) and there's a stored corpus worth querying conversationally.

11. Open questions

Embedding infra — semantic dedupe assumes pgvector is acceptable to add to the existing PostgreSQL instance; confirm no constraint against adding extensions on the target deployment (self-hosted Docker Compose, so this should be low-risk, but worth confirming before committing to the approach).
Cost budget — several ideas (triage pass, embeddings, weekly briefings) add Mistral spend on top of today's one-call-per-article baseline. Token-usage logging (Phase 1) should ship first so subsequent phases can be evaluated against real cost data rather than estimates.
Scoring rubric ownership — relevance scoring is inherently subjective per workspace/offering; the initial rubric will need calibration against ProAir's actual target companies and may need a per-workspace tuning mechanism once multi-tenant support (already on the roadmap in planning.html) lands.
