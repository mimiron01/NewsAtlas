1. Problem statement

Sales teams need an automated way to receive relevant news about the companies they target, so they can act on timely, well-framed outreach opportunities instead of manually monitoring the news. NewsAtlas ingests news about a configurable list of target companies, uses an AI model to summarize each article and connect it to the seller's own product/service offering, and delivers those "signals" both in a web dashboard and via a daily email digest — including ready-to-use outreach snippets.

2. Goals & non-goals

Goals (MVP)

  • Let a user configure their company's offering/description in settings (used as AI context).
  • Let users add/manage a list of target companies with search filters (keywords, aliases, industry).
  • Automatically poll a news API on a schedule and ingest new matching articles per target company.
  • Summarize each article with AI (Mistral), producing: a plain-language summary, an explanation of why it matters for business/outreach, and a ready-to-send outreach snippet.
  • Show ingested & summarized "signals" in a web dashboard, filterable by company/status.
  • Send a daily email digest of new signals to all workspace users.
  • Basic multi-user login (email + password) on a single shared workspace.

Non-goals (explicitly deferred)

  • Multi-tenant/multi-organization support (each with isolated settings) — single workspace for now.
  • CRM integrations (Salesforce/HubSpot push) — future phase.
  • Per-user assignment of target companies / role-based permissions beyond basic auth.
  • Sentiment scoring, lead scoring, or predictive analytics on signals.
  • Real-time/streaming ingestion (push webhooks) — polling is sufficient for MVP.

3. Key decisions

Decisions below were confirmed for this planning pass, or chosen as sensible defaults where the requester had no preference (marked default). Defaults can be revisited in a future session without changing the overall architecture.

Backend: Python + FastAPI. default Chosen over Node/Django because it pairs naturally with background job scheduling (APScheduler), async HTTP calls to NewsAPI/Mistral, and typed request/response models via Pydantic.
Frontend: React + TypeScript (Vite), separate SPA calling the FastAPI backend over REST. default
Database: PostgreSQL. default Relational schema fits the entities well (users, target companies, articles, signals) and supports full-text search on article content later.
Tenancy: Single shared workspace (one company profile/offering), multiple user logins. Matches the initial ProAir deployment. default
Deployment: Docker Compose (app + Postgres) as the baseline, portable to any VPS or cloud VM. Cloud-managed services can be swapped in later without changing app code. default
News source: NewsAPI.org, accessed through a thin NewsClient abstraction so the provider can be swapped later. confirmed
AI provider: Mistral API (e.g. mistral-large-latest) for summarization, business-relevance explanation, and outreach-snippet generation, using the company's offering/description from settings as grounding context. confirmed
Scheduling: Background polling job (interval configurable, default every 6 hours) ingests + summarizes new articles; a separate daily job (default 07:00 workspace-local time) compiles and sends the email digest. Both run via APScheduler inside the FastAPI process for MVP — no separate task queue infra needed yet. confirmed
Auth: Simple email + password accounts (hashed with bcrypt/argon2, JWT session tokens) for all workspace users. confirmed

6. Core workflows

6.1 Settings configuration

User edits workspace_settings.offering_description and company_name from a Settings page. This text is injected into every AI prompt as grounding context, so it should describe what the company sells and who it's a good fit for.

6.2 Target company management

User adds a target company (name + optional keywords/aliases/industry) from a Settings/Targets page. Each active target company defines a NewsAPI query used by the ingestion job.

6.3 Ingestion pipeline (scheduled)

  1. For each active target_company, build a NewsAPI /v2/everything query from name + keywords, sorted by publishedAt, restricted to since-last-run window.
  2. For each returned article, skip if url already exists in articles (dedupe).
  3. Insert new rows into articles.
  4. Enqueue each new article for AI processing (synchronously in the same job for MVP — no separate queue needed at this scale).

6.4 AI processing → signal creation

See Section 7 for prompt design. Result is inserted as one row in signals with status = new.

6.5 Dashboard viewing

Frontend lists signals (joined with article + target company), newest first, filterable by target company and status. Clicking a signal shows full detail: original article link, AI summary, business relevance, outreach snippet (with a "copy" button), and status controls (mark reviewed/archived/dismissed).

6.6 Daily email digest (scheduled)

See Section 8.

7. AI summarization pipeline

Each new article is sent to Mistral with a single structured-output prompt combining:

  • the article's title + description/content snippet from NewsAPI,
  • the target company's name,
  • the workspace's offering_description from settings.

Expected structured JSON response (via Mistral's JSON mode / tool-calling):

{
  "summary": "2-4 sentence plain-language summary of the news article",
  "business_relevance": "Why this news matters for outreach — what changed for the "
                          "target company and how our offering connects to it",
  "outreach_snippet": "A short, ready-to-send paragraph referencing the news, suitable "
                        "for an outreach email or LinkedIn message"
}

Implementation notes:

  • Wrap the Mistral call in an AIClient service class so the prompt/response contract is isolated from the rest of the app.
  • Validate the JSON response against a Pydantic schema; on parse failure, retry once with a stricter instruction, then mark the article as failed (skip signal creation) rather than crashing the ingestion job.
  • Log token usage per call for later cost tracking.

8. Email digest design

  • Runs once daily (default 07:00) via APScheduler CronTrigger.
  • For each user, selects all signals with status = new created since that user's last digest_log entry (or since workspace creation if none).
  • If zero new signals for a user, no email is sent that day.
  • Email body (HTML) lists each signal: target company name, article title + link, AI summary, business relevance explanation, and the outreach snippet in a copy-friendly block; links back to the signal's detail page in the dashboard.
  • Sending is done through a generic EmailClient wrapping SMTP (host/port/user/pass from config), so the concrete provider (SendGrid, Postmark, AWS SES, etc.) is just an SMTP credential swap — provider choice deferred, see Section 14.
  • After sending, insert a digest_log row and flip included signals to status = reviewed... or keep status independent of "was emailed" (recommend: add a separate emailed_at timestamp on signals rather than overloading status, to keep dashboard state and email state orthogonal).

  • Login page — email/password.
  • Signals Feed (home) — filterable/sortable list of signals, status badges.

13. Implementation roadmap

Phase 1 Foundations

  • Repo scaffolding: backend (FastAPI) + frontend (React/Vite) + docker-compose + Postgres
  • DB schema + Alembic migrations for all core tables
  • Auth: signup/login endpoints, password hashing, JWT sessions, frontend login page
  • Settings API + UI: company profile / offering description
  • Target companies CRUD API + UI

Phase 2 Ingestion pipeline

  • NewsClient wrapping NewsAPI.org /v2/everything
  • Query builder from target company name + keywords
  • Dedupe by URL, store into articles
  • Manual "run ingestion now" endpoint for testing before scheduling is wired up

Phase 3 AI summarization

  • AIClient wrapping Mistral API with structured JSON output
  • Prompt template combining article + offering_description + target company
  • Pydantic validation of AI response, retry-once-then-skip on failure
  • Persist result into signals

Phase 4 Dashboard frontend

  • Signals feed page: list, filter by company/status, pagination
  • Signal detail page: summary, relevance, outreach snippet with copy button
  • Status transitions (reviewed/archived/dismissed)

Phase 5 Scheduling & email digest

  • APScheduler wiring for recurring ingestion job (configurable interval)
  • EmailClient (SMTP) + HTML digest template
  • Daily digest job: per-user new-signal selection, send, log in digest_log
  • Manual "send digest now" endpoint for testing

Phase 6 Hardening & deployment

  • Error handling/retries around external API calls (NewsAPI, Mistral, SMTP)
  • Rate-limit awareness for NewsAPI free tier
  • Structured logging + basic monitoring of scheduled jobs
  • Docker Compose production config, deployment docs

14. Open questions & assumptions for follow-up sessions

Email/SMTP provider not chosen yet — plan uses a generic SMTP client so any provider (SendGrid, Postmark, AWS SES, Mailgun) works via config; pick one before Phase 5.
NewsAPI.org plan tier — the free "Developer" plan disallows production/commercial use and delays articles ~24h; confirm whether a paid plan is needed before going live.
Mistral model & cost — default assumption is mistral-large-latest; confirm API key/account and evaluate whether a smaller model (e.g. mistral-small-latest) is sufficient/cheaper for summarization at scale.
Digest scope — MVP assumes every user receives the same digest (all new signals across all target companies). Per-user subscription to specific target companies is a plausible fast-follow, not required for MVP.
Time zone for digest_send_time — assumed a single workspace-wide time zone setting; per-user time zones deferred.

15. Future enhancements (explicitly out of MVP scope)

  • Multi-tenant support (multiple organizations, isolated data)
  • CRM push integrations (Salesforce, HubSpot, Outreach.io)
  • Per-user target-company subscriptions and notification preferences
  • Feedback loop: let users mark signals as "not relevant" to tune future filtering/prompting
  • Multiple outreach snippet variants (email vs. LinkedIn vs. cold call opener)
  • Sentiment/urgency scoring, lead scoring integration
  • Swappable news providers / multi-source aggregation with cross-source dedupe
