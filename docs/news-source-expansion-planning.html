NewsAtlas

News source expansion — adding Google News RSS and NewsData.io alongside NewsAPI.org

Status: Planning / pre-implementation
Builds on: planning.html (base architecture), mistral-ai-roadmap.html (AI layer), role-based-company-management-planning.html (roles/follows)
Doc purpose: Source of truth for implementing multi-source news ingestion in a future session

NewsAtlas today ingests news for target companies from exactly one provider, NewsAPI.org, then runs every article through a Mistral-embedding-based dedupe pass, an optional cheap-model relevance triage, and a full Mistral summarization call before storing the result as a Signal shown in the dashboard and daily digest (see planning.html §6 and mistral-ai-roadmap.html §2 for the current implementation). This document plans adding two more sources on top of that pipeline: Google News RSS (free, keyless, public feed) and NewsData.io (paid API, assumed to be an admin-held commercial account). Both plug into the existing fetch → dedupe → summarize → frontend pipeline rather than requiring a new one.

1. Problem statement

NewsAPI.org's free/developer tier delays articles ~24h and its paid tier is priced per request volume; relying on a single provider also means a single point of failure (rate limits, an outage, or a pricing change stops all ingestion) and a single editorial slice of coverage. Google News RSS adds a free, broad-coverage, real-time source with no API key and no per-request cost, at the expense of thinner article snippets and coarser querying. NewsData.io adds a second commercial provider whose paid tier offers things NewsAPI.org doesn't — full article text (not just a snippet), native duplicate-content removal, native sentiment/AI tagging, and a historical archive going back years — which materially improves summarization quality and lets NewsAtlas backfill a target company's recent history the moment it's added, instead of only seeing news from that point forward.

2. Goals & non-goals

Goals

  • Add GoogleNewsRSSClient and NewsDataClient as new provider implementations feeding the exact same Article → dedupe → triage → summarize → Signal pipeline that NewsAPI.org uses today — no parallel pipeline.
  • Let an admin independently enable/disable each of the three sources workspace-wide, and configure NewsData.io's API key in-app (mirroring the existing Mistral key pattern).
  • Cross-source duplicate coverage of the same story collapses into one Signal, not one per source.
  • Lean into NewsData.io's paid-tier features: full-content ingestion (better-grounded summaries), native duplicate removal, native sentiment/tag capture as extra metadata, and a one-time historical backfill when a target company is created.
  • Give the admin visibility into NewsData.io credit/request consumption, since it's a metered paid account they're paying for directly (unlike the current invisible-cost NewsAPI.org calls).
  • Per-provider failure isolation: one source erroring out (bad key, outage, quota exceeded) must not block ingestion from the other two.

Non-goals (this pass)

  • Per-target-company source selection — sources are enabled/disabled workspace-wide, matching today's single-config pattern (confirmed during planning; see §3).
  • Adding ticker/domain/canonical-ID fields to TargetCompany — all three sources are queried the same way NewsAPI.org is today, by name + free-text keywords.
  • Replacing NewsAPI.org — it stays as a third, equally-weighted source.
  • A pre-AI "raw articles" browsing UI — ingested articles remain visible to users only after becoming a Signal, same as today.
  • Hard credit-budget cutoffs (auto-pausing NewsData.io mid-run on a spend threshold) — flagged as a future enhancement, see §15.

3. Key decisions

Decisions below were confirmed during this planning pass.

Source relationship: Google News RSS and NewsData.io are added alongside NewsAPI.org, not as replacements. All three can be active at once; a target company's articles are simply the union of whatever enabled sources return for it. confirmed
Source granularity: a single workspace-wide on/off toggle per source (three booleans in workspace_settings), not a per-target-company selection. Matches the existing single-tenant, single-config pattern and avoids adding UI/schema surface to every company row. confirmed
NewsData.io depth: the plan leans into the paid tier — full-content ingestion, native removeduplicate param, native sentiment/tag capture, and a one-time historical archive backfill on company creation — rather than treating it as a plain headline+snippet source. confirmed
Google News RSS query shape: mirrors the existing NewsAPI.org query-building pattern (name + keywords, OR-joined, quoted multi-word terms), translated into Google News RSS's ?q= search syntax, with an admin-configurable default country/language (default US/en) rather than per-company locale. confirmed
Dedupe stays semantic, not source-aware: the existing mistral-embed + cosine-similarity dedupe in ingestion.py already compares every new article against recent articles for the same target company regardless of source — this is exactly the "let Mistral deduplicate" step the requester described for the Google News RSS workflow, and it needs no new logic to also cover NewsData.io and cross-source duplicates. confirmed
URL uniqueness stays global: articles.url keeps its existing unique constraint as a first, free dedupe pass across all three sources. NewsAPI.org and NewsData.io both tend to return the publisher's canonical URL, so the same story from both often collapses here for free. Google News RSS returns a news.google.com/rss/articles/... redirect URL instead of the direct publisher URL, so cross-source dedupe with a Google News RSS hit almost always falls through to the semantic (embedding) pass rather than the URL check — expected and fine, just worth knowing why. confirmed
NewsAPI.org gets a matching enable toggle (newsapi_enabled, default true) for symmetry with the two new sources, even though its behavior is unchanged — this gives the admin one consistent three-source panel instead of two toggles plus one implicit always-on source. default

6. Data model changes

articles modified

external_sentimenttext, nullablenew; NewsData.io's own sentiment label (positive/neutral/negative) when the account's sentiment add-on is enabled. Stored verbatim as extra metadata — separate from and not a substitute for Mistral's own confidence/relevance_score on the resulting signal.

Why keep external metadata separate from Mistral's fields rather than merging them: external_sentiment/external_tags come from a different model/vendor than signal_type/entities/confidence, and conflating them would make it unclear which system is responsible for an eventually-wrong tag. They're additive read-only context, not inputs the pipeline currently reasons over — feeding them into the Mistral prompt is a plausible fast-follow, listed in §15.

7. New provider clients

7.1 Shared query builder

NewsClient._build_query() (name + keywords → an OR-joined, quoted search string) is extracted into a small services/news_query.py helper (build_or_query()) so NewsClient, GoogleNewsRSSClient, and NewsDataClient share one implementation instead of three near-identical copies.

7.2 GoogleNewsRSSClient new

services/google_news_rss_client.py, same fetch_articles(name, keywords, since) -> list[NewsArticle] contract as NewsClient.

  • Builds https://news.google.com/rss/search?q={query}&hl={lang}&gl={country}&ceid={country}:{lang} using the shared query builder and the workspace's configured country/language.
  • Fetches and parses with feedparser (new dependency — no RSS parser exists in the codebase today; chosen over hand-rolled XML parsing for the same reason the codebase already leans on well-tested libraries elsewhere).
  • Per entry: title, link (the Google redirect URL — stored as-is; see §3 on why this limits URL-level cross-source dedupe), description (HTML snippet — stripped to plain text), published parsed to a timezone-aware datetime, source_name parsed from the entry's <source> tag (Google News RSS includes this, unlike a plain title-suffix parse).
  • Google News RSS has no server-side date filter, so entries older than since are dropped client-side after parsing.
  • No API key, so NewsClientError is only raised on a request/parse failure, not a missing-credential check.
  • Reuses the existing http(s)-only URL guard from NewsClient._parse_article (stored-XSS protection) — factor that check into a shared helper alongside the query builder rather than duplicating it a third time.

7.3 NewsDataClient new

services/newsdata_client.py, extends the same contract with optional richer output.

  • fetch_articles(name, keywords, since, *, full_content, use_native_dedupe) calls https://newsdata.io/api/1/news (previously /latest in older API versions) with q (shared query builder), apikey, language, from_date/to_date (NewsData.io supports real date filtering, unlike Google News RSS), removeduplicate=1 when enabled, and full_content=1 when enabled.
  • Parses each result into an extended dataclass carrying the same base fields as NewsArticle plus full_content: str | None, sentiment: str | None, tags: list[str] | None — the latter three read defensively (.get() with no error if absent), since they're only present when the account's plan/add-ons include them.
  • Paginates via NewsData.io's nextPage cursor, capped at a small fixed number of pages per target company per run (e.g. 2) to bound credit spend from a single busy company — this cap is the main defense against runaway paid-API usage until the harder budget-alerting idea in §15 is built.
  • fetch_archive(name, keywords, since, until) — a second method hitting https://newsdata.io/api/1/archive (paid-only, historical coverage), used solely by the one-time backfill workflow (§9), not the routine polling loop.
  • Every call returns the provider's reported credit cost (present in the response envelope) so the caller can write a NewsSourceUsageLog row.

8. Ingestion orchestration changes

Changes are confined to services/ingestion.py; the per-article dedupe/triage/ summarize logic in _process_new_articles() is untouched, since it already operates on Article rows regardless of where they came from.

  1. run_ingestion() builds a list of (source_enum, client) pairs from whichever of newsapi_enabled / google_news_rss_enabled / newsdata_enabled are true, instead of constructing a single hardcoded NewsClient.
  2. For each active target company, the existing fetch step becomes a loop over enabled providers. Each provider's fetch_articles() call is wrapped in its own try/except so one provider's failure (revoked NewsData.io key, Google News RSS transiently blocking, NewsAPI.org rate limit) appends to errors and is skipped, while the others still run — matching the existing per-target-company error isolation, just one level deeper.
  3. Every fetched article is tagged with Article.source before insertion. The existing URL-uniqueness check (db.query(Article).filter(Article.url == ...)) is unchanged and now doubles as the first cross-source dedupe pass (see §3).
  4. After insertion across all enabled providers for that company, the rest of the function — embedding, semantic dedupe against recent articles for the company, triage, summarization — runs exactly as it does today, unaware of and unaffected by how many sources contributed the batch.
  5. IngestionRunResult gains a by_source: dict[str, int] breakdown (articles fetched per source) so the manual "run now" UI and logs show which providers are contributing, which matters once three sources can silently succeed/fail independently.
  6. A NewsSourceUsageLog row is written per provider call (mainly useful for NewsData.io's metered credits; recorded for all three for a consistent history).

9. NewsData.io-specific workflow

This is the "fitting workflow assuming a paid NewsData.io account" the requester asked for — the parts that go beyond treating NewsData.io as a drop-in NewsAPI.org replacement.

9.1 Full-content-grounded summarization

When newsdata_full_content_enabled is on and a given article has full_content, AIClient.summarize_article() (and the triage call) are given the full body instead of the short description snippet used for NewsAPI.org/Google News RSS articles. This directly improves business_relevance and supporting_quote grounding — a quote pulled from a full article is far more likely to be a genuine, checkable excerpt than one inferred from a two-sentence snippet. No schema change to AISummaryResult is needed — this is a change to what text is placed in the prompt's "Article description" slot, source-conditional on full_content being present.

9.2 Native duplicate removal as a pre-filter

removeduplicate=1 asks NewsData.io to collapse duplicate coverage on their side before it's even returned. This doesn't replace the existing embedding-based semantic dedupe (which still needs to catch duplicates across NewsData.io, NewsAPI.org, and Google News RSS) — it reduces how many near-duplicate NewsData.io articles reach that stage in the first place, which matters once three sources can each surface the same wire story.

9.3 Native sentiment/tags as auxiliary metadata

external_sentiment/external_tags are stored on the Article row and surfaced read-only on SignalDetail.tsx (e.g. a small "NewsData.io sentiment: negative" line) alongside Mistral's own confidence/relevance_score/ signal_type. Kept separate rather than merged into the AI-generated fields (see the data-model assumption in §6).

9.4 One-time historical backfill on company creation

  1. When a target company is created (self-serve or admin-assigned, get_or_create_company() in services/company_follows.py) and newsdata_enabled and newsdata_backfill_days > 0, a FastAPI BackgroundTasks job is scheduled — not run inline — since an archive pull covering weeks of coverage plus embedding + summarizing every result can take meaningfully longer than a normal request/response cycle.
  2. The background task calls NewsDataClient.fetch_archive() for a window of [now - newsdata_backfill_days, now], tags results source = newsdata, and feeds them through the exact same insertion + dedupe/triage/summarize path as routine ingestion — a backfilled company shows up with a populated history instead of a slow trickle starting from zero.
  3. Only runs once per company (guarded by, e.g., a backfilled_at timestamp on TargetCompany — new nullable column — so re-activating a paused company doesn't re-trigger it and re-spend archive credits).
  4. Existing companies created before this feature ships are not retroactively backfilled automatically — an admin can trigger it manually per company (new POST /target-companies/{id}/backfill, admin-only, same cooldown/rate-limit posture as /ingestion/run-now) if they want the history for a company that predates the feature.

11. Frontend changes

  • SettingsProfile.tsx gains a new "News sources" panel (same card style as the existing "AI configuration" panel): three toggle rows — NewsAPI.org (on by default), Google News RSS (off by default, with country/language selects revealed when enabled), NewsData.io (off by default, with an API key input reusing the exact password-field + status-text + "clear in-app override" pattern already built for mistral_api_key, plus toggles for full-content, native dedupe, and a backfill-days number input).
  • SignalsFeed.tsx / SignalDetail.tsx gain a small source badge per signal (NewsAPI.org / Google News / NewsData.io) sourced from the underlying Article.source — same visual treatment as the existing status/signal-type badges, not a new design pattern.
  • SignalDetail.tsx additionally renders NewsData.io's external_sentiment/external_tags as a small read-only metadata line when present.
  • New admin-only page (or a second tab on the existing AIUsage.tsx, renamed to a shared "Usage" page) showing /news-usage data — request/credit consumption per source over time, mirroring the existing AI usage page's table/summary layout.
  • SettingsTargets.tsx: a company created while NewsData.io backfill is configured shows a brief "backfilling historical coverage..." indicator (polls or just informs the user it's running in the background) instead of appearing to have silently ingested nothing at first.

12. Migration plan

  1. Alembic migration: add articles.source (enum, not null, default 'newsapi' — backfills existing rows correctly since NewsAPI.org is the only source that has ever run), articles.full_content, articles.external_sentiment, articles.external_tags.
  2. Alembic migration: add all new workspace_settings columns from §6, plus target_companies.backfilled_at (nullable timestamptz).
  3. Alembic migration: create news_source_usage_logs.
  4. No backfill-of-behavior step is needed (unlike the role-based-access migration) — every new column has a safe default that preserves today's single-source behavior until an admin opts into the new sources.

14. Open questions & assumptions for follow-up sessions

Google News RSS terms of use — this is a widely-used but unofficial feed; Google doesn't publish a supported API or formal quota for it. Confirm this is an acceptable risk for the deployment before enabling it by default for any workspace, and keep polling conservative (bounded by the existing ingestion interval, no extra hidden polling).
NewsData.io plan tier — full-content, sentiment/AI-tag, and archive access are gated to different paid tiers (Basic/Professional/Corporate). The plan assumes the admin's account includes all three; each is independently toggle-able (newsdata_full_content_enabled, sentiment capture is always attempted but tolerates absence, newsdata_backfill_days) so a lower tier degrades gracefully instead of erroring.
Credit budget alerting — Phase 3-5 add a hard per-run page cap on NewsDataClient pagination, but there's no workspace-wide "stop calling NewsData.io once N credits are used this month" cutoff. news_source_usage_logs (Phase 4) is a prerequisite for building that later with real data instead of guessing at limits.
Embedding dedupe threshold with full content — mistral_dedupe_similarity_threshold (default 0.90) was tuned against short title+description embeddings. Once NewsData.io's full_content (much longer text) is embedded, cosine similarity behavior may shift; re-validate the default threshold once Phase 4 ships, rather than assuming it transfers unchanged.
Google News RSS redirect URL stability — the news.google.com/rss/articles/... link stored as articles.url is a redirect, not the canonical publisher URL; assumed stable enough to remain clickable from a digest email sent hours or days later. Worth spot-checking in practice once this ships, since a broken link in an email a rep is about to use for outreach is a worse failure mode than a broken link in the dashboard.

15. Future enhancements (explicitly out of scope here)

  • Per-target-company source selection, once/if a real need for it shows up (e.g. one company needs NewsData.io's superior coverage in a non-English market while others don't).
  • Hard credit-budget cutoffs / alerting for NewsData.io spend, once news_source_usage_logs has enough real data to set sane thresholds.
  • Feeding external_sentiment/external_tags into the Mistral summarization prompt itself (as extra grounding context) rather than only displaying them alongside the AI output.
  • Ticker/domain fields on TargetCompany for more precise querying against NewsData.io's domain/ticker-style filters, if free-text name+keyword queries prove too noisy at scale.
  • Additional NewsData.io paid endpoints not covered here (e.g. its crypto-news or podcast-transcript verticals), if a future use case calls for them.
