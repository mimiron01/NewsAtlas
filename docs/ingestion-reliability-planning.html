This document covers four related requests against the ingestion pipeline (backend/app/services/ingestion.py and friends):

  1. Give admins a way to stop an in-progress fetch run if something goes wrong.

1. How a fetch run actually executes today

Everything else in this document depends on understanding this precisely, so it's worth stating up front, with evidence:

  • POST /ingestion/run-now (backend/app/api/ingestion.py:27-51) creates an IngestionRun row (status=running) and hands the work off via genuine FastAPI background_tasks.add_task(execute_ingestion_run, run.id) — not an awaited inline call. Starlette runs this after the HTTP response is already sent, on the same process/event loop, but with no reference back to the request or the client connection at all.
  • execute_ingestion_run(run_id) (backend/app/services/ingestion_runs.py:77-121) takes a plain UUID, opens its own fresh SessionLocal(), and runs the whole pipeline (run_ingestion) synchronously to completion, updating the same IngestionRun row's progress fields as it goes (ProgressTracker, ingestion_runs.py:19-39 — one small commit per update, so a poller never reads a half-written row).
  • The scheduled job (backend/app/services/scheduler.py:16-27) calls the exact same execute_ingestion_run(run_id) from an APScheduler background thread — there is only one execution engine for both triggers.
  • Inside the pipeline (backend/app/services/ingestion.py), work commits incrementally, not as one long transaction: a commit after each company's new Article rows are inserted (line 269), and a commit after every article is skipped or turned into a Signal (_skip_article, line 486; signal creation, line 461). There is no point in the run where a half-finished unit of work is uncommitted for long.
Net effect: a fetch run is a purely server-side background job from the moment the trigger request returns. It has no dependency on any browser tab, WebSocket, or open connection — closing the tab, navigating away within the SPA, or even losing network entirely on the client has zero effect on the run in progress.

2. Answer: does the run survive leaving the page?

Yes, unconditionally, by design. Evidence from the frontend confirms this wasn't an accident:

  • frontend/src/pages/SignalsFeed.tsx:126-129 — on every mount, the page calls pollIngestionStatus() once unconditionally, specifically so that reloading the page (or opening it fresh in a new tab, or a different admin loading it) picks up a run already in flight rather than only ever reacting to this browser's own trigger click.
  • Polling (SignalsFeed.tsx:131-136) only runs a setInterval while isRunningIngestion is true, and is torn down on unmount — but that only stops this browser tab from refreshing its own view of the status; it never talks to or affects the backend job itself.
  • GET /ingestion/status (backend/app/api/ingestion.py:54-62) always reflects the latest IngestionRun row from the DB — reloading the page, or any other authenticated user loading the Signals page, sees the same live progress.

So today: a user can close the tab, log out, or go to a completely different page, and the fetch run keeps running exactly as if they'd stayed. The only thing that changes is nobody's watching it update live — the next time anyone opens the Signals feed (or the admin Logs tab), they'll see it either still running or already finished/failed.

Worth surfacing, not fixing: because the progress bar only shows on the Signals feed page, an admin who triggers a run and then navigates to Settings has no visual indicator anywhere else in the app that a run is still in flight. This isn't broken — the run itself is fine — but a small nav badge/toast ("still fetching…") would improve confidence over relying on "I trust it kept going." Noted as a candidate small addition, not required for correctness. See §11.

3. Admin-only stop/cancel control

There is currently no cancellation concept anywhere in the codebase — no STATUS_CANCELLED, no cancel endpoint, no stop button. It needs to be built from scratch, but the pipeline's existing structure makes this straightforward:

  • The outer loop over target companies (ingestion.py:142-165) and the inner loop over a company's new articles (_process_new_articles, ingestion.py:377) are both natural, already-frequent checkpoints — a progress update/commit already happens at nearly every iteration of both.
  • Because every meaningful unit of work commits immediately (§1), stopping between iterations leaves the DB in a fully consistent state. No rollback, no orphaned half-written signals, no distributed transaction concerns.
  • IngestionProgress (ingestion.py:38-46) is already an injected interface with a DB-backed implementation (ProgressTracker) — the natural place to add a cancellation check alongside the existing update()/append_error() methods.

3.1 Design

Cancellation is cooperative, not a hard kill. The pipeline is synchronous/blocking (plain httpx.get calls to news sources and Mistral) inside a background thread — there's no safe way to abort an in-flight HTTP call mid-request without threading/async-cancellation machinery well beyond what this codebase uses anywhere else. A "Stop" click sets a flag; the run checks it at the next checkpoint (before starting the next article, or before starting the next company) and exits cleanly. confirmed Worst-case latency before it actually stops is roughly one in-flight news-fetch or Mistral call plus its commit — a few seconds, not instant, and should be described that way in the UI ("Stopping…" rather than an immediate disappearance).
New column, not a repurposed status: add IngestionRun.cancel_requested: bool (default false) rather than overloading status with a transitional "cancel_requested" value. status gains exactly one new terminal value: STATUS_CANCELLED = "cancelled", set only once the loop actually observes the flag and exits. This keeps get_running_run() (status == STATUS_RUNNING, used by the manual-trigger overlap guard and the scheduler's overlap guard) correct with no changes — a cancel-requested-but-not-yet-stopped run is still, accurately, "running." confirmed
Endpoint is admin-only. POST /ingestion/runs/{run_id}/cancel behind require_admin (matching the existing admin-only pattern used for GET /ingestion/runs). Triggering a run stays open to any authenticated user (unchanged, existing behavior) — only stopping one is admin-gated, per the ask. confirmed
Applies uniformly to scheduled runs too. Since both the manual and scheduled trigger paths funnel through the same execute_ingestion_run/run_ingestion, no special-casing is needed — an admin can cancel a scheduled run exactly the same way as a manual one, by its run_id (there's only ever one running row at a time, so the frontend already has it via GET /ingestion/status). confirmed

3.2 Pipeline changes (sketch)

# ingestion.py — IngestionProgress gains a cancellation check
class IngestionProgress(Protocol):
    def update(self, **fields: object) -> None: ...
    def append_error(self, message: str) -> None: ...
    def should_cancel(self) -> bool: ...          # NEW

class _NullProgress:
    ...
    def should_cancel(self) -> bool:
        return False

class _IngestionCancelled(Exception):
    """Internal control-flow signal — never escapes run_ingestion()."""

def run_ingestion(db, ..., progress=None) -> IngestionRunResult:
    ...
    for idx, target_company in enumerate(target_companies, start=1):
        if progress.should_cancel():
            raise _IngestionCancelled()
        outcome = _ingest_target_company(...)
        ...
        progress.update(companies_processed=idx)
    ...

def _process_new_articles(db, *, progress, ...):
    ...
    for position, article in enumerate(new_articles):
        if progress.should_cancel():
            raise _IngestionCancelled()
        ...

# ingestion_runs.py — ProgressTracker implements the check; execute_ingestion_run
# catches the cancellation distinctly from a genuine crash

class ProgressTracker:
    ...
    def should_cancel(self) -> bool:
        # Single-column read; cheap, and consistent with the once-per-checkpoint
        # DB round trip `update()` already does.
        return bool(
            self.db.query(IngestionRun.cancel_requested)
            .filter(IngestionRun.id == self.run_id)
            .scalar()
        )

def execute_ingestion_run(run_id: uuid.UUID) -> None:
    db = SessionLocal()
    try:
        tracker = ProgressTracker(db, run_id)
        try:
            result = run_ingestion(db, progress=tracker)
        except _IngestionCancelled:
            db.query(IngestionRun).filter(IngestionRun.id == run_id).update({
                "status": STATUS_CANCELLED,
                "finished_at": datetime.now(timezone.utc),
                # whatever partial counters the pipeline had accumulated so far —
                # requires run_ingestion to report partial totals on cancellation
                # too, not just on the success path.
            })
            db.commit()
            return
        except Exception as exc:
            ...  # unchanged
        ...  # unchanged success path
    finally:
        db.close()
Partial-result reporting on cancel: today run_ingestion only returns an IngestionRunResult on the success path, accumulated in local variables across the whole loop. To report "cancelled after 3/8 companies, 4 signals created so far" instead of zeroing everything out, _IngestionCancelled needs to carry the accumulated totals at the point of cancellation (or run_ingestion catches it internally and returns a result with a cancelled=True flag rather than letting the exception propagate all the way to execute_ingestion_run). Either works; the internal-catch approach is slightly cleaner since it keeps execute_ingestion_run's exception handling limited to genuine crashes. Pick one during implementation — flagged here so it isn't decided ad hoc mid-PR.

4. Per-company 10-newest-articles cap

Today there is no cap at all, and it's a real gap, not just a missing nice-to-have:

SourceCurrent behaviorFile
NewsAPI.orgpageSize=20, sorted publishedAt desc by the provider — bounded, but still up to 20news_client.py:42-44
Google News RSSNo page-size param exists on this feed at all; returns the full feed (commonly ~100 entries), filtered only by since client-side. Not recency-sorted — RSS entry order is query-dependent, not a contractual newest-first order.google_news_rss_client.py:30-57
NewsData.ioUp to MAX_PAGES_PER_CALL = 2 pages, page size provider-controlled — this cap exists purely to bound paid-credit spend, not article volume.newsdata_client.py:12,92-121

After merging all enabled sources (ingestion.py:198-227), there's no sort or truncation applied to the combined list at any point — a single company could realistically see 100+ raw fetched items in one run, every one of which (after URL-dedupe) gets embedded, triaged, and potentially fully summarized by Mistral. That's the actual cost/noise problem a cap solves, independent of the misattribution bug in §5.

4.1 Where to apply the cap

Every raw article from every source already carries published_at: datetime | None, populated consistently by each client's own parser (news_client.py:69-75, google_news_rss_client.py:86-92, newsdata_client.py:129-137) — so the data needed for a recency cap exists before any DB row is ever created. Two candidate insertion points, with a recommendation:

OptionWhereTrade-off
A — cap raw fetch resultsRight after fetched_items is assembled across all providers (ingestion.py:227), before the URL/DB dedupe passSimple, but risks under-delivering: if several of the "10 newest raw" turn out to already be in the DB from a previous run, you could end up with fewer than 10 actually-new articles even though other, still-unseen articles existed further down the list.
B — cap after dedupe (recommended)After the URL/DB-existing filter builds the deduped candidate list, sort by published_at desc (treat None as lowest-priority/oldest) and slice to N before constructing Article ORM rows Guarantees up to N genuinely-new articles get processed per company per run — directly bounds the expensive embedding/triage/summarization calls, which is the actual goal ("limit articles per fetch run"). Requires restructuring the loop at ingestion.py:232-261 into two passes: build a deduped list of raw (source, fetched) survivors first, sort+slice, then construct/add() Article rows only for the surviving N — instead of constructing every Article row and only then discarding some.
Recommendation: Option B. It directly bounds AI spend per company per run (the load-bearing motivation) and matches "only take the newest" literally — of whatever's actually new, take the 10 newest. default

4.2 Configuration

New workspace setting, not a hardcoded constant: add workspace_settings.max_articles_per_company_per_run: int, default 10, following the exact same pattern as every other ingestion knob already on that model (newsapi_max_requests_per_day, newsdata_max_requests_per_day/minute, google_news_rss_max_requests_per_minute — all in backend/app/models/workspace_settings.py) — surfaced in Settings the same way, likely on the existing Sources tab (frontend/src/pages/settings/SourcesTab.tsx) near the other per-source limits. default
Confirm scope with the user: the ask was literally "the limit ... should be 10" — read here as "default to exactly 10," admin-adjustable like every other limit in this settings model, rather than hardcoded and unconfigurable. Flagging in case a hardcoded, non-configurable 10 was actually intended.
Backfill flow is out of scope for this cap: the one-time NewsData.io historical backfill (services/newsdata_backfill.py, triggered per-company after a company is created or manually via POST /target-companies/{id}/backfill) is deliberately meant to pull a large historical batch and is architecturally separate from the routine polling loop this cap targets. Confirm this reading — if the backfill should also be capped per company, that's a small additional change to newsdata_backfill.py using the same sort-and-slice helper.

Reporting: outcome.articles_fetched/by_source should keep reflecting the full raw volume actually pulled from providers (useful for the Logs view and cost tracking, unchanged from today), while articles_new/downstream signal counts reflect only the capped subset that was actually processed — worth a one-line tooltip/hint in the Logs tab once this ships, so "47 fetched, 10 new" doesn't read as a bug.

5. Root cause: articles attributed to the wrong company

Reported symptom: after a fetch run, two signals were attached to target company "HWA AG," but the linked article, when opened, never mentions "HWA AG" anywhere. Investigated every layer of the pipeline; findings below, ruled-out hypotheses first so the actual cause is clearly isolated.

5.1 Ruled out, with evidence

HypothesisVerdictEvidence
Attribution decided/reassigned after fetch time (some later join or update touches target_company_id)Ruled outArticle.target_company_id (models/article.py:22-24, non-nullable) is set exactly once, at construction, inside the per-company fetch loop (ingestion.py:248-259): target_company_id=target_company.id. A repo-wide search for writes to this column found none outside this one call site — every other reference is a downstream read/join (signals feed, digest, AI usage).
Dedupe (embedding similarity) merges/reassigns an article across companiesRuled outThe dedupe candidate pool is explicitly filtered to Article.target_company_id == target_company.id before any similarity comparison (ingestion.py:355-359); _find_duplicate only ever compares within that pre-filtered, single-company list (ingestion.py:512-525).
Race condition / closure bug in a loop over companies (e.g. a lambda capturing a loop variable by reference)Ruled outThe company loop (ingestion.py:142-151) calls _ingest_target_company(..., target_company=target_company, ...) as a normal, synchronous, directly-passed argument — not a closure, not scheduled via background_tasks.add_task (the only add_task calls involving a company are single-company, per-HTTP-request operations in target_companies.py, unrelated to the multi-company ingestion loop). One session, one company at a time, no concurrency.
Case-insensitive company-name dedupe (get_or_create_company) silently attaches new follows to a stale/wrong pre-existing rowRuled out as the mechanismThis dedupe (company_follows.py:20-24) only affects which row a follow attaches to when a company is created/re-added by name — it doesn't touch article fetching or attribution, which still flows entirely through §5.1's first row (the per-company fetch loop using that company's own current name/keywords).
"HWA AG" is a hardcoded example/seed/demo string leaking from a prompt template into unrelated outputRuled outRepo-wide search (source + git history) for the literal string HWA found zero matches anywhere in application code, prompts, fixtures, or commit history — only incidental substring hits inside minified node_modules bundles. It's real user-entered data (added via the target-companies UI), not a leaked placeholder.

5.2 Actual root cause: unverified, loose search-provider matching

Primary cause: every news source query is built by build_or_query() (backend/app/services/news_query.py:7-22) as name OR keyword1 OR keyword2 OR ... — an OR, not an AND, of the company name and every configured keyword/alias. All three clients call it identically (google_news_rss_client.py:31, news_client.py:87-88, newsdata_client.py:82). Any single keyword matching is sufficient for a provider to return an article — the company name itself need not appear in it at all. No client, and no code anywhere in ingestion.py, ever verifies that a fetched article's title/description actually contains the company name or a keyword before it's stored as an Article row tied to that company's target_company_id. The pipeline trusts each provider's own search relevance unconditionally.

This is compounded by two contributing factors that make the resulting signal look confidently "correct" even though it isn't:

Contributing factor — triage only filters topical noise, not wrong-company matches. _TRIAGE_SYSTEM_PROMPT (ai_client.py:77-85) explicitly instructs: "Answer false only for articles with no plausible business/outreach angle at all (e.g. pure sports/celebrity/ local-interest noise that happened to match a keyword)." This filters for topical relevance, not "is this actually about the named target company." A real, legitimate business article about a different company that happens to share a loosely-matched keyword with "HWA AG" has a perfectly plausible business/outreach angle — it just isn't for HWA AG — and sails straight through triage.
Contributing factor — the summarization prompt asserts the company as ground truth instead of asking the model to verify it. summarize_article (ai_client.py:213-252) builds its prompt with f"Target company: {target_company_name}" (line 233) as a flat statement of fact, then asks for a summary, business-relevance narrative, and ready-to-send outreach snippets about that company. The model is never asked to first confirm the article text actually references target_company_name — it's simply told this is the target and asked to write fluent, confident sales content accordingly. Given an article that isn't really about HWA AG, the model does exactly what it's instructed to do: writes a plausible-sounding HWA AG signal anyway, because the prompt never gave it permission to say "this isn't actually about HWA AG."

5.3 Fix plan, in priority order

Fix 1 — highest priority Post-fetch grounding filter (code-level, cheap, catches most cases)

After merging each source's raw results for a company (ingestion.py:227, before the URL/DB dedupe pass — the same insertion point identified for the article cap in §4.1, so these two changes naturally land together), drop any fetched item whose combined title + description (+ full_content when NewsData provided it) contains none of: the company name, or any configured keyword, as a case-insensitive substring. This is pure string matching — no extra API/AI calls — and directly prevents the loosely-matched false positives identified in §5.2 from ever becoming an Article row in the first place.

Known limit: doesn't help when the false-positive trigger is the keyword itself being too generic (the article genuinely contains the substring, just not about the right entity) — see Fix 4.

Fix 2 Tighten the triage prompt to check company identity

Update _TRIAGE_SYSTEM_PROMPT (ai_client.py:77-85) to explicitly ask whether the article's actual subject is target_company_name (or a clear alias), not merely whether the topic has some business angle. Second line of defense for cases that pass Fix 1's substring check (keyword technically present) but are still about a different entity — e.g. the company mentioned only in a boilerplate "also see" list.

Fix 3 Let the summarizer say "this isn't actually about them"

Stop asserting Target company: {target_company_name} as unquestioned fact. Instruct the model to first judge whether the target company is genuinely the article's subject, and add an escape hatch to AISummaryResult (ai_client.py:98-131) — e.g. a company_mentioned: bool field — so a mismatch skips Signal creation with a new skip_reason="company_mismatch" (same pattern as the existing "triaged_out"/"ai_error" skip reasons in _skip_article, ingestion.py:482-486) instead of forcing a fabricated-sounding but fluent summary.

Fix 4 — lower priority Product guidance on keyword quality

Add a hint near the "Keywords / aliases" field (SettingsTargets.tsx, i18n key targets.keywordsPlaceholder and the surrounding subtitle) discouraging single generic words/abbreviations that could match unrelated companies — no automated filter fully replaces users picking well-scoped aliases, and this reduces how often Fix 1/2/3 are even needed. Not a code correctness fix, a data-quality nudge.

Optional cleanup Retroactive audit of existing signals

Fixes 1–3 are forward-looking only — the already-stored HWA AG signals (and any other undetected mismatches) aren't touched by a code change. A one-time admin script/endpoint that re-runs Fix 1's substring check (and optionally Fix 2/3's AI check) against existing Article/Signal rows, flagging or soft-deleting mismatches for admin review, would close the loop — sized as a follow-up, not blocking the pipeline fix. See §11.

10. Open questions & assumptions

Cap value — configurable vs. hardcoded: defaulting max_articles_per_company_per_run to 10 but leaving it admin-adjustable, consistent with every other ingestion limit in this app. Confirm if a fixed, non-configurable 10 was actually intended.
Cap and the NewsData backfill flow: assumed the 10-newest cap applies only to the routine polling run (run_ingestion), not the separate one-time historical backfill (newsdata_backfill.py), which is deliberately meant to pull a larger historical batch. Confirm this reading.
Grounding filter scope (Fix 1): proposed as a simple case-insensitive substring check against title+description(+full_content). This will still admit an article where a generic keyword happens to literally appear but the article isn't really about that company (the known limit noted in §5.3) — Fix 2/3 exist specifically to catch that remaining slice. Confirm this two-layer (cheap filter + AI judgment) approach is the right level of investment vs. a heavier entity-recognition/disambiguation system, which is explicitly out of scope here.
Retroactive cleanup: Phase 4's audit of already-stored signals is proposed as optional/follow-up. Confirm whether the existing HWA AG signals (and any siblings) should be manually reviewed/deleted now versus waiting for the automated audit tool.
Cancel button visibility: assumed only admins should see/use the Stop control per the ask ("for admins only"), even though any authenticated user can currently trigger a run. A regular user who triggered a run they now want to stop would need to ask an admin. Confirm this asymmetry (trigger: anyone; cancel: admin-only) is intended rather than cancel also being available to whoever triggered that specific run.

11. Future enhancements (explicitly out of scope here)

  • Hard/instant cancellation (interrupting an in-flight news-source or Mistral HTTP call mid-request) — would require moving the pipeline onto cancellable async tasks or a real task-queue worker (Celery/RQ) with process-level termination; today's cooperative, checkpoint-based cancel is the pragmatic fit for the current synchronous-in-a-background-thread architecture.
  • Full entity recognition/disambiguation for company mentions (e.g. distinguishing "HWA AG" from an unrelated "HWA" acronym via NER/knowledge-base linking) — Fix 1–3 are a pragmatic, cheap-to-build layer; true disambiguation is a materially larger project.
  • Global "a fetch is running" indicator outside the Signals feed page (nav badge/toast) — noted in §2 and Phase 4 as a nice-to-have, not required for correctness since the run itself is already unaffected by navigation.
