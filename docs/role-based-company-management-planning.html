Role-Based Access & Company Management

Planning document — admin/user roles and per-user company visibility for NewsAtlas

Status: Planning / pre-implementation
Builds on: planning.html (base platform architecture)
Doc purpose: Source of truth for implementing role-based access + per-user company follows in a future session

The base platform (see planning.html) explicitly deferred this as a non-goal: "Per-user assignment of target companies / role-based permissions beyond basic auth" (§2), and flagged the shared, all-users-see-everything digest as a placeholder pending "a plausible fast-follow" (§14). This document specs that fast-follow: a two-role permission model (admin / user) plus a many-to-many user↔company relationship so each user only sees and manages the companies relevant to them.

1. Problem statement

Today every authenticated user shares one flat, all-visible list of target companies (target_companies has no owner beyond an informational created_by), one shared workspace_settings row anyone can edit, and a daily digest that mails every user every new signal regardless of which companies they actually care about. As NewsAtlas grows past a single small team, this breaks down: there's no way to onboard a new salesperson with just "their" accounts, no way to keep workspace-wide configuration (AI grounding text, schedules) from being changed by anyone who logs in, and every inbox gets noise from companies the recipient never asked about.

This feature introduces two roles — admin and user — where admins own workspace settings and can set companies up on behalf of a user, and regular users are scoped to only the companies they follow: they see them in the dashboard, receive digest signals for them, and can manage (edit/unfollow) them, but nothing else in the workspace.

2. Goals & non-goals

Goals

  • Add a role to every user: admin or user.
  • Admins can read and update workspace_settings; regular users can no longer write it.
  • Admins can view all users, promote/demote roles, and set companies up for a specific user (creating the company if it doesn't exist, or attaching an existing one).
  • Regular users see, filter, and manage (edit keywords/industry, mute, unfollow) only the companies they follow — never the full workspace catalog.
  • Regular users can still create their own companies (self-service), which auto-follows them; creating a company whose name matches an existing one attaches them to the existing record instead of duplicating ingestion.
  • Signals feed and the daily email digest are scoped per-user to followed companies (this also finally resolves the "everyone gets the same digest" placeholder from the base plan).
  • Ingestion stays company-scoped (unaffected by how many users follow a company) — no duplicate NewsAPI/AI spend just because two users track the same company.

Non-goals (this pass)

  • Full multi-tenant isolation (separate orgs with separate workspace_settings) — still one shared workspace; roles are a permission layer within it, not tenancy.
  • Fine-grained/custom permission sets (e.g. "read-only" or per-feature roles) — just admin vs. user.
  • Team/company-sharing beyond direct admin assignment (e.g. "share my company with a teammate") — only admins can attach a company to another user's follow list.
  • Per-user rate limits or ingestion quotas — out of scope here.

3. Key decisions

Role storage: a role enum column (admin / user) directly on users, default user. confirmed Two roles don't warrant a separate roles/permissions table yet.
Bootstrap admin: the first account ever created (signup #1, or whichever existing user has the earliest created_at at migration time) becomes admin automatically. From then on, only an existing admin can promote/demote other users' roles — there is no separate admin invite code. confirmed Keeps the current invite-code signup flow unchanged for everyone.
User↔company relationship: many-to-many via a new company_follows join table (user_id, target_company_id), not a single owner_id on target_companies. confirmed Matches "companies they follow/see" literally and lets the same company (e.g. two reps both tracking "Acme Corp") be tracked once and shared, rather than duplicated per owner.
Company creation dedupe: creating a company whose name matches an existing target_companies.name case-insensitively attaches a new company_follows row to the existing record instead of inserting a duplicate. default Avoids double ingestion/AI cost for the same company and means a regular user's self-created company can transparently merge with one an admin already set up.
Shared vs. per-user company fields: name, keywords, industry, and is_active (the global ingestion toggle) stay on target_companies and are shared/editable by any current follower or an admin — no per-field ownership lock for MVP (last write wins, same as today). A new per-follow is_muted flag on company_follows lets a user hide a company from their own feed/digest without touching ingestion for other followers. default
Delete vs. unfollow: a regular user's "remove company" action is always an unfollow (delete their company_follows row). The underlying target_companies row (and its ingested articles/signals) is hard-deleted only when an admin deletes it, or when a regular user unfollows and was the company's only remaining follower. confirmed
Settings scope: workspace_settings (company profile / offering description / schedules) becomes admin-only for both read and write — regular users don't need workspace configuration under this model. confirmed
Ingestion/ai pipeline: unaffected structurally — still runs once per active target_company, independent of follower count. Only the read paths (signals feed, signal detail, digest) become follow-scoped. confirmed
Authorization mechanism: a new require_admin FastAPI dependency (wrapping get_current_user, raising 403 if role != admin) alongside existing get_current_user; company-scoped endpoints get a _require_company_access(user, company) helper (403 unless admin or a follower). default

6. Core workflows

6.1 Bootstrap & role assignment

  1. Migration runs: adds role column defaulting to user for everyone, then a one-time data step promotes the single earliest-created existing user to admin (see §8).
  2. Going forward, signup (unchanged invite-code flow) always creates role = user; if the workspace has zero users at signup time, the new account is promoted to admin instead (covers fresh installs with no pre-existing data).
  3. An admin can change any other user's role via PATCH /admin/users/{id}/role. An admin cannot demote themselves if they'd be the last remaining admin (guard against workspace lockout).

6.2 Admin sets a company up for a user

  1. Admin opens the new "Users" admin page, picks a user, and either selects an existing catalog company or types a new one (name + keywords/industry).
  2. POST /admin/users/{user_id}/companies — server looks up target_companies by case-insensitive name; if found, reuses it, else creates it (created_by = admin.id); either way inserts a company_follows row with user_id = the target user and assigned_by = the admin.
  3. The target user sees the company appear in their own dashboard/company list on next load — no action required from them.

6.3 Regular user self-service company creation

  1. User adds a company from their own Companies page exactly as before.
  2. POST /target-companies now dedupes by name (§3) and always creates/reuses + auto-follows for current_user (assigned_by = current_user.id).

6.4 Scoped signals feed & digest

list_signals and the digest builder join Signal → Article → TargetCompany → company_follows filtered to company_follows.user_id = current_user.id AND NOT is_muted (for regular users). Admins additionally get an optional ?scope=all query param to see workspace-wide signals for oversight/QA, defaulting to their own follows like everyone else.

6.5 Ingestion (unchanged)

The scheduled ingestion job still iterates every target_company with is_active = true, regardless of follower count — see base planning §6.3. No changes needed here beyond what's already specced.

PATCH  /admin/users/{id}/role            # admin-only: promote/demote (blocks removing last admin)
POST   /admin/users/{id}/companies       # admin-only: assign existing/new company to a user
DELETE /admin/users/{id}/companies/{company_id}  # admin-only: remove a user's follow

8. Migration & backfill plan

  1. Alembic migration: add users.role enum column, default user, not null.
  2. Alembic migration: create company_follows table with the unique (user_id, target_company_id) constraint and FKs above.
  3. Data backfill (in the same or a follow-up migration):
    • Promote the user with the earliest created_at to role = admin (ties broken by id). If the table is empty, skip — the next signup becomes admin per §6.1.
    • Full cross-join backfill: insert a company_follows row for every existing (user, target_company) pair. This preserves current behavior exactly at migration time (everyone already sees everything today) instead of silently hiding data users could previously see. assigned_by is left NULL for these backfilled rows (no single "who assigned it" is meaningful retroactively).
Backfill cost: the cross-join is O(users × companies) rows — fine at current single-workspace scale; revisit if either grows into the thousands before this migration runs.

9. Frontend changes

  • User type + AuthContext gain role; a useIsAdmin() helper reads it off the current user.
  • New RequireAdmin route guard (mirrors whatever the existing authenticated-route guard looks like), wrapping new admin-only routes; non-admins hitting them redirect to the signals feed.
  • New Admin → Users page: table of all users + role, promote/demote action, and an "assign company" action per user (reuses the existing add-company form, posts to /admin/users/{id}/companies).
  • SettingsTargets.tsx becomes the regular user's own "My Companies" view — same CRUD UX, but the list is now naturally scoped by the backend; add a mute toggle per row backed by POST /target-companies/{id}/mute; "delete" copy changes to "Unfollow" (with a "this also removes it for other users" warning shown only when the current user is the last follower — the DELETE response can indicate that).
  • Workspace Settings (company profile / offering / schedule) page moves under an admin-only nav section; regular users no longer see a link to it.

10. Implementation roadmap

Phase 1 Roles foundation

  • Alembic migration: users.role enum + bootstrap-admin backfill
  • require_admin dependency in api/deps.py
  • /auth/me + UserResponse include role; signup promotes to admin only when the users table is empty
  • Gate /settings GET/PUT behind require_admin

Phase 2 Company follows

  • Alembic migration: company_follows table + full cross-join backfill
  • Dedupe-by-name logic in company creation (shared helper used by both self-serve create and admin assignment)
  • Rewrite target-companies list/patch/delete to be follow-scoped with the admin-vs-follower rules from §5
  • POST /target-companies/{id}/mute endpoint

Phase 3 Admin user management

  • GET /admin/users, PATCH /admin/users/{id}/role (with last-admin guard)
  • POST /admin/users/{id}/companies, DELETE /admin/users/{id}/companies/{company_id}
  • Audit log entries (via existing core/audit.log_event) for role changes and admin company assignments

Phase 4 Scoped signals & digest

  • Join company_follows into list_signals/get_signal, respecting is_muted; add admin-only scope=all
  • Rewrite digest builder to select per-user unseen signals by followed, unmuted companies (replacing the current "every user gets every new signal" broadcast)

Phase 5 Frontend

  • role in auth types/context, RequireAdmin guard
  • Admin → Users page (list, promote/demote, assign company)
  • Update Companies page: mute toggle, unfollow-vs-delete copy
  • Move workspace Settings page under admin-only nav

Phase 6 Tests & hardening

  • Backend tests: role gating on settings/admin routes, follow-scoping on signals/companies, last-admin-demotion guard, dedupe-on-create merge behavior
  • Migration dry-run against a copy of production-shaped data to sanity-check backfill counts
  • Update docs/planning.html §2/§14/§15 to remove the now-implemented "role-based permissions" / "per-user subscriptions" deferrals

11. Open questions & assumptions for follow-up sessions

Shared-field edit conflicts — two followers editing the same company's keywords concurrently is last-write-wins with no locking/notification for MVP; acceptable at current team size, revisit if it causes real confusion.
Admin visibility into "all companies" — assumed admins want an escape hatch (scope=all) to see the full catalog/signals for oversight rather than being follow-scoped like everyone else; confirm this is actually wanted vs. admins also just following companies like anyone else.
Removing a user — deleting a user account (not currently supported anywhere in the app) would cascade-delete their company_follows rows but leave shared target_companies/articles/signals intact if other followers remain; no deactivation flow is specced here since none exists today.
Notifying a user when a company is assigned to them — out of scope for this pass; they'll simply see it next time they load the dashboard. A "you've been assigned X" email could be a fast-follow once the digest rework (Phase 4) is in place.

12. Future enhancements (explicitly out of scope here)

  • Custom/granular roles beyond admin vs. user (e.g. a "manager" role that can assign companies within a team but not touch workspace settings)
  • True multi-tenancy (multiple isolated workspaces, each with its own settings and admin set)
  • Per-company or per-user notification preferences (digest frequency, channel) beyond the existing global schedule
  • Assignment notifications / activity feed for admins auditing who follows what
