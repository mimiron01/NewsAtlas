Bilingual Support: English/German

Planning document — adding proper, state-of-the-art i18n infrastructure to NewsAtlas, starting with German

Status: Planning / pre-implementation
Builds on: planning.html (base platform architecture), role-based-company-management-planning.html (users/roles)
Doc purpose: Source of truth for implementing i18n infrastructure + German locale in a future session

NewsAtlas today has zero i18n infrastructure: every UI string across the React frontend is hardcoded inline in components (e.g. STATUS_OPTIONS and ingestionStatusText() in SignalsFeed.tsx, ARTICLE_SOURCE_LABELS in api/types.ts, tab labels across settings/*Tab.tsx), there's no locale routing, no translation files, and date formatting falls back to whatever toLocaleDateString() does with no explicit locale. This document specs the foundation for proper bilingual (English/German) support: an i18n library, a string-extraction pass across the app, a translation workflow, and a phased rollout — building infrastructure that scales to more languages later rather than a one-off German hack.

1. Problem statement

NewsAtlas is an English-only React 18 + TypeScript SPA (Vite, react-router-dom) backed by a FastAPI service. No i18n library is installed, no locales/ or translation JSON exists, and UI copy is written directly into JSX across roughly a dozen pages/components (Dashboard, SignalsFeed, SignalDetail, Login, ProtectedLayout, ProfileMenu, SettingsTargets, and the six settings tabs). To genuinely support German users we need infrastructure, not a translated copy of the app: a single source of truth for strings, a way to add locales without touching component code, and a mechanism for users to get German either automatically or by choice.

2. Goals & non-goals

Goals

  • Introduce a standard React i18n library and migrate all hardcoded UI strings (nav, buttons, labels, empty states, error/validation messages, settings tabs) into translation resource files.
  • Ship a complete, reviewed German translation for every extracted string.
  • Detect the user's language from the browser (Accept-Language / navigator.language) on first load and default to it when supported, falling back to English otherwise.
  • Localize dates, numbers, and pluralized strings using the active locale instead of ad hoc toLocaleDateString() calls with no explicit locale argument.
  • Make adding a third language later a matter of dropping in a new resource-file set — no component changes.

Non-goals (this pass)

  • Translating article/news content or AI-generated Signal text — confirmed out of scope; see §9. "Bilingual support" here means the interface, not the news itself.
  • Locale-prefixed URL routing (e.g. /de/signals) — not needed since this is an authenticated internal app, not SEO-driven public content.
  • Region-specific variants (de-AT, de-CH, en-GB) — two flat locales, en and de, for now.
  • Standing up CI/CD — the repo has no GitHub Actions pipeline today; not introducing one as part of this effort, though the i18n completeness check (§11) is written so it can slot into one later.

3. Key decisions

Library: react-i18next + i18next + i18next-browser-languagedetector. confirmed The de facto standard for React/Vite SPAs, first-class TypeScript support, and the detector plugin natively implements the browser-based detection this project asked for.
Resource loading: JSON translation files bundled at build time via static imports (no i18next-http-backend / runtime fetch). default The app is small and has no CDN/edge-translation infrastructure; bundling keeps this simple and avoids an extra network waterfall on load. Revisit only if translation volume grows enough to matter for bundle size.
Detection strategy: i18next-browser-languagedetector with order ["localStorage", "navigator"] — checks a stored preference first, falls back to navigator.language, defaults to en if neither matches a supported locale. confirmed Matches the browser-detection approach selected for this project.
Manual override control: add a compact language switcher in the sidebar footer, immediately next to the existing theme toggle in ProtectedLayout.tsx, mirroring the useTheme/cycleTheme pattern already in the codebase. recommended — confirm Auto-detection with no escape hatch means a German browser locale with an English-preferring user (or vice versa) has no way to switch; this was not explicitly requested but is standard practice and is a small addition given the theme-toggle precedent already exists to copy. Flagged in §13 for explicit sign-off.
Persistence: client-side only, localStorage key newsatlas_locale, exactly mirroring useTheme.ts's newsatlas_theme key. default No users table migration needed for MVP. Per-user server-persisted locale (so preference follows the user across devices) is a plausible fast-follow — see §13.
<html lang> sync: reflect the active i18next language onto document.documentElement.lang whenever it changes. confirmed Required for accessibility (screen readers, browser spellcheck/font selection) and costs nothing extra given the existing document.documentElement.dataset.theme pattern already touches the root element.
Namespace structure: one JSON resource file per feature area per locale (not one giant flat file), loaded as i18next namespaces. default Keeps diffs small when only one page's copy changes and matches the app's existing page-based file layout.

4. Architecture & folder structure

frontend/src/ ├── i18n/ │ ├── index.ts # i18next.init(), language detector, namespace registration │ └── locales/ │ ├── en/ │ │ ├── common.json # buttons, generic labels (Save, Cancel, Delete, Loading…) │ │ ├── nav.json # sidebar nav, brand, theme toggle, language switcher │ │ ├── auth.json # Login page, ProfileMenu, logout │ │ ├── dashboard.json # Dashboard page copy │ │ ├── signals.json # SignalsFeed + SignalDetail (STATUS_OPTIONS, ingestionStatusText, etc.) │ │ ├── settings.json # SettingsTargets + all settings/*Tab.tsx pages │ │ └── errors.json # validation/error message strings (see §8) │ └── de/ │ └── (same files, German translations — kept 1:1 key-for-key with en/) ├── hooks/ │ └── useLocaleFormat.ts # Intl.DateTimeFormat/NumberFormat wrapper, keyed off i18n.language └── App.tsx # wraps root in for i18next's async namespace loading

i18n/index.ts is imported once at the app entrypoint (main.tsx), before App renders, following the standard react-i18next bootstrap pattern. Every component swaps hardcoded JSX text for the useTranslation() hook's t() function, namespaced to the relevant JSON file (e.g. useTranslation("signals") in SignalsFeed.tsx).

5. Locale detection, persistence & switcher

Mirrors the existing theme system almost exactly, so the mental model and code shape are already familiar to anyone who's touched useTheme.ts:

Theme (existing)Locale (new)
Storage keynewsatlas_themenewsatlas_locale
Valueslight / dark / systemen / de
Root attributedocument.documentElement.dataset.themedocument.documentElement.lang
Fallback when unsetOS preference (prefers-color-scheme)Browser language (navigator.language)
UI controlIcon button, sidebar footerNew icon/dropdown, sidebar footer, next to theme toggle

Flow on first visit: i18next-browser-languagedetector checks localStorage (nothing yet) → reads navigator.language → if it starts with de, activates German; otherwise (or for any other value) defaults to English. Once a user explicitly picks a language via the switcher, that choice is written to localStorage and takes precedence on every subsequent visit, overriding browser detection — this is what makes the auto-detect approach safe to ship without a routing change: it's a sensible default, not a trap.

6. String extraction scope

Every hardcoded string identified needs a translation key. Concretely, by file:

Recommend adding eslint-plugin-i18next (or a similar lint rule flagging bare string literals in JSX) once extraction is complete, so newly-written components can't reintroduce hardcoded English without a lint failure.

7. Dates, numbers & pluralization

Introduce a useLocaleFormat() hook wrapping Intl.DateTimeFormat and Intl.NumberFormat, constructed with i18n.language as the locale argument, and use it everywhere date/number output currently relies on the browser's implicit default (e.g. new Date(signal.article_published_at).toLocaleDateString() in SignalDetail.tsx:191, which today silently uses whatever locale the browser happens to report rather than the app's active language). Replace hand-built sentence templates like ingestionStatusText() with i18next's interpolation ({{variable}}) and plural (_one/_other key suffixes) support instead of string concatenation, since German pluralization and word order rules differ from English and can't be patched onto an English template string.

8. Backend-originated user-facing text

The FastAPI backend returns some user-facing strings directly (e.g. HTTPException detail messages, validation errors) that the frontend currently displays as-is. Under a "UI strings only" scope, these are chrome too — a German user shouldn't see a raw English server error. Since there's no backend i18n system and standing one up is a significant lift for a handful of strings, the recommended approach is an error-code layer: extend API error responses with a stable error_code field alongside the existing free-text detail (additive, not breaking existing consumers), and have the frontend map known codes to translated strings via the errors namespace, falling back to the raw detail for any code not yet mapped. This avoids full backend i18n while still fully localizing the errors users actually see.

9. Content scope: articles & AI-generated text

Confirmed scope for this initiative is UI strings only. Two categories of user-facing text are explicitly not translated by this work:

  • Ingested article content (Article.title, .description, .full_content) — sourced in English by default. NewsAPI.org and NewsData.io hardcode "language": "en" with no admin control; Google News RSS is the one exception — see §10.
  • AI-generated Signal content (Signal.summary, .business_relevance, .outreach_snippet_email, .outreach_snippet_linkedin, .outreach_call_opener, .supporting_quote) — generated by the Mistral prompt in English today.

Both remain English regardless of the UI's active language after this work ships. This is worth surfacing explicitly (see §13) because "bilingual support" as a product claim could be read either way, and a German-speaking user browsing an all-German interface full of English news summaries is a noticeably different experience than a fully localized one. Translating or regenerating that content (machine translation of articles, or prompting Mistral in German) is a materially larger effort — different cost profile, different quality bar, different failure modes — and is called out as a distinct future initiative in §14, not bundled here.

10. Admin-configurable source language & AI output language

Prompted by a question during review: could an admin configure ingestion to fetch only German articles, so Mistral works from native German source text instead of translating? Investigating the actual ingestion pipeline surfaced one existing capability and one real gap.

Google News RSS is already admin-configurable for German. workspace_settings.google_news_rss_country and .google_news_rss_language (models/workspace_settings.py:54-56) already exist end-to-end: exposed via GET/PUT /settings (api/settings.py) and editable today in Settings → Sources (SourcesTab.tsx:182-200, free-text Country/Language fields feeding GoogleNewsRSSClient's ceid={country}:{language} query param). An admin can already set country=DE, language=de and Google News RSS will fetch German-language articles. already shipped No code change needed for this source.
NewsAPI.org and NewsData.io stay English-only. Both clients hardcode "language": "en" in their query params (news_client.py:43, newsdata_client.py:84) with no equivalent admin setting. confirmed — out of scope Making these configurable was considered and explicitly deferred for this pass. An admin who wants ingestion to be German-only should enable Google News RSS with country=DE/language=de and disable NewsAPI/NewsData.io using their existing newsapi_enabled/newsdata_enabled toggles (already shipped, Settings → Sources) — a configuration-only change, not a code change. Leaving any English source enabled alongside a German-configured Google News RSS will keep producing English articles from that source.
Mistral has no explicit output-language directive. The system prompt in ai_client.py (_SYSTEM_PROMPT, lines 27-56) is written in English and specifies the required JSON schema, but never instructs the model what language to write the content in. In practice, LLMs typically mirror the language of the input they're given when not told otherwise — so feeding it German article titles/descriptions (via a German-only Google News RSS configuration) will likely produce German summary/ business_relevance/outreach-snippet text, but this is emergent behavior, not a controlled setting. Per this review's decision, no explicit language instruction is being added to the prompt in this pass — the recommended path is entirely on the ingestion side (§10 above: configure/disable sources so only German content reaches Mistral), not a prompt change. If any English-fetching source stays enabled at the same time, expect a mix of German- and English-language Signals with no language field on Article/Signal to distinguish them downstream — flagged as a real limitation to watch for, not something this pass resolves.

11. Testing & quality gates

  • i18n completeness script — a small Node script (scripts/check-i18n-completeness.mjs) that diffs keys across every en/*.json / de/*.json pair and fails (non-zero exit) if any key is missing or orphaned in either locale. Runnable manually or via npm run lint today; slots into a CI job later without changes if/when this repo gets a pipeline (it currently has none — see §2 non-goals).
  • Frontend test framework — the frontend currently has zero test tooling (no Jest/Vitest/Testing Library in package.json). Recommend introducing Vitest + React Testing Library alongside this work, at minimum for a smoke test that renders a page in each locale and asserts no missing-key fallback strings leak through, plus a test for the useLocaleFormat hook's date/number output per locale.
  • Manual QA pass — native/professional review of the German translations (not machine-translation-only, given how visible UI chrome is), plus a visual pass for text expansion (German UI strings commonly run ~30% longer than English — check buttons, nav labels, and table headers don't clip or wrap awkwardly).

12. Implementation roadmap

Phase 0 Foundation

  • Install react-i18next, i18next, i18next-browser-languagedetector
  • Create i18n/index.ts config: init, detector order, fallback locale en, namespace registration
  • Wrap app entrypoint with <Suspense> for async namespace loading
  • useLocaleFormat hook (dates/numbers) alongside the existing useTheme hook
  • Sync document.documentElement.lang on language change

Phase 1 Core chrome

  • Extract ProtectedLayout.tsx, ProfileMenu.tsx, Login.tsx into nav/auth/common namespaces
  • Add language switcher control next to the existing theme toggle
  • German translation of extracted core-chrome strings

Phase 2 Feature pages

  • Extract Dashboard.tsx, SignalsFeed.tsx (incl. STATUS_OPTIONS, ingestionStatusText()), SignalDetail.tsx, SettingsTargets.tsx
  • Extract all six settings/*Tab.tsx pages and ARTICLE_SOURCE_LABELS in api/types.ts
  • German translation of extracted feature-page strings

Phase 3 Formatting & backend errors

  • Replace ad hoc toLocaleDateString() calls with useLocaleFormat
  • Convert hand-built template sentences to i18next interpolation/pluralization
  • Add error_code field to key API error responses; map codes in errors.json

Phase 4 QA & tooling

  • Add scripts/check-i18n-completeness.mjs
  • Stand up Vitest + React Testing Library; locale-switch and formatting smoke tests
  • Native-speaker German review pass; text-expansion visual QA
  • Add eslint-plugin-i18next (or equivalent) to catch future hardcoded strings

13. Open questions & assumptions for follow-up sessions

Manual language switcher — only browser-based auto-detection was explicitly requested; this plan adds a manual override control anyway as standard practice (see §3). Confirm this is wanted, or that pure auto-detect (no escape hatch) is acceptable for the initial ship.
Locale persistence scope — assumed client-side localStorage only (matches theme precedent), not a users.locale DB column. Confirm whether cross-device persistence of a user's language preference matters enough to warrant the schema change now vs. later.
"Bilingual" scope — confirmed as UI-only for this pass (see §9); article/Signal content stays English. Flagging because the product framing ("bilingual platform") could be read by stakeholders as including translated news content — worth an explicit sentence in any external communication about this feature so expectations match what ships.
Backend error-code coverage — assumed only a handful of HTTPException call sites actually surface in the UI in ways users see today; the Phase 3 error-code work should start with an audit of which backend error strings the frontend currently displays verbatim, rather than instrumenting every endpoint speculatively.
CI — this repo has no GitHub Actions pipeline today; the completeness script and any new tests run locally/manually for now. Not proposing to introduce CI as part of this initiative, but noting it's the natural next home for these checks if/when one exists.

14. Future enhancements (explicitly out of scope here)

  • Machine-translating or re-generating ingested article content and AI Signal fields into German (would require either a translation-API integration or a German-language Mistral prompt variant, plus a cost/latency tradeoff analysis — materially different scope from UI-string i18n).
  • Explicit output-language control in the Mistral prompt (ai_client.py _SYSTEM_PROMPT) and a language column on Article/ Signal to track and guarantee content language, instead of relying on Mistral implicitly mirroring whatever language the source article happens to be in — see §10.
  • Making NewsAPI.org's and NewsData.io's hardcoded "language": "en" query params admin-configurable, matching what Google News RSS already has — see §10.
  • Locale-prefixed routing (/de/...) for SEO — not relevant to an authenticated internal tool today, but worth revisiting if any public-facing pages are ever added.
  • Per-user server-persisted locale preference syncing across devices.
  • Additional locales beyond English/German (the namespace-per-feature structure in §4 is designed so this is additive — new resource files, no component changes).
  • Region variants (de-AT, de-CH, en-GB) if the user base ever needs them.
