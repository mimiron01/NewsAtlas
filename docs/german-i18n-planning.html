Bilingual Support: English/German

Planning document — adding proper, state-of-the-art i18n infrastructure to NewsAtlas, starting with German

Status: Planning / pre-implementation
Builds on: planning.html (base platform architecture), role-based-company-management-planning.html (users/roles)
Doc purpose: Source of truth for implementing i18n infrastructure + German locale in a future session

NewsAtlas today has zero i18n infrastructure: every UI string across the React frontend is hardcoded inline in components (e.g. STATUS_OPTIONS and ingestionStatusText() in SignalsFeed.tsx, ARTICLE_SOURCE_LABELS in api/types.ts, tab labels across settings/*Tab.tsx), there's no locale routing, no translation files, and date formatting falls back to whatever toLocaleDateString() does with no explicit locale. This document specs the foundation for proper bilingual (English/German) support: an i18n library, a string-extraction pass across the app, a translation workflow, and a phased rollout — building infrastructure that scales to more languages later rather than a one-off German hack.

1. Problem statement

NewsAtlas is an English-only React 18 + TypeScript SPA (Vite, react-router-dom) backed by a FastAPI service. No i18n library is installed, no locales/ or translation JSON exists, and UI copy is written directly into JSX across roughly a dozen pages/components (Dashboard, SignalsFeed, SignalDetail, Login, ProtectedLayout, ProfileMenu, SettingsTargets, and the six settings tabs). To genuinely support German users we need infrastructure, not a translated copy of the app: a single source of truth for strings, a way to add locales without touching component code, and a mechanism for users to get German either automatically or by choice.

2. Goals & non-goals

Goals

  • Introduce a standard React i18n library and migrate all hardcoded UI strings (nav, buttons, labels, empty states, error/validation messages, settings tabs) into translation resource files.
  • Ship a complete, reviewed German translation for every extracted string.
  • Add an admin-configurable workspace Main Language (default English) that sets the standard UI language for every user, with a per-user personal preference that overrides it; pre-login, detect the browser's language (Accept-Language / navigator.language) to localize the Login screen only — see §3/§5.
  • Make Mistral generate Signal content (summaries, business relevance, outreach snippets) only in the workspace Main Language, regardless of the source article's language — see §10.
  • Localize dates, numbers, and pluralized strings using the active locale instead of ad hoc toLocaleDateString() calls with no explicit locale argument.
  • Make adding a third language later a matter of dropping in a new resource-file set — no component changes.

Non-goals (this pass)

  • Translating ingested article content (the raw text fetched from news sources, as opposed to Mistral's Signal summary of it) — confirmed out of scope; see §9. Signal content is now in scope via Main Language (above); the raw source article is not.
  • Locale-prefixed URL routing (e.g. /de/signals) — not needed since this is an authenticated internal app, not SEO-driven public content.
  • Region-specific variants (de-AT, de-CH, en-GB) — two flat locales, en and de, for now.
  • Standing up CI/CD — the repo has no GitHub Actions pipeline today; not introducing one as part of this effort, though the i18n completeness check (§11) is written so it can slot into one later.

3. Key decisions

Library: react-i18next + i18next + i18next-browser-languagedetector. confirmed The de facto standard for React/Vite SPAs, first-class TypeScript support, and the detector plugin natively implements the browser-based detection this project asked for.
Resource loading: JSON translation files bundled at build time via static imports (no i18next-http-backend / runtime fetch). default The app is small and has no CDN/edge-translation infrastructure; bundling keeps this simple and avoids an extra network waterfall on load. Revisit only if translation volume grows enough to matter for bundle size.
Workspace "Main Language" setting: a new admin-configurable workspace_settings.main_language field (en / de, default en) is the standard language for every user in the workspace unless they personally override it. confirmed Editable in Settings (admin-only, alongside the other workspace-wide toggles), analogous to the existing google_news_rss_language field but governing the UI/AI language rather than an ingestion source. This is the org-level default that individual preference (below) layers on top of — see the full data-model/API details in §10, since it also drives Mistral's output language.
Personal preference override: a nullable users.preferred_language column. NULL (the default for every user) means "follow the workspace Main Language"; setting it to en or de overrides that default for that user only, on any device. confirmed This supersedes the earlier plan to keep locale client-side/localStorage-only — a real per-user account setting was explicitly requested, not just a per-browser one.
Effective-language precedence: for a logged-in user: user.preferred_language if set → else workspace_settings.main_language. For the pre-login Login page, where neither a user nor a workspace context exists yet, fall back to i18next-browser-languagedetector reading navigator.language (defaulting to en). confirmed Browser auto-detection — the original ask — still does real work, just scoped to the one screen where there's nothing else to go on; once authenticated, the account/workspace settings are authoritative and stop deferring to the browser, so an admin's Main Language choice can't be silently overridden by whatever a shared/kiosk browser happens to report.
Switcher UX: the sidebar language switcher (still placed next to the theme toggle in ProtectedLayout.tsx) cycles Workspace default → English → German → Workspace default…, directly mirroring the existing theme toggle's system → light → dark → system cycle in useTheme.ts — "Workspace default" is the personal-preference-NULL state, exactly like system means "no override, follow the OS." confirmed Reuses a UX pattern already in the codebase instead of inventing a new one.
Client-side cache: localStorage key newsatlas_locale still exists, but only as an optimistic cache of the last-known effective language (avoids a flash of the wrong language before /auth/me resolves on load) — the database (users.preferred_language + workspace_settings.main_language) is the source of truth. default
<html lang> sync: reflect the active i18next language onto document.documentElement.lang whenever it changes. confirmed Required for accessibility (screen readers, browser spellcheck/font selection) and costs nothing extra given the existing document.documentElement.dataset.theme pattern already touches the root element.
Namespace structure: one JSON resource file per feature area per locale (not one giant flat file), loaded as i18next namespaces. default Keeps diffs small when only one page's copy changes and matches the app's existing page-based file layout.

4. Architecture & folder structure

frontend/src/ ├── i18n/ │ ├── index.ts # i18next.init(), language detector, namespace registration │ └── locales/ │ ├── en/ │ │ ├── common.json # buttons, generic labels (Save, Cancel, Delete, Loading…) │ │ ├── nav.json # sidebar nav, brand, theme toggle, language switcher │ │ ├── auth.json # Login page, ProfileMenu, logout │ │ ├── dashboard.json # Dashboard page copy │ │ ├── signals.json # SignalsFeed + SignalDetail (STATUS_OPTIONS, ingestionStatusText, etc.) │ │ ├── settings.json # SettingsTargets + all settings/*Tab.tsx pages │ │ └── errors.json # validation/error message strings (see §8) │ └── de/ │ └── (same files, German translations — kept 1:1 key-for-key with en/) ├── hooks/ │ └── useLocaleFormat.ts # Intl.DateTimeFormat/NumberFormat wrapper, keyed off i18n.language └── App.tsx # wraps root in for i18next's async namespace loading

i18n/index.ts is imported once at the app entrypoint (main.tsx), before App renders, following the standard react-i18next bootstrap pattern. Every component swaps hardcoded JSX text for the useTranslation() hook's t() function, namespaced to the relevant JSON file (e.g. useTranslation("signals") in SignalsFeed.tsx).

5. Locale detection, persistence & switcher

The switcher's cycle mirrors the existing theme system, so the mental model and code shape are already familiar to anyone who's touched useTheme.ts — but where it stores state now crosses into the backend, since "personal preference" needs to follow the account, not the browser:

Theme (existing)Locale (new)
Valueslight / dark / systemen / de / null ("workspace default")
Source of truthlocalStorage (newsatlas_theme)users.preferred_language (DB), cached in localStorage (newsatlas_locale)
Root attributedocument.documentElement.dataset.themedocument.documentElement.lang
Fallback when unsetOS preference (prefers-color-scheme)workspace_settings.main_language (admin-set, default en)
Pre-login fallbackn/a (theme has no logged-out screen needing it)Browser language (navigator.language) — Login page only
UI controlIcon button, sidebar footerNew icon/dropdown, sidebar footer, next to theme toggle

Resolution order once authenticated: user.preferred_language (if not null) → workspace_settings.main_language. Both come back on GET /auth/me (see §10 for the schema/API changes), so the frontend never needs a separate admin-gated call just to know the effective language. Before login, with neither value available yet, i18next-browser-languagedetector reads navigator.language to localize the Login screen only — a sensible default for an anonymous visitor, not a setting that persists or leaks into the authenticated app. Picking a language via the switcher issues a request to persist preferred_language (or clear it back to null for "workspace default") against the user's account.

Every hardcoded string identified needs a translation key. Concretely, by file:

Recommend adding eslint-plugin-i18next (or a similar lint rule flagging bare string literals in JSX) once extraction is complete, so newly-written components can't reintroduce hardcoded English without a lint failure.

7. Dates, numbers & pluralization

Introduce a useLocaleFormat() hook wrapping Intl.DateTimeFormat and Intl.NumberFormat, constructed with i18n.language as the locale argument, and use it everywhere date/number output currently relies on the browser's implicit default (e.g. new Date(signal.article_published_at).toLocaleDateString() in SignalDetail.tsx:191, which today silently uses whatever locale the browser happens to report rather than the app's active language). Replace hand-built sentence templates like ingestionStatusText() with i18next's interpolation ({{variable}}) and plural (_one/_other key suffixes) support instead of string concatenation, since German pluralization and word order rules differ from English and can't be patched onto an English template string.

8. Backend-originated user-facing text

The FastAPI backend returns some user-facing strings directly (e.g. HTTPException detail messages, validation errors) that the frontend currently displays as-is. Under a "UI strings only" scope, these are chrome too — a German user shouldn't see a raw English server error. Since there's no backend i18n system and standing one up is a significant lift for a handful of strings, the recommended approach is an error-code layer: extend API error responses with a stable error_code field alongside the existing free-text detail (additive, not breaking existing consumers), and have the frontend map known codes to translated strings via the errors namespace, falling back to the raw detail for any code not yet mapped. This avoids full backend i18n while still fully localizing the errors users actually see.

9. Content scope: articles & AI-generated text

Confirmed scope for this initiative is UI strings only for the interface itself. This section originally treated both ingested articles and AI-generated Signals as permanently English; the Main Language decision in §3/§10 changes that for one of the two, so the distinction now matters:

  • Ingested article content (Article.title, .description, .full_content) — not translated by this work, ever. Sourced in whatever language the enabled news APIs return: English by default, since NewsAPI.org and NewsData.io hardcode "language": "en" with no admin control; Google News RSS is the one exception — see §10. Raw article text is displayed as-fetched regardless of UI or Main Language settings.
  • AI-generated Signal content (Signal.summary, .business_relevance, .outreach_snippet_email, .outreach_snippet_linkedin, .outreach_call_opener, .supporting_quote) — is now in scope, via the Main Language setting rather than the UI-language setting: Mistral is instructed to always write these fields in the workspace's main_language (§10), translating from the source article when needed. This is a deliberate, admin-controlled behavior, not a byproduct of UI localization — a workspace can run its interface in English while Main Language (and therefore every Signal) is German, or vice versa; the two settings are independent.

Net effect: a German-speaking user no longer necessarily sees English AI summaries — that now depends entirely on the admin's Main Language choice, not on this pass being "UI-only." Only the raw ingested article text (title/description/full_content, as opposed to Mistral's summary of it) remains permanently in its source language, since translating or regenerating that raw text is a materially larger effort (a translation-API integration, or re-ingesting from German-native sources) and stays out of scope here — see §14.

10. Admin-configurable source language & AI output language

Prompted by a question during review: could an admin configure ingestion to fetch only German articles, so Mistral works from native German source text instead of translating? Investigating the actual ingestion pipeline surfaced one existing capability and one real gap.

Google News RSS is already admin-configurable for German. workspace_settings.google_news_rss_country and .google_news_rss_language (models/workspace_settings.py:54-56) already exist end-to-end: exposed via GET/PUT /settings (api/settings.py) and editable today in Settings → Sources (SourcesTab.tsx:182-200, free-text Country/Language fields feeding GoogleNewsRSSClient's ceid={country}:{language} query param). An admin can already set country=DE, language=de and Google News RSS will fetch German-language articles. already shipped No code change needed for this source.
NewsAPI.org and NewsData.io stay English-only. Both clients hardcode "language": "en" in their query params (news_client.py:43, newsdata_client.py:84) with no equivalent admin setting. confirmed — out of scope Making these configurable was considered and explicitly deferred for this pass. An admin who wants ingestion to be German-only should enable Google News RSS with country=DE/language=de and disable NewsAPI/NewsData.io using their existing newsapi_enabled/newsdata_enabled toggles (already shipped, Settings → Sources) — a configuration-only change, not a code change. Leaving any English source enabled alongside a German-configured Google News RSS will keep producing English articles from that source.

This was revisited after a follow-up decision: rather than leaving Mistral's output language as emergent behavior tied to whatever an admin configures Google News RSS to fetch, the workspace's new Main Language setting (introduced in §3) becomes the single, explicit source of truth for Signal content language — regardless of source-article language, and independent of the ingestion-side source-language question above.

Mistral always generates in the workspace Main Language. _SYSTEM_PROMPT in ai_client.py gains an explicit instruction — e.g. "Write every text field (summary, business_relevance, supporting_quote, the three outreach snippets) in {main_language_name}, regardless of the language of the source article. JSON keys and structure stay as specified." — with the language name interpolated from workspace_settings.main_language and passed into summarize_article(..., output_language=...) as a new parameter. confirmed This supersedes the earlier assumption above (no code change had been planned for the prompt): Mistral now translates into Main Language whenever the source article is in a different language, and leaves it unchanged when source and Main Language already match. This also resolves the "mixed English/German Signals with nothing to distinguish them" limitation flagged earlier — output is normalized to one language per workspace, so the missing language column on Signal stops being a correctness gap for current signals (though see the Main-Language-change edge case in §13).
Triage prompt unaffected. _TRIAGE_SYSTEM_PROMPT (relevance filtering, ai_client.py:63-71) only ever returns {"relevant": bool, "reason": ""} — reason is a short internal string, never shown to end users (checked: not rendered anywhere in the frontend), so it's left in English regardless of Main Language to avoid an unnecessary prompt/token-budget change to the cheap triage call. default

Data model changes

TableColumnType / defaultPurpose
workspace_settingsmain_languageString(8), NOT NULL, default "en"Admin-set workspace standard language; also the language Mistral is instructed to write Signals in
userspreferred_languageString(8), nullable, default NULLPer-user override; NULL means "follow main_language"

Two new Alembic migrations, following the same pattern as the existing users.role addition (see role-based-company-management-planning.html §4) — small nullable/defaulted columns, no backfill logic needed since both have safe defaults.

API surface changes

  • WorkspaceSettingsResponse / WorkspaceSettingsUpdate (schemas/settings.py) gain main_language; PUT /settings (admin-only) writes it same as every other workspace field.
  • UserResponse (schemas/auth.py) gains preferred_language: str | None and workspace_main_language: str — the latter is the workspace's current main_language, embedded directly into the existing GET /auth/me response. confirmed /auth/me is already called by every authenticated user regardless of role, so this avoids standing up a separate non-admin-gated endpoint just so regular users (who can't call the require_admin-gated GET /settings) can learn the workspace default.
  • New PATCH /auth/me/language endpoint, body {"preferred_language": "en" | "de" | null}, any authenticated user (not admin-gated) — writes their own preferred_language. null clears the override back to "follow workspace default."

11. Testing & quality gates

  • i18n completeness script — a small Node script (scripts/check-i18n-completeness.mjs) that diffs keys across every en/*.json / de/*.json pair and fails (non-zero exit) if any key is missing or orphaned in either locale. Runnable manually or via npm run lint today; slots into a CI job later without changes if/when this repo gets a pipeline (it currently has none — see §2 non-goals).
  • Frontend test framework — the frontend currently has zero test tooling (no Jest/Vitest/Testing Library in package.json). Recommend introducing Vitest + React Testing Library alongside this work, at minimum for a smoke test that renders a page in each locale and asserts no missing-key fallback strings leak through, plus a test for the useLocaleFormat hook's date/number output per locale.
  • Manual QA pass — native/professional review of the German translations (not machine-translation-only, given how visible UI chrome is), plus a visual pass for text expansion (German UI strings commonly run ~30% longer than English — check buttons, nav labels, and table headers don't clip or wrap awkwardly).

12. Implementation roadmap

Phase 0 Foundation

  • Alembic migrations: workspace_settings.main_language (default en), users.preferred_language (nullable)
  • Backend: add both fields to WorkspaceSettingsResponse/Update and UserResponse (incl. workspace_main_language on /auth/me); new PATCH /auth/me/language endpoint
  • Install react-i18next, i18next, i18next-browser-languagedetector
  • Create i18n/index.ts config: init, Login-page-only detector, namespace registration
  • Wrap app entrypoint with <Suspense> for async namespace loading
  • useLocaleFormat hook (dates/numbers) alongside the existing useTheme hook
  • Sync document.documentElement.lang on language change

Phase 1 Core chrome

  • Extract ProtectedLayout.tsx, ProfileMenu.tsx, Login.tsx into nav/auth/common namespaces
  • Add language switcher (workspace default → English → German cycle) next to the existing theme toggle, wired to PATCH /auth/me/language
  • Add Main Language control to admin Settings (Company tab or a new Localization section)
  • German translation of extracted core-chrome strings

Phase 2 Feature pages

  • Extract Dashboard.tsx, SignalsFeed.tsx (incl. STATUS_OPTIONS, ingestionStatusText()), SignalDetail.tsx, SettingsTargets.tsx
  • Extract all six settings/*Tab.tsx pages and ARTICLE_SOURCE_LABELS in api/types.ts
  • German translation of extracted feature-page strings

Phase 3 Formatting, backend errors & Mistral output language

  • Replace ad hoc toLocaleDateString() calls with useLocaleFormat
  • Convert hand-built template sentences to i18next interpolation/pluralization
  • Add error_code field to key API error responses; map codes in errors.json
  • Add explicit language directive to _SYSTEM_PROMPT in ai_client.py; thread main_language into summarize_article() as output_language

Phase 4 QA & tooling

  • Add scripts/check-i18n-completeness.mjs
  • Stand up Vitest + React Testing Library; locale-switch and formatting smoke tests
  • Native-speaker German review pass; text-expansion visual QA
  • Add eslint-plugin-i18next (or equivalent) to catch future hardcoded strings

13. Open questions & assumptions for follow-up sessions

Main Language change vs. historical Signals — Mistral is instructed to write in the current main_language at generation time; if an admin later changes the workspace's Main Language (e.g. en → de), previously generated Signal rows stay in whatever language they were written in — there's no retroactive re-translation, and no language column recording which one applies to a given row. Acceptable for an initial ship (Main Language is expected to be a rare, deliberate admin choice, not something toggled often) but worth a per-Signal language column as a fast-follow if this causes real confusion in a mixed-history feed.
"Bilingual" scope — UI strings follow the account/workspace language exactly as specced; Signal content now also follows Main Language (§9/§10), but the raw ingested article text does not and stays in its source language regardless of any setting here. Flagging because the product framing ("bilingual platform") could still be read by stakeholders as covering the underlying news articles too, not just the AI's summary of them — worth an explicit sentence in any external communication about this feature so expectations match what ships.
Backend error-code coverage — assumed only a handful of HTTPException call sites actually surface in the UI in ways users see today; the Phase 3 error-code work should start with an audit of which backend error strings the frontend currently displays verbatim, rather than instrumenting every endpoint speculatively.
CI — this repo has no GitHub Actions pipeline today; the completeness script and any new tests run locally/manually for now. Not proposing to introduce CI as part of this initiative, but noting it's the natural next home for these checks if/when one exists.

14. Future enhancements (explicitly out of scope here)

  • Machine-translating or re-generating the raw ingested article text itself into German (would require a translation-API integration, or re-ingesting from German-native sources) — distinct from Signal summaries, which Main Language already covers (§9/§10).
  • A language column on Signal/Article recording which language each row was actually generated/fetched in, for accurate display if Main Language is changed later — see the historical-drift question in §13.
  • Making NewsAPI.org's and NewsData.io's hardcoded "language": "en" query params admin-configurable, matching what Google News RSS already has — see §10.
  • Locale-prefixed routing (/de/...) for SEO — not relevant to an authenticated internal tool today, but worth revisiting if any public-facing pages are ever added.
  • Additional locales beyond English/German (the namespace-per-feature structure in §4 is designed so this is additive — new resource files, no component changes).
  • Region variants (de-AT, de-CH, en-GB) if the user base ever needs them.
