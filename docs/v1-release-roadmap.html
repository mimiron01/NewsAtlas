1. Goals & how to read this document

v1 has three goals: ship the six requested features, fix known bugs, and make the application safe to hand to real end users. Items are grouped into four phases by risk and dependency, not by the order they were listed in the original request:

  • Phase 1 — small, additive changes to existing tables/prompts/UI. No new concepts, no migrations beyond a couple of nullable columns.
  • Phase 2 — one new bulk-input surface (CSV import) built entirely on the existing TargetCompany model.
  • Phase 3 — the one genuinely new concept in this release (theme/topic search). Biggest, riskiest, and the only item that changes the app's mental model, so it's sequenced last and should get its own detailed planning doc (in this same docs/*-planning.html style) before implementation starts.
  • Phase 4 — the release gate. Security and stabilization aren't "a phase you do last instead of skip" — they gate whether v1 ships at all, regardless of which feature phases are done.
Decision: phases are sequential for planning/estimation purposes, but Phase 4's security items can and should start in parallel with Phase 1 — nothing about hardening the app depends on the new features existing first.

2. Phase 1 — Quick wins

Phase 1 Foundations

2.1 AI uses target-company industry & keywords backend only

Original ask: "Nutzen von Branche und ggf. Schlüsselwörtern auch für die AI nutzen" — industry and keywords should feed the AI, not just ingestion.

Partially done already: AIClient.summarize_article() (backend/app/services/ai_client.py:249) already accepts an industry parameter and includes a Target industry: … line in the prompt when set, and ingestion.py:522 already passes industry=target_company.industry on the full-summarization call. Do not re-implement this — extend it.
Actual gaps:
  • AIClient.triage_article() (ai_client.py:320) — the cheap first-pass relevance filter that runs before summarization and decides whether an article is worth analyzing at all — takes no industry or keywords parameter whatsoever. It only ever sees offering_description + target company name + article. This is the more consequential gap: articles get triaged out (see §2.4) using strictly less context than the model that would otherwise judge them.
  • keywords is never passed to either AI call. It is only ever used in news_query.build_or_query() to build the ingestion search query (ingestion.py:287, :402). A company's keyword list (e.g. "Tier 1 supplier", "EV battery", "manufacturing") encodes exactly the kind of context that should inform relevance scoring, and today it's discarded once the article is fetched.

Change:

def triage_article(

    article_title, article_description,

    keywords: list[str] | None = None,
    headline_only: bool = False,
) -> tuple[TriageResult, MistralUsage]:
    ...
    context_lines = [f"Our offering: {offering_description or '(not provided)'}",
                      f"Target company: {target_company_name}"]
    if industry:
        context_lines.append(f"Target industry: {industry}")
    if keywords:
        context_lines.append(f"Target company keywords/context: {', '.join(keywords)}")
    context_lines += [f"Article title: {article_title}", f"Article description: {article_description or '(no description available)'}"]

# summarize_article() — add the missing keywords param next to the existing industry one
def summarize_article(..., industry: str | None = None, keywords: list[str] | None = None, ...):
    ...
    if keywords:
        context_lines.append(f"Target company keywords/context: {', '.join(keywords)}")

Update both call sites in ingestion.py (the triage call at :490 and both summarize_article calls at :516 and :605) to pass keywords=target_company.keywords, and the triage call to also pass industry=target_company.industry.

No schema changes — industry and keywords already exist on TargetCompany.

Acceptance criteria: both the triage and summarization prompts include industry/keywords whenever the target company has them set; existing ai_client unit tests updated to assert the new context lines; a company with no industry/keywords produces byte-identical prompts to today (no regression for the common case of a sparsely-configured company).

2.2 Unternehmensprofil (offering description) — guidance over new fields frontend + i18n

Original ask: "mehr Filter wenn möglich bzw. genauer darstellen, was darin enthalten sein muss und wie es verwendet wird" — more filters if possible, or failing that, make clearer what must be in it and how it's used.

Decision (this conversation): keep the single free-text offering_description field — do not split it into structured sub-fields for v1. Instead: (a) make the "how it's used" and "what belongs in it" explicit in the UI, and (b) satisfy the "more filters" half of the ask via §2.1 (industry/keywords now reach the AI) and Phase 3 (theme search) rather than new structured profile fields. This is a scope decision worth keeping visible — don't let a later session quietly re-open "should this be structured fields" without checking here first.

Current state: CompanyTab.tsx (frontend/src/pages/settings/CompanyTab.tsx:65-73) renders one <textarea> with only a translated placeholder — no explanation of what the text is for, no examples, no quality signal.

Change (UI/copy only, no backend changes):

  • Add an explainer directly above the textarea: this text is sent to the AI on every incoming article to score relevance and draft outreach copy — it should describe what you sell, who buys it, and what problem it solves, in plain language (not marketing copy).
  • Add a collapsible "Show example" block with one good example (~4-6 sentences: product, buyer, problem solved, differentiator) and one clearly-too-vague counterexample, so users can self-calibrate.
  • Add a soft word-count nudge under the textarea (not a hard validation error) — e.g. below ~30 words, show a hint that more detail improves signal accuracy.

Acceptance criteria: new company.offeringHelp.* i18n keys added to en/settings.json and de/settings.json; no change to WorkspaceSettings schema or the /settings API contract.

2.3 Google News RSS query tightening & source allowlists backend + frontend + migration

Original ask: the vague final bullet ("Spezifizierung bzw. Verbesserung") was clarified in this conversation to mean: improve the Google News RSS search query for better results, optionally restricted to specific sources.

Current state: news_query.build_or_query() (backend/app/services/news_query.py:7) builds a flat name OR kw1 OR kw2 OR … query, shared by all three providers (NewsAPI.org via news_client.py, Google News RSS via google_news_rss_client.py, NewsData.io via newsdata_client.py). A generic keyword can match on its own without the company name appearing anywhere in the article. There is also no way to restrict Google News RSS to a set of trusted domains — Google's RSS search supports a site: operator, but nothing in the codebase constructs one.

Decisions (this conversation):
  • Scope the query-logic change to Google News RSS only — introduce a new, separate query builder rather than changing build_or_query(), which stays untouched for NewsAPI.org/NewsData.io (out of scope; those providers weren't part of the complaint and have their own relevance ranking).
  • New logic: require the company name AND at least one keyword — name AND (kw1 OR kw2 OR …) — falling back to just name when a company has no keywords, so this never makes a bare-name company unsearchable.
  • Source allowlists exist at both levels: a workspace-wide default list of trusted domains, and a per-target-company list that adds to it.

Change:

# news_query.py — new builder, additive; build_or_query() is untouched
def build_google_news_query(name: str, keywords: list[str], sources: list[str]) -> str:
    terms = _dedupe_quote([kw for kw in keywords])  # existing quoting logic, extracted/reused
    base = f'"{name}"' if " " in name else name
    if terms:
        base = f"{base} AND ({' OR '.join(terms)})"
    if sources:
        site_clause = " OR ".join(f"site:{domain}" for domain in sources)
        base = f"({base}) ({site_clause})"
    return base

GoogleNewsRSSClient.fetch_articles() (google_news_rss_client.py:30) switches from build_or_query(name, keywords) to this new builder, and gains a sources: list[str] parameter. The effective source list passed in from ingestion.py is target_company.google_news_source_allowlist or [] unioned with workspace_settings.google_news_source_allowlist or [] (union, i.e. "extend" — see deferred decision below on override vs. extend semantics).

Schema changes:

TableColumnTypeNotes
workspace_settingsgoogle_news_source_allowlistARRAY(String), default []Workspace-wide default domains, same pattern as existing google_news_rss_country/_language.
target_companiesgoogle_news_source_allowlistARRAY(String), nullable, default []Per-company additions.

API/UI: new field on the settings PUT payload + a new tag-input in a "News Sources" area of CompanyTab.tsx (or its own settings tab); reuse TagInput (already used for company keywords in SettingsTargets.tsx:192) for both the workspace-level and per-company lists, with basic domain-format validation (reject anything that isn't a bare hostname — no scheme, no path — since it's spliced directly into a query string, not fetched).

Acceptance criteria: Alembic migration for the two columns; unit tests for build_google_news_query covering no-keywords, no-sources, and both-present cases; existing NewsAPI.org/NewsData.io tests unaffected (they still call build_or_query).

2.4 Faster access to skipped/dismissed signals backend + frontend

Original ask: "Möglichkeit einbauen schneller an übersprungene Signale zu kommen."

Current state — this is actually two separate "skipped" systems today, both buried:
  • Triaged-out articles (never became a Signal at all): Article.skip_reason (backend/app/models/article.py:61) set to "triaged_out" by the AI triage step. Surfaced only via GET /articles/skipped?reason=triaged_out, rendered in ReviewTab.tsx, which lives under Settings — not the primary navbar (ProtectedLayout.tsx only links Dashboard/Signals/Targets). A user has to know this tab exists.
  • Dismissed signals (a real Signal the user explicitly set to SignalStatus.DISMISSED via SignalDetail.tsx:50's updateStatus()): reachable only through the generic status filter dropdown in SignalsFeed.tsx:20 — one option among four, no dedicated view, no count shown anywhere.
Neither system has an undo affordance at the moment of action — ToastContext.tsx only supports a plain auto-dismissing message (message + variant, no action button) — so reverting a skip means navigating back through filters/tabs after the fact, which is exactly the friction this item is about.

Change:

  • Visibility: extend the existing dashboard counts endpoint (backend/app/api/dashboard.py:24, which already computes new_signal_count/favorite_count/open_todo_count via simple .count() queries) with dismissed_signal_count and skipped_article_count, using the same query pattern. Surface both as a dashboard card ("N skipped — review them") linking to a consolidated view, rather than inventing new top-level nav real estate for what's a secondary workflow.
  • Consolidated view: a single page/tab showing both lists together (reusing the existing /articles/skipped?reason=triaged_out and /signals?status=dismissed endpoints — no new list endpoints needed), so "where did my skipped stuff go" has one answer instead of two.
  • Undo: extend ToastContext's Toast type with an optional action: { label: string; onClick: () => void }, rendered as a button inside the toast, and extend its auto-dismiss timeout when an action is present (e.g. 6-8s instead of 3s). Wire it into both dismiss points: SignalDetail.updateStatus("dismissed") shows "Signal dismissed — [Undo]" that calls updateStatus(previousStatus); ReviewTab.handlePromote's inverse (a "skip" action doesn't currently exist as a distinct UI action beyond triage itself, so this mainly applies to the signal-dismiss path).

Acceptance criteria: dashboard response schema gains the two new counts (with tests); a consolidated skipped view reachable in ≤2 clicks from the dashboard; dismissing a signal shows an undo toast that correctly reverts status.

3. Phase 2 — CSV import of target companies (Salesforce)

Phase 2 Bulk import

Original ask: "Import von zu suchenden Unternehmen via csv von Salesforce."

Current state: no CSV/bulk-import capability exists anywhere in the codebase. Target companies are created one at a time via POST /target-companies, driven by the form in SettingsTargets.tsx:50.
Decisions (this conversation):
  • On a name collision with an existing target company, skip that row (don't create, don't merge, don't fail the whole batch) and report a per-row skipped count/reason in the response.
  • Expected scale is small — under ~100 rows per import — so this is a synchronous request/response, no background job or progress bar needed for v1.

Column mapping: Salesforce Account list exports commonly include Account Name, Industry, and free-text fields that don't map cleanly to keywords. Rather than hard-coding Salesforce's exact default column names (which vary by org/report config), the importer should:

  • Require a name column (case-insensitive match against common headers: Account Name, Company, Name).
  • Optionally map an Industry/industry column to TargetCompany.industry.
  • Have no first-class keyword column from Salesforce (it doesn't have an equivalent concept) — leave keywords empty on import; users add them manually afterward, same as today's single-add form already requires.
  • Show a column-mapping preview step before committing (parse headers, let the admin confirm/adjust which CSV column maps to name / industry, rather than assuming fixed header names) — this is the one part of the flow that has to tolerate real-world export variance.

API: new endpoint, admin-only (matches the existing admin-gating pattern on target_companies.py create/delete):

POST /target-companies/import  (multipart/form-data, file field "csv")
→ 200 { created: [TargetCompany...], skipped: [{row: int, name: str, reason: "duplicate"}...],
        errors: [{row: int, reason: str}...] }

Reuse the existing TargetCompanyCreate validation (name required, keywords array, industry optional) per row rather than inventing a parallel schema. Enforce a request body size cap (ties into the Phase 4 hardening item on request size limits) and a hard row-count ceiling (e.g. 500) so an oversized file can't be used to mass-create companies and blow through news-provider rate limits on the next ingestion run.

Frontend: new "Import CSV" button/flow on SettingsTargets.tsx — file picker → column-mapping preview → submit → result summary (N created, N skipped as duplicates, N errored) using the existing toast pattern.

Acceptance criteria: importing a file with a mix of new/duplicate/malformed rows creates only the valid new ones, reports the rest accurately, and never partially commits a row (each row either fully succeeds or is reported, no half-created companies); CSV parsing rejects non-CSV content types and oversized files before parsing.

4. Phase 3 — Theme/topic search ("Automotive", "Startup Series B")

Phase 3 New concept — plan in detail before building

Original ask: "Ermöglichen von Suche nach Begriffen wie 'Automotive', 'Startup Series B' etc. anstatt nur spezifische Firmensuche."

Scope flag: this is the one item in the roadmap that doesn't fit the existing data model at all, and it's sized differently from everything else here — it deserves its own docs/theme-search-planning.html with the same level of code-grounded detail as this section's siblings, written right before implementation starts (so it reflects whatever Phase 1/2 changes have landed by then). What follows is the scope and shape, not a full spec.

Decision (this conversation): theme search is a new parallel concept, not a fuzzier version of company keyword matching. A "Theme Watch" (e.g. "Automotive", "Startup Series B") is its own trackable thing, independent of any single named company, producing its own feed of matching articles/signals — with an optional promote-to-Target-Company flow for companies discovered inside a theme's results.

Why this doesn't fit today's model: every ingestion path (ingestion.py, all three provider clients) is anchored to exactly one TargetCompany row per fetch, and article_mentions_company() (news_query.py:32) grounds every article back to that one company's name/keywords before a Signal can be attributed to it (see ingestion-reliability-planning.html §5). A theme like "Automotive" has no single name to ground against — it's a topic, and the "target" of a resulting signal is whatever company the article turns out to be about, not known in advance.

Shape of the solution (for the follow-up planning doc to detail):

  • New model: ThemeWatch (name, query terms, optional industry/source-allowlist reusing the §2.3 pattern, owner/follow scoping mirroring TargetCompany/CompanyFollow).
  • Ingestion: a theme-scoped fetch that queries providers on the theme's terms alone (no name-anchoring), and — critically — needs its own grounding/relevance step, since article_mentions_company's single-name check doesn't apply. This likely means every theme-matched article needs an AI extraction pass to identify *which* company (if any) it's actually about, before it can become an attributable signal.
  • Feed: a theme results view parallel to SignalsFeed.tsx, showing raw/AI-summarized theme matches rather than per-company signals.
  • Promotion: reuse the exact "promote" pattern already built for triaged-out articles in ReviewTab.tsx:27 (handlePromote → POST /articles/{id}/create-signal) — a theme match that names a real company gets a "Track this company" action that creates a TargetCompany from the extracted name.
  • AI cost: theme queries are inherently broader than a named company's keyword list, so expect meaningfully higher triage/summarization volume per theme than per company — the follow-up doc should size this against max_articles_per_company_per_run's existing per-company cap (workspace_settings.py:23) and propose an equivalent per-theme cap.

Acceptance criteria (for this roadmap doc, not the feature): a dedicated planning doc exists, reviewed and approved, before any theme-search code is written — this section is scope-setting, not an implementation spec.

5. Phase 4 — Security hardening & stabilization (release gate)

Phase 4 What actually gates v1

Already done: security-review.html found and remediated 17 issues (TLS, exposed Postgres, open signup, JWT defaults, rate limiting, container hardening, URL-scheme validation, password policy, etc.) as of 2026-07-03. A 2026-07-21 spot-check confirmed these are still true and one item improved further: config.py still refuses to boot in production on a default/short JWT_SECRET or missing invite code; /auth/login (10/min) and /auth/signup (5/hour) still carry rate limits; is_safe_article_url's scheme guard is applied uniformly across all three news clients; and the review's own L2 finding ("aging python-jose/passlib") is now fully resolved — the backend has since migrated to pyjwt==2.13.0/bcrypt==5.0.0.

Deeper audit (2026-07-21) — authorization, injection, secrets, dependencies

The 2026-07-03 review was scoped to infrastructure/transport/basic-auth. This pass targeted everything it didn't touch: per-endpoint authorization (IDOR), injection surfaces, secrets handling end-to-end (not just at rest), and dependency currency, across both backend and frontend. Ranked by severity:

Checked and confirmed clean — no action needed, but recorded so a future pass doesn't re-check these from scratch:
  • Every other endpoint in signals.py, signal_todos.py, dashboard.py, admin.py, settings.py, ingestion.py, digest.py correctly uses scope_to_follows/accessible_signal_row/get_accessible_signal or require_admin as appropriate.
  • No raw-SQL/string-formatted query construction anywhere in the backend — the ORM is used consistently. No log statement anywhere logs a JWT, request body, or the Mistral/NewsData API keys.
  • GET /settings is admin-gated and the response schema (schemas/settings.py) only ever returns *_configured/*_source/*_last4 for the two secret fields — the raw key never round-trips to any browser. This confirms the plaintext-at-rest gap below is a DB-access-only exposure, not also an API-level one.
  • CORS is configured with explicit origins (not wildcard) alongside credentials — correct combination.
  • No XSS path in the frontend: no dangerouslySetInnerHTML/innerHTML/markdown renderer anywhere; AI-generated signal text and article titles are all plain JSX interpolation, which React escapes by default.
  • Both target="_blank" links in the whole frontend (article links in SignalDetail.tsx:186 and ReviewTab.tsx:63) already carry rel="noreferrer" — no reverse-tabnabbing gap.
  • No sensitive data in client-side storage beyond the already-known/accepted JWT; the only other localStorage use is a non-sensitive theme preference.
  • No committed secrets anywhere in git history — only .env.example was ever committed, never a real .env.
  • Dependencies are current-generation on both sides: backend on pyjwt/bcrypt (see above); frontend on React 18.3, Vite 5.4, React Router 6.26, TypeScript 5.5 — nothing abandoned or notably outdated.
Still open — plaintext secrets at rest: WorkspaceSettings.mistral_api_key and .newsdata_api_key (backend/app/models/workspace_settings.py:51,77) are stored as plaintext Text columns in Postgres — readable by anything with DB access (a backup, a support person running a query, a future read-only reporting integration), even though the API layer correctly never exposes them (see above).

Change: encrypt these two columns at rest (application-level encryption with a key from the environment — e.g. Fernet via a new APP_SECRET_KEY env var, encrypt on write in api/settings.py, decrypt on read where the key is actually used in ai_client/newsdata_client construction) rather than plaintext. Never log the decrypted value; the existing "empty string = fall back to env var" convention stays unchanged, just applied to the decrypted value.

New attack surface introduced by this roadmap's own features

Every new user-facing input surface added above needs the same scrutiny the existing ones got in the prior review:

  • CSV import (Phase 2): file size cap, row-count cap, and content-type validation before parsing (already specified in §3); CSV values are never evaluated as formulas by this app, but if an exported/re-downloaded report ever opens in Excel/Sheets, values starting with =/+/-/@ are a classic CSV-injection vector for whoever opens that export later — sanitize (prefix with a leading ' or reject) on the way in, not just on the way out.
  • Source allowlist domains (§2.3): validate as bare hostnames only (already specified) — these get spliced into a query string, not fetched directly by the server, so this is a query-integrity concern rather than SSRF, but malformed input shouldn't be allowed to break the constructed URL.
  • Theme search (Phase 3): a new endpoint that triggers external provider calls on user-supplied free text — needs the same per-workspace rate-limit enforcement news_rate_limiter.py already applies to company ingestion, sized for however many active theme watches exist, not just companies.

Stabilization

  • Carry over the one known-and-left-open bug from dashboard-favorites-todos-planning.html: .signal-row-actions action buttons overlapping the title on narrow screens — flagged there as needing "its own redesign decision," which v1 is exactly the right point to make.
  • Run pip-audit/npm audit (or equivalent) once before tagging v1 to catch anything newly disclosed — the prior review's L2 finding (aging python-jose/passlib) is already resolved (migrated to pyjwt/bcrypt), this is just a final currency check, not new remediation work.
  • General regression pass across ingestion, digest email delivery, and auth flows after Phases 1-3 land — not a fixed list of bugs, but a scheduled QA pass before tagging v1.

Acceptance criteria: ai_usage.py's require_admin gap closed and covered by a regression test (a non-admin request must 403); keywords/industry length caps added to the target-company schemas; create_signal_from_skipped_article rate-limited to match the other manual-trigger endpoints; CSP header added to the Caddyfile; the shared-TargetCompany-edit question resolved with the user (fix or explicitly accept as intended); secrets-at-rest fix shipped with a migration for existing plaintext values (re-encrypt in place); dependency audit run with zero known-critical/high CVEs outstanding; the carried-over mobile layout bug either fixed or explicitly re-deferred with a reason; new endpoints from Phases 2-3 covered by the same rate-limiting/size-cap conventions as existing ones.

7. Open questions deferred to implementation

  • §2.3 allowlist semantics: "workspace default + per-company additions" was decided as the shape, but override-vs-union needs a final call at implementation time. This doc's working assumption is union (workspace defaults always apply; a company's list only adds more sources, never removes the workspace defaults) since that's the simpler mental model — flag if a full per-company override is actually wanted instead.
  • §3 CSV column detection: the exact header-matching heuristic (case-insensitive fuzzy match against a known-header list vs. a strict user-driven mapping step every time) should be finalized against a real sample Salesforce export if one becomes available before implementation.
