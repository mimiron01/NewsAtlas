1. Decisions already made

Resolved in the planning conversation on 2026-07-22, recorded here so implementation doesn't have to re-litigate them:

Decision — parallel concept, not fuzzy company matching: a Theme Watch is its own trackable thing (a ThemeWatch row), independent of any single named company, producing its own feed. This was already decided in v1-release-roadmap.html §4 and isn't revisited here.
Decision — new ThemeMatch model, not a nullable-company Signal. Theme matches live in their own table with their own feed endpoint, parallel to Signal. This keeps Signal.article_id/the company-join assumption baked into signal_queries.py, the dashboard, and the digest completely untouched — no risk to the Phase 1/2/4 work just shipped. A theme match is promoted into a real per-company pipeline only once its company is tracked (see §7's promotion endpoint), not by mutating it into a Signal in place.
Decision — shared catalog, any authenticated user can create a theme. ThemeWatch mirrors TargetCompany/CompanyFollow exactly: case-insensitive dedupe by name (reusing the get_or_create_company pattern), many users can follow the same theme without duplicating ingestion for it, any user can create one (not admin-gated) exactly like adding a target company today.
Decision — company-less matches are kept, not dropped. An article that matches a theme's query terms but where the AI extraction pass can't confidently name a specific company still appears in the theme's feed as a raw AI-summarized item with extracted_company_name = NULL. Industry-trend awareness (not only lead-finding) is part of what a theme watch is for.
Decision — "Track this company" creates/follows only, no forced signal. When a match names a real, not-yet-tracked company, the promotion action creates the TargetCompany (case-insensitive deduped, same as CSV import/single-add) and follows it for the clicking user. It does not force a paid summarization call to retroactively turn that specific match into a Signal — the newly tracked company gets ordinary coverage starting from the next scheduled ingestion run. This avoids stacking a third "forces a paid Mistral call" endpoint on top of the two (create_signal_from_skipped_article, backfill) that already need cooldown/rate-limit protection.
Decision — Google News RSS only for theme ingestion. NewsAPI.org and NewsData.io are not queried for themes in v1. NewsData.io's per-call credit cost and its native-dedupe/historical-backfill machinery are built around one company at a time; a broad topical query would be expensive and the backfill concept doesn't map onto a theme at all. NewsAPI.org has the same "no name to ground against" problem ingestion-reliability-planning.html §5 already fixed for companies. Google News RSS is free, keyless, and already has the exact query-tightening/site-allowlist pattern (build_google_news_query, §2.3 of the v1 roadmap) this doc reuses.
Decision — dedicated cost controls, not reuse of the per-company cap. A new workspace-wide max_articles_per_theme_per_run setting (same shape/UI as max_articles_per_company_per_run) plus a hard ceiling on total active ThemeWatch rows per workspace (proposed default: 10, admin-configurable — see §9). A theme's query is inherently broader than a named company's keyword list, so it fans out into more candidate articles per run; uncapped concurrent themes is a materially bigger cost blast radius than uncapped companies ever was.
Decision — cross-path dedup, not two independent dedupe pools. The existing embedding-similarity duplicate check extends across both the per-company (Article/Signal) and per-theme (ThemeMatch) paths rather than each maintaining its own blind pool — see §6.
Decision — new top-level nav item. Unlike the deliberately-secondary /skipped page, "Themes" gets its own entry in ProtectedLayout.tsx's primary nav alongside Dashboard/Signals/My companies — the roadmap describes this as a first-class parallel feed, not a corner case.
Decision — excluded from the daily digest for v1. The digest stays company-Signal-only. Theme matches are in-app only until there's a reason to believe people want a themes digest too.
Decision — naming. "Theme Watch" in code/models, "Themes" as the nav/UI label (matches "My companies" being the UI label for TargetCompany).

3. Query building & provider scope

New function in news_query.py, alongside build_or_query/build_google_news_query:

def build_theme_query(query_terms: list[str], sources: list[str] | None = None) -> str:
    """Pure OR of the theme's own terms — unlike build_google_news_query, there's no
    company name to require, so this is what that function degrades to when a company
    has no keywords, generalized to the case where there's no name at all."""
    terms = _quote_terms(query_terms)  # reuses the existing private helper
    base = " OR ".join(terms)
    if sources:
        site_clause = " OR ".join(f"site:{domain.strip()}" for domain in sources if domain.strip())
        if site_clause:
            base = f"({base}) ({site_clause})"
    return base

GoogleNewsRSSClient.fetch_articles() needs no changes — it already accepts sources (added in v1 roadmap §2.3); a theme-ingestion call site just builds its query via build_theme_query instead of build_google_news_query and passes the theme's own name parameter... except fetch_articles(*, name, keywords, since, sources) currently requires name and calls build_google_news_query(name, keywords, sources) internally. Two ways to reconcile this, pick one during implementation:

  • (a) Add an optional query_override: str | None param to fetch_articles(); when set, skip build_google_news_query entirely and use it as the raw query string. Theme ingestion passes query_override=build_theme_query(theme.query_terms, sources). Smaller diff, keeps the company path completely untouched.
  • (b) Give GoogleNewsRSSClient a second method, fetch_articles_for_query(query, since), factoring the shared feed-fetch/parse/date-filter logic (lines 39–57 of the current file) out of fetch_articles into a private helper both call. Cleaner separation, slightly bigger diff.
Proposed default: (a) — smaller, lower-risk diff against a file that already works correctly for every existing company. Flag if a cleaner split is preferred.

4. AI extraction & relevance

The existing grounding guard (article_mentions_company) and the company_mentioned/company_mismatch machinery in ai_client.py/ingestion.py both assume exactly one known company name to check identity against (ingestion-reliability-planning.html §5). A theme has none — the whole point is that the matching company is unknown until the AI reads the article. Two new pieces are needed:

4.1 Theme-specific triage

A theme's query is broader than a company's keyword list, so the cheap pre-filter matters at least as much here, arguably more. New method, same shape as triage_article but checking topical relevance to the theme rather than company identity:

def triage_theme_article(
    self, *, company_name, offering_description, theme_name, query_terms,
    industry=None, article_title, article_description, headline_only=False,
) -> tuple[TriageResult, MistralUsage]

Reuses the existing TriageResult shape (relevant: bool, reason: str) and the same fail-open-on-AIClientError convention already in _process_new_articles.

4.2 Combined extraction + summarization

One call does both jobs (not two, to avoid doubling the per-article AI cost that's already the doc's biggest stated risk):

class ThemeArticleResult(AISummaryResult):
    # Everything AISummaryResult already has (summary, business_relevance,
    # supporting_quote, relevance_score, signal_type, confidence, entities), plus:
    extracted_company_name: str | None = None  # None = no specific company (kept, see §1)

def summarize_theme_article(
    self, *, company_name, offering_description, theme_name, query_terms, industry=None,
    article_title, article_description, output_language="en", headline_only=False,
) -> tuple[ThemeArticleResult, MistralUsage]

The prompt asks the model to (a) summarize and score relevance the same way summarize_article does, framed against the theme + our offering rather than a specific target company, and (b) name the specific company the article is actually about, if any, in extracted_company_name — verbatim as it appears in the article, not normalized against existing TargetCompany rows (that matching happens in Python, see 4.3). No company_mentioned field — there's no single company to assert the truth of, so the concept doesn't apply; a hallucinated extracted_company_name is caught the same way any other AI content is: it's just wrong text a user can see and ignore, not something that mis-attributes a signal to the wrong company's feed (there's no per-company feed for it to land in) the way the original grounding bug did.

4.3 Auto-linking to an existing tracked company

After summarize_theme_article returns, before creating the ThemeMatch row: if extracted_company_name is set, do a case-insensitive lookup against TargetCompany.name (same query get_or_create_company already does for dedupe) and set matched_target_company_id if found. This is metadata only in v1 — it does not trigger anything automatically (no signal, no notification); it just changes what the frontend renders ("Tesla — already tracked" instead of a "Track this company" button, see §8).

5. Ingestion pipeline integration

Extends run_ingestion() rather than introducing a second scheduler entry point, so theme and company fetches share the same run, the same news_source_usage_logs-backed rate limiting against ArticleSource.GOOGLE_NEWS_RSS, and the same cancellation checkpoints:

  • After the existing per-TargetCompany loop (current ingestion.py lines 154–190), add a second loop over active ThemeWatch rows, structured identically: cancellation check, per-theme headroom check (check_headroom/wait_for_minute_headroom against GOOGLE_NEWS_RSS only, since that's the only provider in play), fetch via build_theme_query (§3), triage via triage_theme_article, summarize via summarize_theme_article (§4).
  • IngestionRunResult gains theme_matches_created/themes_processed counters, surfaced in IngestionRunStatus/the progress UI the same way signals_created/companies_processed already are.
  • Cap via the new max_articles_per_theme_per_run setting (§1), same "newest N by published_at, 0 = unlimited" semantics as the company-side cap.
  • No grounding-guard equivalent of article_mentions_company is needed before storing a ThemeMatch — a theme's query terms are the whole point of the match (there's no separate "identity" to verify the way a company name needs verifying), so the AI's own topical-relevance judgment via triage is the only gate, same as the company path's triage step.

6. Cross-path deduplication

Per §1's decision, the same embedding-similarity check (_find_duplicate, cosine_similarity, mistral_dedupe_similarity_threshold) spans both tables rather than each maintaining a blind pool:

  • Exact-URL check (cheap, mandatory, do this regardless of embeddings): before creating a ThemeMatch, check whether an Article with the same url already exists; if so, skip creating the ThemeMatch entirely (the story is already covered via some company's feed). Symmetric on the company side: before creating an Article, also check ThemeMatch.url. Both tables keep their own unique constraint on url independently — this is an application-level cross-check, not a shared DB constraint.
  • Embedding similarity (catches near-duplicate stories from different outlets, not just the same URL): when building the candidate pool for a company's _find_duplicate call, additionally include ThemeMatch rows whose matched_target_company_id equals that company. When checking a new ThemeMatch for duplicates, the candidate pool is other ThemeMatch rows for the same theme, plus (if extracted_company_name resolved to a matched_target_company_id) that company's own recent Article rows.
The embedding-similarity half is more involved than the exact-URL half (two different tables feeding one similarity scan) — if implementation time is tight, ship the exact-URL check first (it's cheap and catches the common case: the same wire-service story appearing under a company's own keyword search and under a theme's broader query) and treat the embedding cross-check as a fast-follow rather than blocking v1 theme search on it.

8. Frontend

  • Nav: new NavLink to="/themes" in ProtectedLayout.tsx's primary nav block, new icon in ./icons/NavIcons, new nav:links.themes i18n key.
  • SettingsThemeWatches.tsx (new, mirrors SettingsTargets.tsx): add-theme form (name + query terms via TagInput, same pattern as company keywords + industry/source-allowlist fields), tracked-themes list with edit/mute/pause/delete, creator-or-admin edit gating identical to SettingsTargets.tsx's canEdit().
  • ThemesFeed.tsx (new, mirrors SignalsFeed.tsx): theme filter dropdown, status filter, search/sort — same polling-for-in-flight-ingestion convention (POLL_INTERVAL_MS, sawRunningRef) since it's watching the same IngestionRunStatus. Row rendering reuses SignalRow-equivalent presentation: a "Track this company" button appears only when extracted_company_name is set and matched_target_company_id is NULL; when matched_target_company_id is set, show a plain badge/link to that company instead (it's already tracked, nothing to do).
  • Dashboard: no new stat tile proposed for v1 (unlike the Skipped tile) — themes get their own primary nav entry, so a dashboard summary card is optional polish rather than required for discoverability. Flag if a "N new theme matches" tile is wanted anyway.

9. Cost controls & rate limiting

  • workspace_settings.max_articles_per_theme_per_run — new column, same default (10) and "0 = unlimited" convention as max_articles_per_company_per_run, same Settings UI section.
  • workspace_settings.max_active_theme_watches — new column, proposed default 10 (admin-configurable, same numeric-input pattern as the other caps). Enforced in POST /theme-watches: reject creation (or reactivation via PATCH is_active=true) once the workspace's active-theme count is at the ceiling. proposed default — flag if 10 is too low/high for expected usage.
  • Theme ingestion shares google_news_rss_max_requests_per_minute with company ingestion (same provider, same headroom check) — no separate theme-specific rate limit needed, since the existing per-source limiter already throttles combined usage.

10. Security considerations

Every new user-facing input surface gets the same scrutiny the v1 roadmap's Phase 4 gave existing ones:

  • query_terms/industry/google_news_source_allowlist on ThemeWatch get the identical length/count caps and hostname validation already added to TargetCompany (v1 roadmap §5) — reuse the same Pydantic validators, don't reinvent them.
  • POST /theme-watches/{id}/track-company creates a TargetCompany — same collision/dedupe handling as POST /target-companies already has (case-insensitive, returns the existing row rather than erroring).
  • New endpoints follow the existing scope_to_follows/require_admin conventions verbatim — no new authorization pattern to invent, just mirroring target_companies.py/signals.py.
  • No new rate-limited "forces a paid AI call" endpoint is introduced (per §1's decision that promotion doesn't force a summarization call), so no new cooldown/rate-limit wiring is needed beyond what §9 already covers at the ingestion level.

11. Explicitly out of scope for v1

  • Favorites/todos on ThemeMatch (§2.3 note).
  • Digest email inclusion (§1 decision).
  • NewsAPI.org/NewsData.io as theme providers (§1 decision).
  • Automatically converting an already-tracked company's theme match into a Signal (§1 decision — "track" never forces a summarization call, even when the company turns out to already be tracked).
  • Embedding-based cross-table dedup, if implementation time is tight (§6 — exact-URL dedup is the mandatory floor).

13. Open questions deferred to implementation

  • §3 query builder integration: option (a) vs (b) for reconciling GoogleNewsRSSClient.fetch_articles()'s required name param with a theme having none — pick whichever reads cleaner once the surrounding ingestion-loop code is actually being written.
  • §7 track-company permission: any follower of the theme, or admin-only? Creating a TargetCompany is already something any user can do today (POST /target-companies is not admin-gated), so admin-gating it only when reached via a theme match would be an inconsistent carve-out — proposed default is "any authenticated user," same as direct company creation, but flag if theme-sourced company creation should be more restricted.
  • §9 default cap values: max_articles_per_theme_per_run=10 and max_active_theme_watches=10 are placeholders reasoned from "same order of magnitude as the company cap," not from any real usage data (none exists yet for a feature that hasn't shipped) — revisit after the first few weeks of real theme usage.
  • §4 exact prompt wording: this doc specifies the shape of triage_theme_article/summarize_theme_article's inputs/outputs, not their exact prompt text — that's a normal implementation-time task, same as any other prompt, not an architectural fork.
  • Historical backfill for themes: NewsData.io's backfill concept is explicitly out of scope (§1), but should a newly created theme get some form of "search back N hours further than the standard lookback window" on its first run, the way a new company implicitly benefits from MIN_LOOKBACK_HOURS? Proposed default: no special-casing — a theme's first run uses the same lookback_hours calculation as every other run, exactly like a newly added company does today.
