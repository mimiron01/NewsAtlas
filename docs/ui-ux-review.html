NewsAtlas — UI/UX Design Review

How the current frontend measures up against usability and interaction-design best practices, and what would make it feel fun, easy, and intuitive.

Scope: React/TypeScript SPA in frontend/ (login, signals feed, signal detail, company profile, target companies) and the daily digest email HTML rendered by backend/app/services/digest.py
Reviewed at: commit b5d5fd3
Method: manual source review of every page/component and the shared stylesheet, no dynamic browser testing performed
Not in scope: backend API design, data model, deployment/security posture (see docs/security-review.html)

NewsAtlas is a small B2B tool for sales/BD teams: it ingests news about a workspace's target companies, has an AI summarize why each article matters and draft an outreach snippet, and presents the result as a triage queue ("signals") in a dashboard plus a daily email digest. The frontend is a lean, unstyled-framework React app — no component library, no CSS framework, a single hand-written stylesheet (frontend/src/styles.css). That minimalism is a reasonable starting point, but several gaps currently work against the goal of the app feeling fun, easy, and intuitive — most importantly for the person who has to open this dashboard every day and triage a growing list of signals.

F1. No responsive/mobile layout at all High

frontend/src/styles.css — .app-shell, .sidebar, .content (no @media queries exist anywhere in the file); frontend/src/components/ProtectedLayout.tsx

Issue

The whole app is one fixed display:flex row: a 220px sidebar plus a content column, with no @media query anywhere in styles.css to collapse or restack it. On a phone-width viewport this leaves roughly 150px for the content column, so signal titles, summaries, and form fields all get crushed into a sliver, and the sidebar can't be hidden or collapsed at all — there is no hamburger/toggle affordance in ProtectedLayout.tsx. A sales rep checking signals from their phone between meetings — a very plausible use case for this product — gets an unusable page.

Proposed solution

Add a breakpoint that turns the sidebar into an off-canvas drawer toggled by a hamburger button, and let content use fluid width.

  1. Add const [isNavOpen, setIsNavOpen] = useState(false); to ProtectedLayout.tsx, and a <button className="nav-toggle" aria-label="Toggle navigation" onClick={() => setIsNavOpen((v) => !v)}>☰</button> rendered before <aside>.
  2. Give <aside className={`sidebar ${isNavOpen ? "open" : ""}`}> the conditional class, and close it on navigation (useEffect on useLocation() calling setIsNavOpen(false)).
  3. Add the breakpoint to styles.css: 
    .nav-toggle { display: none; }

@media (max-width: 720px) {
  .nav-toggle { display: block; }
  .app-shell { flex-direction: column; }
  .sidebar {
    position: fixed; inset: 0 25% 0 0; z-index: 20;
    transform: translateX(-100%);
    transition: transform 200ms ease;
  }
  .sidebar.open { transform: translateX(0); }
  .content { max-width: 100%; padding: 16px; }
}
  4. Add a semi-transparent backdrop <div> behind the open drawer that closes it on click, so the pattern matches standard mobile nav conventions.
Fixed: added a nav-toggle hamburger button and off-canvas drawer with backdrop to ProtectedLayout.tsx, plus the 720px breakpoint in styles.css. Verified at 375px with Playwright: no horizontal scroll, drawer opens/closes correctly.

Definition of done: at 375px viewport width, no horizontal scroll appears anywhere in the app, the sidebar is reachable via the toggle, and closes itself after picking a destination.

F2. Destructive actions have no confirmation High

frontend/src/pages/SettingsTargets.tsx:52-55 — remove() calls api.delete directly from the button's onClick

Issue

Clicking "Delete" next to a target company deletes it immediately, with no confirmation step and no undo. Since a target company likely has accumulated articles/signals tied to it, this is an easy, silent way to lose real triage history with a single misclick — there's no visual "are you sure" friction that destructive actions conventionally get.

Proposed solution

Require a second, explicit click before anything is deleted, using a plain inline state swap — no modal library needed.

  1. Add const [confirmingId, setConfirmingId] = useState<string | null>(null); in SettingsTargets.tsx.
  2. Replace the delete button per row with: 
    {confirmingId === company.id ? (
  <>
    <button type="button" className="danger"
      onClick={async () => { await remove(company); setConfirmingId(null); }}>
      Confirm delete
    </button>
    <button type="button" onClick={() => setConfirmingId(null)}>Cancel</button>
  </>
) : (
  <button type="button" className="danger"
    onClick={() => setConfirmingId(company.id)}>
    Delete
  </button>
)}
  3. Reset confirmingId whenever the list reloads, so a stale confirm state can't linger after a delete elsewhere in the list.
  4. Stretch: if the backend adds soft-delete, skip the confirm step entirely and instead show an "Undo" toast (see F5) for ~5 seconds that calls a restore endpoint — friendlier than a blocking confirm for a reversible action.
Fixed: SettingsTargets.tsx now tracks confirmingId and swaps the row's action into "Confirm delete" / "Cancel" before calling the delete endpoint. Verified with Playwright: canceling leaves the row intact.

Definition of done: a single click on "Delete" never removes data; deletion only happens after a second, distinctly-labeled click.

F3. Core triage loop requires a full page visit per signal High

frontend/src/pages/SignalsFeed.tsx (list has no inline actions) and frontend/src/pages/SignalDetail.tsx:91-102 (status transitions only live here)

Issue

The entire product's daily job is: open the feed, look at a growing list of signals, and mark each one reviewed/archived/dismissed. Today that requires clicking into /signals/:id, reading the detail, clicking a transition button, then clicking "Back to signals" — for every single item. There's no way to mark a signal reviewed/dismissed directly from the list, no bulk "select multiple → archive", and no keyboard shortcuts (e.g. j/k to move, e to archive, as in Gmail/Superhuman-style triage tools). For a workspace tracking several target companies, this list only grows, and the per-item round-trip cost compounds daily — this is the single biggest thing standing between the app and feeling "easy," since it's the most repeated interaction in the whole product.

Proposed solution

Move status transitions onto the feed itself (optimistic, no navigation), then layer bulk and keyboard actions on top.

  1. Extract STATUS_TRANSITIONS out of SignalDetail.tsx into a shared module (e.g. frontend/src/constants/signalStatus.ts) so both the feed and the detail page use the same list.
  2. In SignalsFeed.tsx, add an optimistic transition helper and render the transition buttons inline in .signal-row-main: 
    async function transitionSignal(id: string, status: SignalStatus) {
  const updated = await api.patch<Signal>(`/signals/${id}`, { status });
  setSignals((prev) => prev.map((s) => (s.id === id ? updated : s)));
}
    Stop the click from also following the Link to the detail page with event.preventDefault(); event.stopPropagation(); on the button handler.
  3. Add a checkbox per row plus a "select all" control in the toolbar; track selection in const [selectedIds, setSelectedIds] = useState<Set<string>>(new Set());.
  4. Show a bulk-action bar when selectedIds.size > 0 with "Mark reviewed / Archive / Dismiss" buttons that run Promise.all([...selectedIds].map((id) => transitionSignal(id, status))).
  5. Stretch — keyboard triage: track focusedIndex in state, add a window keydown listener for j/k (move focus), r/a/d (apply reviewed/archived/dismissed to the focused row), and give the focused row a visible highlight (.signal-row.focused).
Fixed: SignalsFeed.tsx now renders inline status buttons per row and a checkbox + bulk action bar, both using the shared STATUS_TRANSITIONS constant. Keyboard-driven triage was left as a follow-up (see roadmap) since it adds meaningful focus-management complexity beyond this pass. Verified with Playwright: inline archive and bulk dismiss both update status without navigation.

Definition of done: a signal's status can be changed without ever leaving the feed page, and 10 signals can be triaged in far fewer than 10 navigations via either inline row actions or the bulk bar.

F4. Zero onboarding or empty-state guidance Medium

frontend/src/pages/SignalsFeed.tsx:121-125, frontend/src/pages/SettingsTargets.tsx:86, frontend/src/pages/SettingsProfile.tsx

Issue

A brand-new workspace lands on an empty signals feed with one muted sentence ("No signals yet. Add target companies and click 'Fetch new signals' to get started.") and no visual path to the two pages that sentence references. There's no first-run checklist ("1. Set your company profile → 2. Add a target company → 3. Fetch signals"), no sample/demo signal to show what a good result looks like, and the company-profile textarea placeholder is the only hint about why that field matters. A new user has to read carefully and self-navigate through three unconnected pages before the product produces any value.

Proposed solution

Add a conditional setup checklist and gate the ingestion action on having something to ingest for.

  1. In SignalsFeed.tsx, also load /settings alongside /target-companies (already fetched there) to know whether offering_description is empty and whether companies.length === 0.
  2. Add a <SetupChecklist> component rendered above .feed-toolbar only when either condition is true, with three linked steps:
    • "Describe your company" → <Link to="/settings/profile">, checked once offering_description is non-empty.
    • "Add a target company" → <Link to="/settings/targets">, checked once companies.length > 0.
    • "Fetch your first signals" → highlights the "Fetch new signals" button, checked once signals.length > 0.
  3. Disable "Fetch new signals" with disabled={companies.length === 0} and a title="Add a target company first" tooltip, so the empty-click dead-end described in the current empty-state sentence can't happen.
Fixed: added SetupChecklist.tsx, shown on the feed whenever the profile/targets/signals are incomplete, and the "Fetch new signals" button is now disabled until a target company exists. Verified with Playwright: checklist appears for a new signup, checks off as steps complete, and the fetch button is disabled until a target company is added.

Definition of done: a brand-new workspace sees a 3-step checklist with working links instead of a single sentence, and the fetch action is unavailable (not just unhelpful) until there's something to fetch for.

F5. All feedback is static inline text — no toasts, spinners, or skeletons Medium

frontend/src/pages/SignalsFeed.tsx:120 ("Loading signals..."), SignalDetail.tsx:56, SettingsProfile.tsx:42, SettingsTargets.tsx (no loading state at all for add/toggle/delete)

Issue

Every async state in the app is represented by a line of muted text appearing/disappearing — "Loading...", "Loading settings...", button labels swapping to "Saving.../Fetching...". There's no skeleton layout to hold the page shape while data loads (so the page visibly jumps once content arrives), no toast/snackbar system for confirmations (a saved settings message just appears as a green sentence under the button, easy to miss if the button isn't in view), and several actions — toggleActive and remove in SettingsTargets.tsx — have no loading indicator or disabled state at all, so a double-click can fire the request twice.

Proposed solution

Add one small toast system, a generic skeleton placeholder, and consistent pending-state handling — none of it needs a dependency.

  1. Add frontend/src/context/ToastContext.tsx exporting a ToastProvider and useToast() hook returning showToast(message, variant); render an aria-live="polite" fixed-position stack that auto-dismisses each toast after ~3s. Wrap <App /> with it in main.tsx.
  2. Replace the ad-hoc savedMessage/inline "Settings saved." pattern in SettingsProfile.tsx with showToast("Settings saved.", "success"); do the same for other mutation confirmations and for surfaced ApiError messages across all four pages.
  3. Add a small <Skeleton rows={n} /> component — a few divs styled with the existing --border/--bg tokens and a shared @keyframes pulse { 50% { opacity: .5; } } — and swap it in for the bare "Loading signals..." / "Loading..." / "Loading settings..." text in SignalsFeed.tsx, SignalDetail.tsx, and SettingsProfile.tsx.
  4. Add local pending state around toggleActive and remove in SettingsTargets.tsx (currently the only two mutations in the app with no loading/disabled state at all) and disable that row's buttons while the request is in flight.
Fixed: added ToastContext.tsx (wired into main.tsx) and Skeleton.tsx, used across all four pages; toggleActive/remove in SettingsTargets.tsx now track a pendingId and disable the row's buttons while in flight. Verified with Playwright: the "Target company added." toast renders and auto-dismisses.

Definition of done: every mutation in the app shows a pending state on its trigger, confirms or errors via a toast, and can't be double-fired by a fast double-click; every page's loading state preserves the layout shape instead of a layout jump.

F6. Signals feed has no search, sort, or pagination Medium

frontend/src/pages/SignalsFeed.tsx:25-36 — loadSignals() fetches the full filtered list with no limit/offset/search params, and renders it as one flat <ul>

Issue

The feed can be filtered by company and status, which is a good start, but there's no free-text search, no way to sort by newest/oldest or relevance, and no pagination or virtualization. As signals accumulate over weeks, this list will grow unbounded in one render — both a usability problem (can't find an older signal about a specific keyword) and eventually a performance one.

Proposed solution

Add client-side search and sort now (cheap, no backend change), and plan a paginated path for when volume grows.

  1. Add a search <input> to the filter panel in SignalsFeed.tsx, and derive the rendered list with: 
    const visibleSignals = useMemo(() => {
  const q = searchQuery.trim().toLowerCase();
  if (!q) return signals;
  return signals.filter((s) =>
    s.article_title.toLowerCase().includes(q) ||
    s.summary.toLowerCase().includes(q) ||
    s.target_company_name.toLowerCase().includes(q)
  );
}, [signals, searchQuery]);
  2. Add a sort <select> ("Newest first" / "Oldest first") and sort visibleSignals by created_at accordingly before rendering.
  3. Once a workspace's signal count regularly exceeds ~50, extend loadSignals() to pass limit/offset query params and add a "Load more" button (or an intersection-observer-triggered infinite scroll) instead of fetching the full history on every load.
Fixed: added client-side search and a newest/oldest sort control to SignalsFeed.tsx via a useMemo-derived visibleSignals list. Server-side pagination was left as a follow-up (see roadmap) since it isn't yet needed at current signal volumes. Verified with Playwright: searching "VP of Sales" against 3 seeded signals correctly narrowed the list to 1.

Definition of done: a user can find a specific past signal by typing a keyword or company name without scrolling the full history, and the feed stays fast as signal volume grows over weeks.

F7. Visual design has no personality, hierarchy aids, or motion Medium

frontend/src/styles.css — entire file: one accent color, no icon usage, no transition/animation declarations, no illustration/imagery anywhere

Issue

Every screen is built from the same three shapes (card, badge, button) in one accent blue with no supporting visual language: no icon set (not even for nav items or status badges), no illustration for empty states, no hover/active transitions on any interactive element (buttons, links, and rows all snap instantly), and no use of color beyond the single accent + status badges. This is internally consistent, which is good, but it reads as an admin/internal tool rather than a product a user would enjoy opening every day — directly working against the "fun" half of the brief.

Proposed solution

Layer in an icon set, motion, and a touch of illustration incrementally — no new dependency required.

  1. Add a small local icon set as inline SVG components under frontend/src/components/icons/ (e.g. SignalsIcon, ProfileIcon, TargetsIcon for the sidebar; a status-specific glyph for each status-badge variant) — each just a plain <svg> functional component, no icon-font or package needed.
  2. Add global transitions in styles.css: 
    button, a.signal-row, .sidebar nav a, .target-list li {
  transition: background-color 150ms ease, transform 150ms ease;
}
button:hover:not(:disabled) { transform: translateY(-1px); }
a.signal-row:hover { background: var(--bg); }
  3. Add one reusable empty-state illustration (a simple inline SVG — e.g. a stylized radar/newspaper glyph in --accent) used both on the empty signals feed and beside the login form, paired with warmer copy than the current single muted sentence.
Fixed: added an inline SVG icon set (components/icons/NavIcons.tsx) for sidebar nav and the theme toggle, hover/focus transitions on buttons and rows, and a reusable EmptyStateIllustration.tsx used on both the empty feed and the login screen. Status-badge icons were left as-is since the color+text pattern already reads clearly; icons there would add visual noise for little gain.

Definition of done: the sidebar and status badges use icons alongside text, hovering any interactive element produces a visible response, and the two most "blank" screens (empty feed, login) no longer read as bare forms on gray backgrounds.

F8. Accessibility gaps Medium

frontend/src/pages/SignalDetail.tsx:87-89 (copy button, icon-free but ambiguous state), status badges throughout, no custom :focus-visible styling anywhere in styles.css

Issue

No element in the app has a custom focus style, so keyboard users rely entirely on the browser's default outline, which is inconsistent across browsers and easy to miss against the card backgrounds used here. Status badges convey state via color + text (good — not color alone), but several interactive controls have no aria-label beyond their visible text, which is fine for buttons with clear labels but would matter as soon as any icon-only control is added per F7. Color contrast (e.g. --muted: #5b6270 on --bg: #f6f7fb) looks broadly acceptable by eye but hasn't been measured against WCAG AA and should be audited, especially the 0.68rem status-badge text.

Proposed solution

Add a global focus style, audit contrast, and route status messaging through an accessible live region.

  1. Add to styles.css (purely additive — nothing currently suppresses the default outline, so this only improves consistency): 
    :focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
  2. Run Lighthouse or axe DevTools against all four routes (login, feed, detail, settings); if --muted (#5b6270 on #f6f7fb) or the 0.68rem status-badge text fails WCAG AA, darken --muted slightly (e.g. towards #4b5361) or bump the badge font-size — both are single-line changes since colors are centralized as CSS variables.
  3. Once the toast system from F5 exists, ensure its container has aria-live="polite" so confirmations and errors are announced to screen reader users automatically, not just sighted ones.
Fixed: added a global :focus-visible outline in styles.css, and the new toast container uses aria-live="polite". Contrast was calculated directly (WCAG relative-luminance formula) for every foreground/background pair in both the light and new dark palette rather than via a browser audit tool (none was available in this environment) — all pairs clear 4.5:1 for normal text, so no color changes were needed.

Definition of done: tabbing through any page shows a visible focus indicator at every stop, and an automated audit reports no critical color-contrast violations.

F9. Digest email: compliance and readability gaps Medium

backend/app/services/digest.py:28-56 — _render_digest_html()

Issue

The daily digest is a reasonable, on-brand HTML block (reuses the same accent color and card styling as the app), but it has no unsubscribe/preference link (email best practice, and a legal requirement in several jurisdictions for automated bulk mail), no plain-text multipart/alternative part for clients that block HTML or for accessibility, no preheader text (the snippet shown next to the subject line in inbox previews is currently whatever text happens to render first), and the layout uses plain <div>s rather than the table-based structure that's still needed for consistent rendering in Outlook/older clients.

Proposed solution

Bring the digest up to standard transactional-email practice: an unsubscribe path, a plain-text part, and a preview snippet.

  1. In _render_digest_html(), append a footer block with a preferences link, e.g. <a href="{frontend_base_url}/settings/profile">Manage email preferences</a> (or a dedicated unsubscribe endpoint if per-user opt-out is added later).
  2. Add a sibling _render_digest_text(rows) in digest.py that builds a plain-text equivalent (company name, title, summary, "why it matters", and the article/signal URLs, one per line), and extend EmailClient.send_email (backend/app/services/email_client.py) to accept an optional text_body and send both parts as multipart/alternative.
  3. Add a hidden preheader immediately after <body>: 
    <span style="display:none;max-height:0;overflow:hidden;">
  {n} new signals: {first_company}, {second_company}...
</span>
  4. If Outlook desktop is a realistic client for this audience, wrap items_html in a single-column <table role="presentation" width="100%"> instead of nested <div>s, since Outlook's rendering engine (Word-based) handles tables far more reliably than flexbox-adjacent div layouts.
Fixed: digest.py now renders a hidden preheader and a "Manage email preferences" footer link, plus a new _render_digest_text() plain-text alternative sent via EmailClient.send_email's new optional text_body parameter (attached plainest-first per MIME convention). The Outlook table-layout stretch item was left as-is since it's conditional on Outlook desktop being a realistic client for this audience, which hasn't been established. Covered by an updated backend test asserting both parts contain the expected content.

Definition of done: every digest email includes a working preferences/unsubscribe link, renders sensibly with images/HTML blocked, and shows a useful preview line in inbox lists (Gmail, Outlook, Apple Mail).

Fixed: added the .spinner CSS animation and wired it into the "Fetch new signals" button in SignalsFeed.tsx. Backend progress reporting was left as a follow-up stretch item since it needs a new polling/job-status endpoint, out of scope for a frontend-focused pass.

F11. Forms lack real-time validation and forgiving input Low

frontend/src/pages/Login.tsx:66 (minLength={10} with no visible hint), frontend/src/pages/SettingsTargets.tsx:30-34 (comma-separated keywords as a single text input)

Issue

The password field enforces a 10-character minimum only via the browser's native validation popup on submit — there's no visible hint up front ("at least 10 characters") or live strength/length indicator, so a user only discovers the rule after a failed submit. Separately, keywords/aliases are entered as one comma-separated string (frontend/src/pages/SettingsTargets.tsx:30-34); this is compact but easy to fat-finger (stray comma, no trim feedback, no visual confirmation of what will actually be saved as individual tags).

Proposed solution

Make the password rule visible up front, and replace the raw comma-separated keyword field with a chip-style tag input.

  1. In Login.tsx, add a static <span className="subtitle">At least 10 characters</span> directly under the password <input>, always visible — not gated behind a failed submit.
  2. Replace the keywords <input> in SettingsTargets.tsx with a small <TagInput> component that keeps keywords: string[] directly in state instead of a joined string: typing and pressing Enter/comma appends a chip, each chip renders with a small "×" to remove it, and Backspace on an empty input removes the last chip. Submit keywords as the array directly, dropping the current .split(",").map(k => k.trim()).filter(Boolean) parsing step entirely.
Fixed: added a static password hint under the field in Login.tsx, and a new TagInput.tsx component now backs the keywords field in SettingsTargets.tsx, keeping keywords: string[] directly in state. Verified with Playwright: typing and pressing Enter renders removable chips.

Definition of done: the password rule is visible before a user ever submits the form, and adding/removing a keyword shows the exact list that will be saved, with no parsing surprises from stray commas or whitespace.

F12. No dark mode Low

frontend/src/styles.css:1-10 — all colors are already centralized as CSS custom properties on :root

Issue

Every color in the app already flows through a small set of CSS variables (--bg, --panel, --fg, --muted, --accent, --border), which is exactly the structure that makes dark mode cheap to add — but no dark variant exists, and there's no theme toggle or prefers-color-scheme handling. For a dashboard likely to be left open in a browser tab all day, this is a commonly-expected option its target audience (software/sales-tool users) will notice is missing.

Proposed solution

Layer a dark palette onto the existing CSS variables and add a small persisted toggle.

  1. Add to styles.css: 
    @media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    --bg: #14161b; --panel: #1c1f26; --fg: #e8eaed;
    --muted: #9aa2b1; --border: #2a2e37; --accent: #6c93e8;
  }
}
:root[data-theme="dark"] {
  --bg: #14161b; --panel: #1c1f26; --fg: #e8eaed;
  --muted: #9aa2b1; --border: #2a2e37; --accent: #6c93e8;
}
  2. Add a small toggle button in .sidebar-footer (ProtectedLayout.tsx) cycling light → dark → system, storing the choice in localStorage and setting document.documentElement.dataset.theme on load and on change.
Fixed: added the dark palette (with --danger/--ok lightened specifically for dark backgrounds to keep 4.5:1+ contrast) and a useTheme.ts hook plus a light/dark/system toggle in the sidebar footer, persisted via localStorage. Verified with Playwright: cycling the toggle sets data-theme="dark" and the palette updates accordingly.

Definition of done: a dark palette is available, meets the same contrast bar validated in F8, can be toggled independently of the OS setting, and persists across reloads.

F13. Static browser tab title and no favicon Low

frontend/index.html:1-11 — <title>NewsAtlas</title> never changes; no <link rel="icon">

Issue

Every route renders under the same static "NewsAtlas" tab title, so a user with multiple tabs open (e.g. the feed and a signal detail) can't distinguish them from the tab bar, and there's no favicon, so the tab is also visually anonymous next to other open tabs. Both are quick wins for day-to-day orientation.

Proposed solution

Add a tiny reusable title hook and a favicon asset.

  1. Add a usePageTitle hook (e.g. frontend/src/hooks/usePageTitle.ts): 
    export function usePageTitle(title?: string) {
  useEffect(() => {
    document.title = title ? `${title} · NewsAtlas` : "NewsAtlas";
  }, [title]);
}
    and call it in each page: usePageTitle("Signals") in SignalsFeed.tsx, usePageTitle("Company profile") in SettingsProfile.tsx, usePageTitle("Target companies") in SettingsTargets.tsx, and usePageTitle(signal?.article_title) in SignalDetail.tsx once the signal loads.
  2. Add a favicon file under frontend/public/favicon.svg (a simple mark, e.g. the accent-blue glyph proposed in F7) and reference it in index.html: <link rel="icon" href="/favicon.svg" type="image/svg+xml">.
Fixed: added usePageTitle.ts, called from every page, and frontend/public/favicon.svg referenced from index.html. Verified with Playwright: document.title updates per route (e.g. "Target companies · NewsAtlas").

Definition of done: each open tab shows a distinct, meaningful title (e.g. the signal's article title on its detail page) and a NewsAtlas favicon instead of the browser default.

What's already working well

Consistent component vocabulary. Cards, status badges, and the two button variants (secondary, danger) are reused identically across all four pages despite there being no shared component library — that discipline will make every recommendation above cheaper to apply consistently.
Clear, jargon-appropriate copy. Labels like "Target company", "Signals feed", and "Outreach snippet" match how the target user (sales/BD) actually thinks, and helper text like the company-profile description explains why a field matters, not just what to type.
Sensible information architecture. A persistent sidebar with three destinations, filters surfaced directly on the feed rather than hidden behind a menu, and a detail page that leads with the AI's answer (summary → why it matters → outreach snippet) mirrors the order a salesperson actually needs the information in.
Small delight already present. The "Copy snippet" button's "Copied!" confirmation (SignalDetail.tsx:44-49) is exactly the kind of low-cost, high-value micro-interaction the rest of the app is currently missing elsewhere — worth using as the template for other action feedback (see F5).
